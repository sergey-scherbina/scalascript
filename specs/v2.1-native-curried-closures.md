# ScalaScript 2.1 native curried closures

## Overview

The self-hosted parser preserves Scala-style parameter-list calls as nested
applications while a definition's parameter clauses are represented by one
flattened parameter list. The lowerer must reconcile those two source shapes
before CoreIR: a nested call to a known definition is flattened only when its
combined arguments satisfy that definition's total arity. CoreIR closures keep
their strict exact-arity ABI.

## Interface

For a known source definition `f` of total arity `n`:

- source `f(a...)(b...)` lowers to one CoreIR `App(f, a..., b...)` when the
  combined argument count is `n`;
- a single application with fewer or more arguments keeps the strict runtime
  arity error;
- a nested application whose inner function is not a known flattened
  definition remains nested, preserving genuine function-returning-function
  semantics.

The function body receives arguments once, in source order. This is a generic
definition-arity reconciliation pass, not a named-function or example-specific
dispatch path.

## Behavior

- [x] A known two-parameter definition called as `f(a)(b)` lowers identically
      to `f(a, b)` on the native VM and direct ASM.
- [x] Arguments retain left-to-right order and evaluate once.
- [x] A single under-application such as `required()` retains the explicit
      arity failure instead of becoming a closure.
- [x] Supplying more arguments retains the explicit arity failure instead of
      dropping or reinterpreting arguments.
- [x] Installed `examples/algebraic-effects.ssc` advances through the curried
      `withLogger` helper identically on VM/direct ASM.

## Design

The lowerer pre-scans assembled source definitions into a name/total-arity
table. When resolving a nested application, it combines clauses only for a
known definition whose combined count is exactly that arity. `Runtime.run` and
the direct-bytecode `Emit.app` shim continue to reject every non-exact closure
application identically.

## Out of scope

- changing parser or typer representation of parameter-clause boundaries;
- changing first-class partial-application syntax;
- accepting too few or too many arguments at runtime;
- compatibility-frontend fallback or v1 dispatch.

## Decisions

- **Definition-aware frontend reconciliation** — chosen because parameter-list
  structure is source syntax while CoreIR has one exact-arity application ABI;
  rejected generic runtime under-application because it silently accepts
  invalid single-clause calls such as `required()`.

## Results

Implemented in the self-hosted lowerer with a definition/total-arity pre-scan.
The installed VM/direct-ASM fixture proves equivalent `f(a)(b)` and `f(a, b)`
evaluation, while the native-entry negative gates retain exact errors for both
under- and over-application. `examples/algebraic-effects.ssc` now completes all
eleven lines identically on VM/direct ASM. The consolidated release gate is
green with 35 identical, 31 both-fail, 129 skipped, zero mismatch/one-sided
rows, and `release.ready=true`.
