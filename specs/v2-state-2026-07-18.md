### 1a. `v2/conformance/check.sh` is RED on HEAD (the ROADMAP claims it green)

`check.sh` exits **1**. The failures (all reproduced twice, byte-for-byte):

1. **`FAIL ssc0c uselib.ssc0 тАФ ir differs`** (log line 679). A **multi-file differential**: the
   ssc0c self-compiler's IR for a multi-file program differs from the Scala reference front's.
   The single-file `ssc0c FIXPOINT` and the `MULTI-FILE FIXPOINT` (self-reproduction) both still
   PASS тАФ so this is a self-hosting *correctness* gap on multi-file input, not a fixpoint break.
   Stack-independent.

2. **`FAIL ssc-run exit=1` (├Ч2) тАФ `java.lang.StackOverflowError`** in
   `Compiler.compileEffectAwareApplication` / `evaluateRemainingAsStep` (`Runtime.scala:1512/1530`),
   retried and failed again. This is the **known-open** `coreir-compiler-unbounded-depth` (BUGS.md)
   / ROADMAP **K62.3** ("compile-recursion robustness (StackOverflow on large programs)").
   **Stack-size sensitive:** `check.sh`'s `ssc()` helper runs `java -jar` with the **default** stack
   (line 105), while the real `bin/ssc` launcher uses `-Xss64m` and its "big program" cases use
   `sscx()` with `-Xss512m`. So this is partly the CI-vs-local stack asymmetry AGENTS.md warns about тАФ
   *and* a real unbounded recursion in the compiler.

**Note on the earlier "green":** running `bash check.sh 2>&1 | tail -40` reports exit 0 тАФ but that is
`tail`'s exit code, not `check.sh`'s. Capture the real code with
`bash v2/conformance/check.sh > log 2>&1; echo $?`.

## 3. R2 тАФ backend capability matrix (empirical)

There are **two source surfaces**, not one тАФ this is essential to reading the matrix honestly:

- **Family A тАФ full `.ssc`** (native front тЖТ CoreIR), run directly: `native`, `JVM-bytecode`, `v2-JS`.
- **Family B тАФ the tower's own languages** ssc0 / Mira (тЖТ CoreIR), emit-and-compile: `Rust`, `WASM`.
  ssc0 is a *smaller* language: it has no `.foldLeft`/`Map`/`Set`/`s""` method surface, so most
  Family-A feature cells are **N/A** there тАФ tested with ssc0-equivalent programs instead.
- **Family C тАФ Swift** (`emit-swift`, full `.ssc` тЖТ CoreIR тЖТ SwiftPM package): **emit-only**.

### Family A тАФ full `.ssc` (test files in scratchpad `r2_{core,list,mapset,effects}.ssc`)

| Feature | native `ssc run` | bytecode `ssc run --bytecode` | v2-JS `ssc-tools run-js --v2` |
|---|---|---|---|
| enum + match | тЬУ `green` | тЬУ `green` | тЬУ `green` |
| closures / HOF | тЬУ `16` | тЬУ `16` | тЬУ `16` |
| string `substring` | тЬУ `hello` | тЬУ `hello` | тЬУ `hello` |
| string `split` | тЬУ `2` | тЬУ `2` | тЬУ `2` |
| `s""` interpolation | тЬУ | тЬУ | тЬУ |
| big-int `2147483648` | тЬУ | тЬУ | тЬУ |
| big-int `9223372036854775807` | тЬУ | тЬУ | тЬУ |
| `List.map` | тЬУ | тЬУ | тЬУ |
| `List.filter` | тЬУ | тЬУ | тЬУ |
| `List.foldLeft` | тЬУ `15` | тЬУ `15` | тЬЧ **crash:** `__method__: no dispatch for .foldLeft on List` |
| `List.reduce` | тЬУ `15` | тЬУ `15` | тЬЧ unreached (file aborts at foldLeft) |
| `List.mkString(a,b,c)` | тЬУ | тЬУ | тЬЧ unreached |
| `Map` build/get/size | тЬУ | тЬУ | тЬЧ **crash:** `not callable: <map>` |

| effects + handler + `resume` | тЬУ `List(Hello, World!)` | тЬУ | тЬЧ **crash:** `unimplemented primitive: effect.perform.oneshot` |

**native and JVM-bytecode: full parity, byte-identical output on every cell.**
**v2-JS: partial** тАФ arithmetic / enum / string / HOF / interpolation / big-int all correct
(note: `run-js --v2` gets 64-bit ints right, unlike the deprecated `emit-js` which is documented
32-bit-nonconforming), but `List` combinators past `map`/`filter`, `Map` access, and effects are
**missing** on the v2-JS backend. This is the concrete basis for the "K3 JS overstated" verdict.

### Family B тАФ tower Rust / WASM (ssc0 surface; VM = `v2/ssc run` oracle)

| Program (feature) | VM | Rust `ssc0-rust`+rustc | WASM `ssc0-wasm` |
|---|---|---|---|
| `fact` (Int, recursion) | `120` | `120` | `120` |
| `tco` (1e6 tail calls) | `500000500000` | `500000500000` (constant stack) | `500000500000` (constant stack) |
| `calc` (ADT + match + env + let) | `42` | `42` | `42` |
| `quicksort` (list, HOF) | `List(1, 1, 2, 3, 4, 5)` | same | same |
| `bigfact` (**BigInt** 30!) | `265252859812191058636308480000000` | тЬЧ **empty** | тЬЧ **empty** |
| `async-future` (async lib) | `List(1, 2, 10, 20, 7, 7)` | тЬЧ **empty** | тЬЧ **empty** |

- **BigInt is silently dropped** by the tower Rust backend: `bigfact.rs` compiles (2172 B) but its
  body emits `V::U` (Unit) wherever `#big.*`/`#i->big` should be тАФ no `i128`/BigInt in the output тАФ
  so the result is Unit and `show` prints nothing. Fail-open, exit 0. WASM reuses the Rust backend,
  same gap.
- **async-future** fails for a *different* reason: `ssc0-rust` emits a **0-byte** `.rs` because the
  **VM StackOverflows in `Compiler.compile`** while running `backend-rust.ssc0` on it (same K62.3
  recursion as ┬з1a). Not a "Rust can't do async" limitation.
- **Pure algebraic effects DO run on tower Rust:** `check.sh` shows
  `ok effect run (State) -> Rust => Pair(105, 5) (rustc)`.
- **Mira:** `v2/mira-rust hm-adt-match.hm` тЖТ `5` (correct). Caveat: `v2/mira` prints the **inferred
  type** ("Int"), not a value тАФ it is the type-checker front; Mira *execution* is via
  `mira-rust`/`mira-js` (or ssct-emit тЖТ VM), so there is no direct "Mira-VM run" launcher.

### Family C тАФ Swift (`ssc-tools emit-swift file.ssc`)

Emits a **SwiftPM package** (`Package.swift` + `Sources/`) from full `.ssc` via checked v2 CoreIR.
Confirmed it emits for `r2_core.ssc`. **Emit-only** тАФ ssc does not execute it; you `swift build`
separately. **UNVERIFIED:** I did not run `swift build` on the emitted package (heavy; the Swift
backend is under an active sibling claim `swift-renderer-port`), so Swift *execution* correctness is
untested here. `NativeUiSites`/`SwiftBackend` reference confirm the emit path is real.

**Which backends run-directly vs emit-only:** native, JVM-bytecode, v2-JS, WASM run directly (WASM
via node WASI inside `ssc0-wasm`). Rust is emit-then-`rustc`. Swift is emit-only (SwiftPM package).

**`Prims` тАФ the ╬┤ table.** The single `resolveBuiltinRaw` `op match` spans **2497тАУ3825 (~1328 lines)**.
Measured minimal ╬┤:
- The **self-hosting fixpoint** (`ssc1-lower` + `ssc1-front` + P6.5 `F`) uses **23 distinct `#`-prims**.
- The **whole ssc0 tower** (`v2/lib/*.ssc0`) uses **66 distinct `#`-prims**.
- The remaining ~1200 lines of the table are **not for the tower**: type-dispatched arithmetic,
  `__method__`/`__arith__`/`__unary__`/`__eq__` dynamic dispatch, `Map`/`Set` factories,
  `__regfields__`, and the `dec.*`/`effect.*`/`ui`/actor prims тАФ they exist to run the **full v1-lowered
  `.ssc` surface** through the native FrontendBridge, i.e. the seam `v2-finish` intends to retire.

**Invariant flags** (ROADMAP "Invariants": *kernel stays untyped; no effects/continuations; no target
backend baked in; if it can be a program on the tower, it is*):

- **`PortableEffects` breaks "no effects/continuations in kernel."** It is a continuation-manipulating
  handler driver in Scala. It also *contradicts K3's own text* ("effects/async/actors are libraries on
  the tower, not kernel features"). Effects are half-in-kernel today.
- **`Emit` + the JVM-bytecode emitter break "no target backend baked in."** A JVM backend's runtime
  surface ships inside the kernel jar.
- **`NativeUiSites` breaks "if it can be a program on the tower, it is."** It is a UI-target compile
  pass тАФ and it is **not even used by the v2 kernel's own pipeline**: the only references are the v1
  CLI `RunNativeV2.scala`, a v2 `ui-plugin`, and the `SwiftBackend`. Dead weight in the kernel jar
  from the tower's perspective.
- The **~1200 lines of FrontendBridge/method-dispatch ╬┤ prims** amount to a whole `.ssc` language
  runtime baked into the "minimal untyped kernel."

**F5 progress (2026-07-19, `v2-f5-kernel-small`):** `NativeUiSites` (127) **relocated** out of `v2/src`
into the `v2/nativeui` sbt module тАФ kernel `v2/src` is now **6228 lines** (was 6355), and the kernel jar
(`scala-cli package v2/src`) no longer ships it, fixpoint byte-identical to origin/main at the same commit
(169,133 B @31cde7db6 тАФ the absolute count climbs each p65 push; the invariant is equality-to-origin, not a
fixed number).
Remaining candidates: `Emit` (JVM-backend surface; kernel refs it only in comments тАФ likely relocatable),
`PortableDecimal`/`PortableEffects` (the latter is an effect driver that may back a ╬┤-prim тЖТ possibly
irreducible), `FastCode`/`SelfRec*` (perf тЖТ relocate only if byte-neutral).

**Minimal-kernel target (rough estimate тАФ analysis only, nothing moved):**
irreducible = `CoreIR` (415) + `Ssc0` (311) + `Main` (62) + Value/trampoline (145) + `Runtime`
driver (313) + a lean `Compiler` (~500 of 973, dropping effect-aware + fast paths) + a **minimal ╬┤
table** (~66 tower prims тЙИ ~450 lines vs 1328) + Ir{Encode,Decode,ToData} (164) + `Show` (44)
тЙИ **~2,400тАУ2,800 lines** тАФ roughly **40тАУ45 %** of today's 6,355. The delta to remove/relocate:
`FastCode` (820) + `SelfRec/SelfTailRec` (142) perf layers, `Emit` (292) JVM-backend surface,
`NativeUiSites` (127), `PortableDecimal` (171, тЖТ tower library over BigInt prims),
`PortableEffects` (221, тЖТ tower library per K3), `V2PluginRegistry`/`V2EffectContext` (95) interop
glue, and the ~1200 FrontendBridge/method-dispatch prim lines.

Caveat: the line count is an *estimate* тАФ no extraction was performed; the 23/66-prim figures are
measured (`grep -hoE '#[A-Za-zтАж]+' v2/lib/*.ssc0 | sort -u`), the ╬┤-table savings are approximate.
