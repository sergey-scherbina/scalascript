# v2.2 P6.0/P6.1 spike — results & verdict

Status: **GREEN** — P6.0 (precedence gate) + P6.1 (total error model) both landed (2026-07-13).
Gate for `specs/v2.2-self-hosted-dialect.md` §9. Spike code:
`uniml/core/src/test/scala/scalascript/uniml/spike/ScalaSpike.scala` (dialect + projection) and
`ScalaSpikeSpec.scala` (11 tests). Run: `cd uniml && sbt
"uniml/testOnly scalascript.uniml.spike.ScalaSpikeSpec"`. §P6.1 at the bottom.

## The gate question

> Does UniML's frame VM (`Open`/`Close`/`Emit`/`Reframe`) express operator precedence tolerably, or
> must we lock decision (B) as "UniML structure + a separate Pratt expression layer"?

## Verdict — YES, tolerable; decision (B) is *refined*, not forced

UniML gives **no precedence for free** (expected — the VM encodes nesting, not binding). But a
separate *parser layer* is **not** needed. The clean pattern is:

1. The dialect's structural pass runs a normal **Pratt parse internally** (a few lines of Scala) to
   the correct expression tree — this is the parsing intelligence UniML never claimed to provide.
2. It then **mechanically serialises** that tree to `VmToken`s with one rule: **a frame opens on the
   first token of its subtree and closes (`Reframe.closeAfter`) on the last.** This is source-order
   and lossless; `Reframe` handles the cases where one token must open several frames (a left-nested
   product like `1 * 2 + 3`) or close several (`… 2 * 3` at expression end).
3. UniML's VM rebuilds exactly that tree as a lossless CST.

So the "Pratt layer" lives *inside the dialect* and its output flows into the *same* UniML CST — we
keep one lossless tree (with trivia/spans/error nodes) for tooling, and precedence is correct. No
second parser, no second tree. `SpikeEmit.walk` (the open-on-first / closeAfter-on-last serialiser)
is ~12 lines and dialect-agnostic; the precedence knowledge is isolated in `SpikeParse` (standard
climbing).

## What was proven

**Gate (CST shape), 6/6 green** — `ScalaSpikeSpec`:
- `1 + 2 * 3` → `add(1, mul(2,3))` (`*` binds tighter);
- `1 * 2 + 3` → `add(mul(1,2), 3)` (left-nested product — the `Reframe` multi-open case);
- `(1 + 2) * 3` → `mul(paren(add(1,2)), 3)` (parens override);
- significant-token source order preserved (lossless up to trivia — see caveat);
- `if x then g(x, 1+2) else 0` → the expected `if`/`call`/arg frames.

**End-to-end (projection → existing lowerer → run-ir), 4/4 green** — projection `SpikeProject`
emits the ssc-v2 `Pair(tag,data)` AST (via the `mk*` builders), fed to the **unchanged**
`v2/lib/ssc1-lower.ssc0` `lowerProg`, encoded to Core IR, executed with `run-ir`:

| toy | expr | run-ir | vs `ssc1-front` | Core IR |
|---|---|---|---|---|
| add-mul | `1 + 2 * 3` | 7 | 7 | **byte-identical** |
| mul-add | `1 * 2 + 3` | 5 | 5 | **byte-identical** |
| paren   | `(1 + 2) * 3` | 9 | 9 | **byte-identical** |
| nested  | `2 * (3 + 4) - 5` | 9 | 9 | **byte-identical** |

The Core IR is **byte-for-byte identical** to what the hand-written `ssc1-front` produces — the
projection is faithful, and §7's "reuse the mature lowerer via projection" is validated.

**Dual-compile (scalac oracle), green** — the same expressions as standard Scala 3
(`SpikeToy.scala`, `object SpikeToy { def … }`) compile under scalac (3.8.4) and evaluate to
`7 5 9 9`, identical to ssc-v2 run-ir. The subset is Scala-valid and semantically agrees on the toy.

## Reproduce the end-to-end diff

`SpikeSpec` emits `<SPIKE_OUT>/*.proj` (projection) and `*.toy.ssc` (source) — default
`SPIKE_OUT=…/scratchpad/p6.0`, override via env. For each toy the harness compared:
- reference: `ssc.jar run v2/bin/ssc1-run.ssc0 <toy>` → Core IR → `run-ir`;
- spike: a 2-line ssc0 driver `import "../lib/ssc1-lower.ssc0"; def main = () =>
  #io.print(#coreir.encode(lowerProg(<proj>)))` (placed in `v2/bin/` so the relative import
  resolves) → Core IR → `run-ir`.
Both the executed value and the Core IR text were diffed. (Driver is transient — not committed.)

## Caveats / deferred to later phases (do NOT block P6.1 on these)

- **Trivia losslessness**: the spike's parser skips whitespace tokens (they are lexed but not put in
  the Node tree), so the CST is lossless only up to trivia. Full trivia attachment is P6.2 polish;
  it does not affect the gate (precedence + source order of significant tokens).
- **Error nodes**: the spike has a placeholder `spike.errExpr` fallback but does not implement the
  total error model — that is P6.1.
- **Grammar breadth**: only the toy subset (arith `+ - *`, call, paren, if, single/param def). P6.2
  grows it to full front coverage.
- **Import resolution gotcha**: nested ssc0 `import "..."` resolve relative to the ENTRY file's dir,
  so the driver must live under `v2/bin/` (not an arbitrary dir) for `../lib/…` to resolve.

## Conclusion

Gate passed. Proceed to **P6.1 (error model + typed holes)**. The architecture is confirmed:
UniML owns the lossless CST + structure; a Pratt pass inside the dialect owns precedence and
serialises into that CST via `Reframe`; the projection reuses the existing `ssc1-lower` (byte-exact).

## P6.1 — total, error-resilient pipeline (error nodes + typed holes)

Status: **GREEN**. The pipeline is now total end-to-end and "an error is a node, not an exception."

**What changed.** The dialect's parser is rewritten to be **total**: an EOF-safe cursor
(`peek`/`advance` return `Option`), so no input crashes it. On an error it records a `Diagnostic`
(code + `Severity.Error` + span) — collected in the dialect's `ProcessBatch.diagnostics`, which
`UniML.parse` aggregates into `ParseResult.diagnostics` and reflects in `status` (Error ⇒
`Incomplete`). Recovery **resyncs to the next `def` boundary**: junk between defs is skipped into a
`spike.error` CST frame (salvaged, lossless), and a broken expression stops at the boundary so the
following `def` parses cleanly. The projection `SpikeProject` is **total**: any error / missing
subtree becomes the ssc-v2 hole `Pair("prim", Pair("__notImplemented__", Nil))` — the generalisation
of `???`/`__notImplemented__` (`b77862d7f`), which compiles fine and throws only if evaluated.

**Design note on "error nodes" vs "typed holes".** Present-but-wrong tokens → a `spike.error` CST
node (real tokens, diagnostic). Absent tokens (missing operand/body) → no token to emit, so it is a
diagnostic + a projection **hole**. Because the spike reuses the *untyped* `ssc1-lower`, the hole is
`__notImplemented__` — the *typed* refinement (a hole carrying its expected/inferred type) arrives
with the v2.2 typer (a later phase); the total-pipeline + containment property it enables is proven
here.

**Proven (11/11 green):**
- Parser + projection are **total** — never throw on a garbage corpus (`""`, `def (`, `)))`,
  `1 + + 2`, `def main(): Int =`, `if if if`, `def a b c d e`, …).
- **Containment (unit):** `def broken(): Int = 1 +` ⧺ `def main(): Int = 2 * 3` → `status =
  Incomplete`, a `spike.missing-operand` diagnostic, **both** defs survive, `main`'s body is a clean
  `spike.mul`, and `broken`'s `add` has no `bin.right` (→ hole).
- **Containment (end-to-end):** the same input projects to a two-def AST where `broken`'s body is a
  `__notImplemented__` hole; fed to the **unchanged** `ssc1-lower` it compiles, and `run-ir` of
  `main` returns **6** — the broken sibling does not poison it (the hole is never evaluated).
- Junk between defs → a `spike.error` CST frame; both surrounding defs survive.
- Diagnostics carry a `spike.*` code, `Error` severity, and a source span.
- All four P6.0 well-formed toys still produce **byte-identical Core IR** vs `ssc1-front` (the
  error-model changes did not regress the happy path).

**Verdict:** error-as-node + total projection + containment demonstrated. A malformed region
degrades to a hole and the rest of the file compiles and runs. Proceed to **P6.2** (grow the dialect
to full front coverage, differential-tested against `ssc1-front`). Deferred (do not block P6.2):
trivia losslessness (whitespace still dropped), typed holes proper (needs the v2.2 typer), and
statement-level (intra-def) resync — current resync granularity is the `def` boundary.

## P6.2 (slice 1 of N) — the full infix precedence table

Status: **GREEN**. First slice of "grow the dialect to full front coverage": the complete ssc-v2
infix operator set, differential-tested byte-exact against `ssc1-front`.

**What changed.** The lexer now reads a **greedy maximal run of operator characters**
(`+-*/%<>=!&|^~:`) as one token (so `==`, `<=`, `>>>`, `:=`, `++` lex as single ops; a lone `=`/`:`
stay `spike.eq`/`spike.colon`), and `SpikeParse.opPrec` is an **exact mirror of `ssc1-front`'s
`opPrec`** (`~ ~>`=9, `* / %`=8, `+ -`=7, `++ :+ << >> >>>`=6, `== != < <~ > <= >= &`=5, `&& ^ |`=4,
`||`=3, `!`=2, `:=`=1), left-associative. All infix nodes are now one generic `spike.infix` frame
carrying the operator token (role `bin.op`); the projection reads the op from that token and emits
`mkInf(op, l, r)`. (`::` / `->` / `to` / `until` are special-cased in `ssc1-front` outside `opPrec`
— deferred to a later slice, with offside layout.)

**Proven (16/16 tests + e2e).** New CST-shape tests: mixed precedence `1 + 2 * 3 - 4 / 2`,
left-associativity `10 - 3 - 2`, cross-tier `1 + 2 < 3 * 4`, boolean tiers `&& > ||`, maximal
multi-char lexing (`== <= >> != ++`). **End-to-end differential oracle:** an operator corpus
(`ops-prec`, `ops-assoc`, `ops-shift`, `ops-cmp`, `ops-bool`, `ops-bit`) all produce **Core IR
byte-identical to `ssc1-front`** — precedence, associativity, and operator lexing match exactly.
run-ir results confirm (`ops-prec`→5, `ops-assoc`→5, `ops-shift`→12, `ops-cmp`/`ops-bool`→true,
`ops-bit`→5). Containment (P6.1) still holds (`broken-sibling`→6).

**Verdict:** the Pratt-inside-dialect + `spike.infix` design scales to the full operator table with
zero divergence from the hand-written front. Next P6.2 slices: `::`/`->`/`to`/`until` + prefix ops;
**offside layout** (the big one — significant indentation → `Reframe`, where UniML's YAML-proven
indent machinery applies); then patterns / `match` / `given`-`using` / declarations.

## P6.2b (slice 2) — offside layout (indented def-body blocks)

Status: **GREEN**. An indented `def` body is now a **block** of `val` bindings + a final expression,
byte-identical Core IR vs `ssc1-front`.

**Approach — offside in the parser, not synthetic tokens.** `ssc1-front`'s `layout` pass rewrites the
token stream, injecting virtual `{ ; }` tokens; for a *lossless* UniML CST we must not fabricate
tokens, so the spike instead computes block structure from token **columns** in the RD parser: after
`=`, a body on a **later line** opens a `spike.block` at that token's column; statements continue
while `col >= blockCol`; a dedent (`col < blockCol`), EOF, or a top-level `def` ends it. Statement
separation is implicit — `parseExpr` stops at a non-operator, and a line **starting with an infix
operator** is consumed as a continuation of the previous statement (matching `ssc1-front`'s `isCont`
rule for the common cases). The block frame is emitted through the same uniform "open-on-first /
`Reframe.closeAfter`-on-last" mechanism; a multi-block dedent naturally accumulates multiple closes on
one token. So UniML's `Reframe` still expresses layout — the dialect just decides *where* from columns.

**Projection = `ssc1-front`'s block AST.** `spike.block` → `Pair("block", [stmt…])`, `spike.val` →
`mkVal(name, rhs)`, final `spike.exprStmt` → `mkSExpr(e)`. The **unchanged** `ssc1-lower`'s
`lowerBlock` folds these into nested single-binding `IrLet`s (source order, de-Bruijn 0 = most
recent), final expr bare — exactly what `ssc1-front` produces from its `{ ; }` stream.

**Proven (20/20 tests + e2e).** CST: indented body → `block(val, val, exprStmt)`; inline body stays a
bare expr (no block); a leading-operator continuation line stays one statement; a dedent to a
top-level `def` ends the block (both defs survive). **Differential oracle:** `block-vals` (→12),
`block-single` (→3, collapses to inline IR), `block-cont` (leading-`+` continuation, →30) all produce
**Core IR byte-identical to `ssc1-front`**. Deferred: nested/`if`-`then`-`else` layout blocks, `var`,
non-final expr statements, and the full `canEndLine`/`canStartLine`/`isCont` continuation matrix
(current handling covers leading-operator continuations; other continuation starters — `.`/`::`/`->`/
`else`/`match` — arrive with those constructs). Next: P6.2c (`::`/`->`/`to`/`until` + prefix ops,
then patterns / `match` / `given`-`using` / declarations).

## P6.2c (slice 3, part 1) — `match` + patterns

Status: **GREEN**. `EXPR match { case … }` (braced) and offside `EXPR match` ⏎ indented `case`
arms, with int-literal / `_` / variable-binder patterns and `if`-guards — byte-identical Core IR vs
`ssc1-front`. (Constructor patterns `Some(x)`/`Cons(h,t)` — needing uppercase `uid` lexing — are the
next sub-slice.)

**Design.** `match` is a **postfix on the atom** (`parsePostfix` wraps `parseAtom`), mirroring
`ssc1-front`'s `buildPostfix` so its precedence vs. infix ops is identical. The braced form consumes
`{`/`}` (now lexed); the offside form has no braces — the arm loop just runs `while isKw "case"`
(dedent/EOF/next-`def` ends it), which reaches the same arm parser as the braced form (exactly how
`ssc1-front`'s layout pass makes both forms converge on `parseMatchArms`). Patterns: `pat.lit`→
`Pair("lpat", Pair("int", n))`, `_`→`Pair("wpat","")`, binder→`Pair("vpat", name)`; an arm is the
bare 2-tuple `Pair(pattern, body)` (no tag), and a guard folds in as `Pair("gpat", Pair(pat, g))` —
all mirroring `ssc1-front` so `lowerMatch`/`lowerLitArms` produce identical IR (an all-literal match
lowers to `IrLet` + an `IrIf`/`__eq__` chain, no `IrMatch`).

**Proven (23/23 tests + e2e).** CST: offside literal+wildcard arms, braced form parses the same arms,
variable-binder + guard captured. **Differential oracle:** `match-lit` (→42), `match-var` (var binds
scrutinee, →107), `match-guard` (`if n > 5`, →16), `match-braced` (→30) all produce **Core IR
byte-identical to `ssc1-front`**. Deferred: constructor/tuple/alternative/typed/bind patterns
(`cpat`/`apat`/`tpat`/`bpat`), block/multi-statement case bodies, and `uid` (uppercase) lexing for
ctor patterns + applications — the next sub-slice.

## P6.2c (slice 3, part 2) — constructor & tuple patterns + `uid` + tuple literals

Status: **GREEN**. Constructor patterns (`Some(x)`/`Cons(h,t)`/nullary `None`/`Nil`), tuple patterns
`(a, b)`, uppercase-constructor applications (`Some(5)`), and tuple literals `(4, 5)` — all
byte-identical Core IR vs `ssc1-front`.

**Design.** The lexer now classifies an uppercase-initial identifier as **`spike.uid`** (Int, Some,
Cons) vs lowercase `spike.id` — so `def` type positions (`Int`) had to accept `uid` too (an
`expectType` helper). A `uid` projects to `mkUVar` and `Some(5)` to `mkApp(mkUVar("Some"), …)`; the
resolve pass turns bare `uid` `None`/`Nil` and `uid`-applications into `ctorap`/`IrCtor`, matching
`ssc1-front`. Pattern side: `spike.cpat` → `Pair("cpat", Pair(name, [subpats]))` (recursive), a
nullary ctor → `Pair("cpat", Pair(name, Nil))`; a tuple pattern `spike.tuppat` lowers exactly as
`ssc1-front` does — `Pair("cpat", Pair("Pair", [a,b]))` for a 2-tuple, `"TupleN"` for ≥3, collapse
for 1. Tuple **literals** `(a, b)` (spike.tuple → `mkTup([elems])`) round out the pair, so a tuple
pattern has something to match; `(e)` stays grouping (spike.paren).

**Proven (25/25 tests + e2e).** CST: `Some(x)`/`None` → `spike.cpat` with the ctor name; `(a,b)` →
`spike.tuppat`; `Some(5)` projects to `mkApp(mkUVar("Some"),…)`. **Differential oracle:** `ctor-some`
(→5), `ctor-none` (→42), `ctor-cons` (→3), `tuple-pat` (`(4,5)` matched by `(a,b)`, →9) all produce
**Core IR byte-identical to `ssc1-front`** (and the paren/grouping path is unregressed). Deferred:
alternative/typed/bind patterns (`apat`/`tpat`/`bpat`), nested-obligation ctor fields, block case
bodies. Next families: `::`/`->`/`to`/`until` + prefix ops; `given`/`using`/`extension`;
`enum`/`trait`/`object`/`type` declarations.

## P6.2d (slice 4) — `case class` declarations + field access

Status: **GREEN (Core IR byte-identical)**. `case class Name(f1: T1, f2: T2)` top-level declarations
+ `.field` access — the projection is byte-identical to `ssc1-front`.

**The pleasant surprise:** *all* the declaration machinery — the constructor `IrDef`, the `Mirror`
metadata def, the `_sel_<field>` accessor defs (each an `IrMatch` extracting field N), and the
start-up `__regfields__` registration — is generated **by `lowerProg` itself** from the `casecls`
AST node. So the projection only has to emit the right node: `mkCaseCls(name, [fieldNames],
[fieldTypes], Nil)` → `Pair("casecls", …)`, and `.x` → `mkSel(obj, "x")` → `Pair("sel", …)`.
`lowerProg`'s registry (`caseFieldsCell`/`caseFieldOrderCell`, populated before bodies are lowered)
turns `.x` into `IrApp(IrGlobal("_sel_x"), [obj])` and keeps a **user** ctor `Point(3,4)` as
`IrApp(IrGlobal("Point"), …)` (a call to the generated ctor def — NOT inline `IrCtor`, unlike
built-ins). Because our AST matches `ssc1-front`'s exactly, all of that is reproduced byte-for-byte.

Lexer/parser additions: `class` keyword; `.` (`spike.dot`); a top-level `case`→`parseCaseClass`
route; and a postfix loop for chained `.field` selections (`spike.sel`) that composes with `match`.

**Proven (26/26 tests + e2e).** CST: `case class Point(x,y)` → `spike.casecls` (name + 2 fields), the
body `Point(3,4).x` → `spike.sel`; projection emits `mkCaseCls("Point",…)` + `mkSel`. **Differential
oracle:** `cc-field`, `cc-arith`, `cc-match` (case-class construction + field access + ctor pattern on
a user type) all produce **Core IR byte-identical to `ssc1-front`**.

**Note on run-ir:** these toys can't execute on the *bare* kernel VM (`ssc.jar run-ir`) — it reports
`unimplemented primitive: __regfields__` (a plugin-runtime registration prim), **identically for both
`ssc1-front` and the spike** (the Core IR is the same). So the harness validates them by Core IR
byte-equality (the definitive parse/projection proof); running them needs the full plugin VM. Deferred:
`derives`, `extends`, type params, body methods, `enum`/`trait`/`object`/`type`.

## P6.2e (slice 5) — `given` instances + `summon` (typeclass resolution core)

Status: **GREEN (Core IR byte-identical + runs)**. `given name: T = expr` instances + `summon[T]`.

**The verdict that made this tractable** (from a dedicated tractability probe): typeclass instance
resolution / dict-passing is **entirely in `lowerProg`'s resolve pass**, driven by tables it rebuilds
from the AST — NOT in the parser. So `given`/`summon` are AST-node-driven like `casecls`: the
projection emits `Pair("given", Pair(name, Pair(typeStr, body)))` and `Pair("summon", typeStr)`, and
`lowerProg`'s `buildGivenTable` + `findGiven` + `summon` resolution do the lookup and produce the
`IrDef` dict + the resolved reference. The two genuine hidden side-channels in the family
(`usingSigCell` for named `using`, `tcExtendsCell` for `trait … extends …`) are **avoided** in this
slice, so no cell reproduction is needed.

Lexer/parser additions: `given` keyword; `[`/`]` brackets; top-level `given`→`parseGiven`; `summon`
intercepted in `parseAtom` (an id followed by `[T]` → `spike.summon`, else a plain var).

**Proven (27/27 tests + e2e).** CST: `given g: Int = 42` → `spike.given`, the body `summon[Int]` →
`spike.summon`; projection emits `Pair("given", Pair("g", Pair("Int", …)))` + `Pair("summon","Int")`.
**Differential oracle:** `given-summon` (→**42**) and `given-summon2` (`summon[Int] + 1` →**8**) both
produce **Core IR byte-identical to `ssc1-front`** AND run correctly on the bare VM (unlike case
classes — `given`/`summon` need no plugin prim). This proves the dict-passing resolve path works
end-to-end via the AST-node → `lowerProg` approach. Deferred (harder / hidden cells): the full loop
`trait Show[A]` + `given … with { }` + context-bound `[A: Show]` + `summon[Show[A]].show(x)` dispatch
(context-bound needs the deterministic `[A:TC]`→`__tc_TC`-param rewrite; the `with`/dispatch forms
were finicky even in `ssc1-front`), named `using`, `extension`, and `enum`/`trait`/`object` decls.

## P6.2f–i (slice 6) — enum, extension, pattern completion, `::`/`->` (remaining families batch)

Status: **GREEN**. Four more families, all byte-identical Core IR vs `ssc1-front`; 32 spike tests,
34-toy corpus, 0 diffs.

- **P6.2f `enum`** — `enum E: case A; case B(x: Int); case Red, Green` (offside/braced + comma-nullary
  sugar) → `("enum",(name,[(caseName,[fieldNames])…]))`; `lowerProg` reuses the case-class ctor path.
  `Cur.peek2Lexeme` distinguishes a trailing top-level `case class` from an enum case. (`enum-nullary`,
  `enum-params` — Core IR ≡ `ssc1-front`; run-ir needs the plugin VM like case class, `__regfields__`.)
- **P6.2g `extension`** — `extension (recv: T)` ⏎ indented `def m: R = body` → three statements
  (`extension_start`, the method def with the receiver **prepended** to its params, `extension_end`);
  `lowerProg`'s `collectExtensionMethods` registers `m` for `.m` dispatch. Also lifted: **parameterless
  defs** (`def f: T = e`, optional param clause). Offside form used — an inline `extension … def …`
  has an `ssc1-front` boundary quirk that absorbs the next `def`. (`ext-method`: `5.double` → **10**,
  Core IR ≡, runs.)
- **P6.2h pattern completion** — alternatives `case A | B | C` (`apat`) and binds `case n @ P`
  (`bpat`), via a `parseArmPattern` wrapper; lexer gains `@`. (`pat-alt` → **100**, `pat-bind`
  `n @ Some(v)` → **9**, Core IR ≡.)
- **P6.2i `::` / `->`** — `::` (prec 5, **right-associative**) and `->` (prec 1) added to the Pratt
  `opPrec`; the existing `spike.infix` projection emits `mkInf(op,l,r)`, which the resolve pass turns
  into `IrCtor("Cons")` / `IrCtor("Pair")`. (`op-cons` `1 :: 2 :: Nil` → **1**, `op-arrow` `1 -> 2`
  matched by `(a,b)` → **3**, Core IR ≡.)

**Deferred (edge cases, honestly out of scope for the arc):** `to`/`until` ranges, prefix ops
(`-x`/`!x`), typed patterns `case p: T`, named `using` clauses + `trait … extends …` (the two hidden
side-channels `usingSigCell`/`tcExtendsCell`), and the full `trait`+`given … with`+context-bound
dispatch loop (finicky even in `ssc1-front`).

## Arc conclusion

Across **nine** grammatical families (P6.0 gate, P6.1 total error model, P6.2a infix, P6.2b offside,
P6.2c match+all-patterns, P6.2d case class, P6.2e given+summon, P6.2f enum, P6.2g extension, P6.2h
alt/bind patterns, P6.2i `::`/`->`) — spanning the entire core of the 2899-line hand-written
`ssc1-front` (expressions, precedence, layout, patterns, declarations, typeclass resolution) — the
UniML `ScalaSpike` dialect + projection produces **byte-identical Core IR** on a 34-program corpus,
with a total, error-resilient pipeline. The P6.0 gate question ("does UniML's model fit the whole
front") is answered comprehensively: **yes**, on every family attempted, including the most intricate
(dict-passing). The v2.2 self-hosted architecture — Pratt-inside-the-dialect + a lossless UniML CST +
a thin projection into the existing `ssc1-lower` — is validated end-to-end. What remains is bounded
polish (the deferred edge cases above), not architectural risk.

## P6.2j–k (slice 7) — prefix ops, ranges, typed patterns, type parameters

Status: **GREEN**. The remaining tractable edge cases, all byte-identical Core IR vs `ssc1-front`.

- **P6.2j prefix / range / typed** — prefix `-e`/`!e`/`~e` → `mkPre(op,e)` (lowers to `i.sub(0,e)` /
  `if e false true` / `i.not`); range words `a to b` / `a until b` → `mkApp(mkSel(a,word),[b])`
  (id-infix, loosest binding, only at the outer expr level); typed pattern `case p: T` →
  `Pair("tpat",(pat,typeHead))`. (`op-prefix` `-5+3`→**-2**, `op-range` `(1 until 5) match _`→**7**,
  `pat-typed` `x:Int` on a scalar→**0** — matches `ssc1-front`: a type-test on a scalar `Int` doesn't
  bind. Core IR ≡.)
- **P6.2k type parameters** — plain `def f[A](x:A)` / `case class Box[A](…)` / `enum Opt[A]:` — the
  `[ … ]` clause is erased (balanced-bracket skip), exactly as `ssc1-front` does for un-bounded params.
  (`tparam`: `given g: Int = 99; def useIt[A](x)=summon[Int]; useIt(5)`→**99**, Core IR ≡.)

### The honest boundary — what is genuinely out of reach for a byte-identical slice

Two remaining things are **not** deferred by choice but by nature:

1. **The full context-bound typeclass dispatch loop** (`trait TC[A]` + `[A: TC]` + `summon[TC[A]].m(x)`)
   — the minimal forms of this error **inside `ssc1-front` itself** (`unbound global: __summon_value_TC`
   / `__missing_tc_Show`); a *working* instance from the corpus needs higher-kinded `F[_]`, `given …
   with { }` method bodies, and multi-bound `[A: Monoid: Pretty]` together — a large, interacting
   feature set, not a clean single slice. The typeclass **resolution core** (given + summon on a
   concrete type) is already proven (P6.2e); the concrete-`summon` + type-param path also works (P6.2k).
2. **Named `using` clauses** and **`trait … extends …`** — the only two features whose parser writes a
   *non-reconstructable hidden `#cell`* (`usingSigCell` / `tcExtendsCell`) that `lowerProg` never
   rebuilds from the AST. Matching them would require the projection to `import ssc1-front` and
   replicate those `#cell.set` calls — a different integration model than the pure AST-node projection
   the whole spike is built on.

Everything the spike's architecture *can* reach byte-identically has been reached. The corpus is now
**38 programs, all Core IR byte-identical to `ssc1-front`** (37 diffs + 1 containment), across the
entire core of the hand-written front.

## P6.3 — unify the hybrid (dialect composition by injection)

Status: **GREEN**. A whole `.ssc` — YAML front-matter + Markdown prose + fenced ScalaScript — parses
as **one lossless UniML tree**, and the code half still compiles byte-identically.

New JVM-only module `unimlScala` (`uniml/scala/`, test scope) `dependsOn(unimlCross % "test->test",
unimlMarkdown, unimlYaml)` — the one place that can see the ScalaScript spike (core test), the Markdown
dialect, and the YAML dialect at once. `SscCompose.parse(source)`:

1. the **Markdown ScalaScript profile** frames the file (front-matter, prose, fences);
2. every `markdown.code-block` whose info string is ScalaScript (`scalascript`/`scala`/`ssc`/empty) has
   its fence body re-parsed by `SpikeDialect`; the resulting `spike.program` CST is spliced under the
   block (edge role `scalascript`) and the raw `code` leaves dropped;
3. the `markdown.front-matter` YAML body is re-parsed by `YamlDialect` and spliced in (edge role `yaml`);
4. everything hangs under one synthetic `ssc.file` root.

**Injection, never reinterpretation** — each dialect only ever sees its own bytes (the fence body for
ScalaScript, the front-matter body for YAML), exactly as the spec's dialect-composition rule requires.
A foreign fence (e.g. ```` ```json ````) is left an inert `markdown.code-block` — no dialect re-lexes it.

Proven (7/7 `SscComposeSpec` + the differential harness):
- `hybrid-basic` (front-matter + heading + prose + ```` ```scalascript `def main(): Int = 2 + 3 * 4` ````)
  → the `.ssc` is one `ssc.file` tree carrying a `markdown.front-matter` (with a spliced YAML subtree), a
  `markdown.heading`, and a `markdown.code-block` (with a spliced `spike.program` — `+(2, *(3,4))`,
  precedence intact); the spliced subtree projects to **run-ir=14, Core IR ≡ `ssc1-front`**.
- `hybrid-cc` (one fence: `case class Point` + accessor) → Core IR ≡ `ssc1-front`.
- Extraction is **lossless**: `Composed.scalaSource` is exactly the bare `.ssc0` the fences carry
  (single trailing fence-EOL dropped; internal newlines and multi-fence order preserved).

The `.proj` fed to the harness is projected from the subtree **actually spliced into the composed `.ssc`
tree**, so the chain is closed end to end: wrapped `.ssc` → (lossless framing) → ScalaScript subtree →
(`SpikeProject`) → ssc0 → Core IR ≡ hand-written `ssc1-front`.

This is the "unify the hybrid" prize: the YAML/Markdown/code triad of a real `.ssc` is a single UniML
tree with no cross-dialect byte reinterpretation, and the code half is byte-identical to today's front.
(The spec's literal P6.3 interpolator injection — `s`/`f`/`md` — is the same boundary-injection mechanism
applied inside an inline token instead of a fence; the fence/front-matter composition proven here is the
harder, load-bearing case.)

### P6.3b — the registry hook (built-in set, user-closed)

Which dialect a fence (or front-matter) selects is now resolved through a `DialectRegistry`, not
hardcoded. `SscCompose.builtins` is the **closed** built-in set — ScalaScript (`SpikeDialect`, aliases
`scalascript`/`scala`/`ssc`), Markdown, YAML, JSON. `SscCompose.registryWith(extra*)` layers user
dialects on top but **cannot override a built-in name** (`DialectRegistry.register` returns `Left` on a
duplicate) — that is what "user-closed" means. `parse(source, registry = builtins)` consults it per
fence: a registered language is injected, an untyped ```` ``` ```` fence defaults to ScalaScript, and a
fence naming **no** registered dialect stays an inert `markdown.code-block`. Proven (10/10 `SscComposeSpec`):
a built-in ```` ```json ```` fence is injected as a `json` subtree via the registry (not ScalaScript);
`builtins` resolves exactly the four built-in languages; re-registering a built-in fails; a fresh
`MermaidDialect` registers and drives ```` ```mermaid ```` injection. The scala hybrid toys stay
byte-identical (`hybrid-basic→14`, `hybrid-cc`) under the registry-driven path.

### P6.3c — trailing newline is tolerated, not stripped

The spike lexer emits a trailing EOL as a `spike.ws` *Trivia* token that the parser's `skipTrivia()`
drops, so a trailing newline is already invariant to the projection (hence Core IR). `SscCompose` now
feeds the **raw, lossless** fence body (trailing EOL included) to `SpikeDialect` — tolerance proven, not
worked around — while `Composed.scalaSource` is the clean stripped accessor (presentation only). A new
`ScalaSpikeSpec` test asserts the projection is byte-identical with/without a trailing `\n`, `\n\n`,
`\r\n` (single- and multi-statement bodies). The harness now carries the EOL on *both* sides (the `.toy.ssc`
is `scalaSource + "\n"`, the `.proj` comes from the raw-body subtree), so byte-identity proves both the
spike **and** `ssc1-front` tolerate it.

### Still open in P6.3

**Inline interpolator injection** (`s`/`f`/`md`) — genuinely a bigger slice, deferred honestly. `ssc1-front`
*does* support strings + interpolation (`mkStr`, `interpParts`/`buildSInterp`, f-format, `md"…"` routing),
but it lexes an interpolated string as one `str` token and splits it at **lower** time — it does **not**
inject a sub-dialect at parse time. So a byte-identical interpolator slice means (1) adding string literals
to the spike grammar (escapes, triple-quote, `s`/`f`/`md` prefixes) and (2) replicating `ssc1-front`'s
interp lowering exactly. That is a real grammar extension, not a registry wiring — scoped as the next
slice. The registry hook above is the mechanism it will sit on (an interpolator prefix resolves to a
dialect the same way a fence info string does).
