# ScalaScript 2.1 core-free native Async provider

Status: TI-8.2d3i implementation contract (2026-07-12)

## Goal

The standard ScalaScript 2.1 launcher must execute built-in `Async` operations
under `runAsync` and `runAsyncParallel` on the shared native VM/direct-ASM
runtime. The provider must not load the compatibility frontend, `PluginBridge`,
v1 interpreter, Scalameta, Scala compiler, or Java compiler.

The release target is the complete `examples/async-demo.ssc` contract: the
ordered outputs `6`, `List(1, 4, 9, 16)`, `after delay`,
`List(20, 40, 60)`, and `56`. The real-thread
`examples/async-parallel-demo.ssc` surface must also execute deterministically;
concurrency is verified structurally with latches rather than timing thresholds.

## Baseline

Installed `origin/main` through `a64b738bb` has no standard native Async
runner. VM and direct ASM both exit 1 at the first `runAsync` block with
`unbound global: runAsync` and no stdout. The compatibility tier prints the
five expected lines above. Runtime taxonomy contains one blocking
standard-provider row for `async-demo.ssc`.

## Ownership and interface

The existing required `v2/runtime/std/effect-runners-plugin` owns the Async
runner because it already hosts portable standard effect runners over
`NativePluginContext.withEffect`. It registers:

- `runAsync(body)` for deterministic caller-thread execution;
- `runAsyncParallel(body)` for JDK 21 virtual-thread execution;
- dynamic operations `Async.delay`, `Async.async`, `Async.await`,
  `Async.parallel`, and the bounded `Async.recvFrom` bridge through the active
  `Async` effect context.

The self-hosted lowerer continues to represent `Async` as an empty portable
object value. The runtime's existing dynamic effect dispatch sends its methods
to the nearest active handler. No fallback global, stub, compatibility bridge,
or new core special case is allowed.

## Future and execution contract

An Async future is an opaque provider value backed by a completion cell.
ScalaScript source cannot observe or depend on the host representation.

- Under `runAsync`, `Async.async(thunk)` evaluates the thunk immediately on the
  caller thread and returns an already-completed future. `Async.parallel`
  evaluates its thunk list sequentially in declared order.
- Under `runAsyncParallel`, `Async.async(thunk)` starts one virtual thread and
  returns immediately. `Async.parallel` starts every thunk before joining and
  collects results in declared order regardless of completion order.
- Every child virtual thread reinstalls the same Async handler because native
  effect scope is thread-local.
- `Async.await(future)` joins the owned completion cell. Awaiting a non-future
  is an explicit error; no plain-value coercion is permitted.
- A thunk failure completes its future exceptionally and is rethrown at
  `await` or the enclosing `parallel` boundary with the owning Async operation
  named. Failures are never converted to Unit or an empty result.
- `Async.delay(ms)` accepts Int; positive values sleep and non-positive values
  complete immediately. Interrupted parallel tasks preserve interruption as a
  failure.
- Nested runners push/pop their own handler and finish completely before the
  outer computation resumes.

## `recvFrom` boundary

`Async.recvFrom(ws)` may call a plugin-owned named method object exposing a
zero-argument `recv`. Other receiver shapes fail with a bounded diagnostic.
This provider does not introduce a dependency on the HTTP/WebSocket provider
implementation and does not synthesize a compatibility socket value.

## Isolation

- The existing provider continues to depend only on plugin SPI, v2 core, Scala
  runtime, and JDK.
- VM, direct ASM, and `build-jvm` use the same provider and portable values.
- No new standard jar is introduced; the required provider is already part of
  standard/slim distributions and artifact allowlists.
- The Scala 3 seed remains unchanged and no compiler-backed regeneration runs.

## Verification

- [ ] Unit tests cover deterministic async/await/parallel, true concurrent
  start with ordered results, nested runners, delay validation, malformed
  futures/thunks, callback errors, and the named-method `recvFrom` boundary.
- [ ] A focused assembled fixture is exact on VM/direct ASM and exercises
  sequential, parallel, nested, and propagated-value behavior without a
  wall-clock assertion.
- [ ] `examples/async-demo.ssc` is exact on VM/direct ASM/build-jvm; the
  parallel demo runs with deterministic output on the native engines.

  deterministic artifact, corpus/parity/taxonomy, and fresh `v2-*` conformance
  gates pass.

  parity remain zero.

## Out of scope

- Timeouts, cancellation tokens, worker pools, priorities, and JS worker
  threads.
- Making `Async` a platform type or exposing `CompletableFuture` to `.ssc`.
- Transparent fallback to compatibility implementations.
