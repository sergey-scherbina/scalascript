# ScalaScript 2.1 native `md` interpolator

## Overview

The standard ScalaScript 2.1 native frontend must implement the normative
`md"..."` string interpolator entirely inside the self-hosted ScalaScript
frontend. It combines the ordinary interpolation grammar with deterministic
indent stripping and must not resolve `md` as a runtime global, provider, or
host parser.

This closes the remaining half of the public content-helper example after the
core-free host provider supplied `doc(...)` and `render(...)`.

## Interface

```scalascript
md"literal $name ${completeExpression}"
```

`md` is the reserved built-in prefix specified by `SPEC.md` section 5.7. Its
result is `String`. `$name` and `${...}` use the same self-hosted interpolation
grammar and value-to-string behavior as `s"..."`; a braced expression may
contain balanced parentheses, braces, and quoted strings.

After interpolation, the result is normalized as follows:

1. split on LF;
2. remove leading and trailing blank lines;
3. find the minimum leading ASCII-space count among nonblank body lines;
4. remove that count from every nonblank body line and normalize blank body
   lines to empty strings;
5. join the remaining lines with LF, without adding a trailing LF.

An empty or all-blank body produces the empty string.

## Behavior

- [ ] A single-line `md` string preserves interpolated text and requires no
      runtime global named `md`.
- [ ] A multiline `md` string removes blank edge lines and common indentation
      after evaluating `$name` and `${expr}` with ordinary string semantics.
- [ ] Uneven indentation removes only the minimum common ASCII-space prefix;
      internal and relative indentation remains intact.
- [ ] Native VM, direct ASM, and deterministic `build-jvm` produce
      byte-identical focused output and complete `examples/content.ssc`.
- [ ] Non-`md` interpolator prefixes retain their existing ordinary dispatch;
      the change does not reserve or rewrite arbitrary user-defined prefixes.
- [ ] The standard artifact remains free of Scalameta, v1 frontend/parser
      classes, and new parser/runtime dependencies.

## Out of scope

- Changing the permanent Scala 3 seed or compatibility frontend.
- Adding a Markdown parser, renderer, AST, or CommonMark dependency.
- Changing the already-landed provider contract for `doc` / `render`.
- Generalizing the self-hosted interpolation grammar beyond the existing
  `s"..."` behavior.
- Reworking user-defined `StringContext` extension dispatch.

## Design

`v2/lib/ssc1-front.ssc0` already owns complete self-hosted interpolation
splitting and expression parsing through `buildSInterp`. It will recognize a
string token immediately following the reserved `md` identifier, build the
same concatenation expression, and wrap that expression directly in the
existing `prim("__mdStrip__", ...)` AST shape.

The generic self-hosted lowerer already preserves and lowers `prim` nodes, and
both VM and ASM already implement `__mdStrip__`. Emitting the primitive at the
language front avoids both an unbound global and a modification to
`v2/lib/ssc1-lower.ssc0`, which is independently owned by the live parser
recovery slice.

## Decisions

- **Reserve only `md` at the self-hosted front** — chosen because the global
  spec defines it as a built-in language interpolator. Rejected: a provider or
  runtime global named `md`, which would make pure syntax depend on artifact
  composition and permit lexical-handler ambiguity.
- **Reuse `buildSInterp` before stripping** — chosen so `$name`, `${expr}`, and
  display conversion cannot drift from ordinary interpolation. Rejected: a
  second Markdown-specific interpolation parser.
- **Reuse `__mdStrip__`** — chosen because it already matches the compatibility
  frontend and v1 behavior on VM/ASM. Rejected: duplicating indent logic in the
  parser or adding a host library.
- **Do not edit the generic lowerer** — chosen because a direct `prim` node is
  already a valid frontend/lowerer contract and keeps this slice disjoint from
  the active parser-recovery claim.
