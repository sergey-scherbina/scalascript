# ScalaScript 2.1 core-free distributed local-loopback provider

Status: TI-8.2d3k implementation contract (2026-07-12)

## Goal

The standard ScalaScript 2.1 launcher must execute the canonical distributed
join and log-aggregation examples against fixed local inputs on native VM,
direct ASM, and `build-jvm` without `PluginBridge`, the v1 actor scheduler,
Scalameta, or a compiler.

This provider implements the examples' deliberately local
`localLoopbackCluster` contract. It does not claim network or failure-detector
semantics.

## Baseline and acceptance data

## Ownership and interface

A required provider at `v2/runtime/std/distributed-plugin` owns:

- `NamedHandler` and the `HandlerRegistry` method object (`register`, `lookup`,
  `apply`, `applyPredicate`, `applyKey`, `applyCombine`, `clear`, and
  `registeredNames`);
- portable `Node`, `MapOp`, `FilterOp`, `FlatMapOp`, `Stage`, and
  `ShuffleStage` constructors;
- `localLoopbackCluster`, `runDistributed`, and `runDistributedShuffle`;
- portable `DistributedResult(results, failures)` fields and the local
  cluster's idempotent `close` method.

Every operation validates its argument shape. Missing handlers, invalid stage
operations, and use of a closed cluster fail explicitly; none becomes Unit,
`Stub`, `Op`, or a compatibility value.

The self-hosted frontend represents a method on an imported uppercase object
as structural `__method__(name, ctor Object, args...)`. Therefore the provider
exports each `HandlerRegistry` operation under the exact
`HandlerRegistry.<operation>` intrinsic ABI in addition to the first-class
method-object value used by direct provider consumers. This is explicit
provider ownership, not an effect runner or a catch-all fallback: an unknown
`HandlerRegistry` operation must remain unhandled.

## Named-handler registry

- Registration is process-local and last-registration-wins by exact name.
- Registry iteration is insertion ordered. Replacing a name retains its
  original position; `clear` removes every entry.
- A handler is an ordinary portable callback and is invoked through
  `NativePluginContext.invoke`; closures are never serialized.
- Provider installation starts from an empty registry. Normal standard CLI
  invocations install a fresh provider; tests also prove explicit clear and
  reinstall isolation.

## Local distributed execution

- A local-loopback cluster retains source-order nodes and is open until
  `close`; close is idempotent.
- Input is partitioned into contiguous ceiling-sized chunks for the declared
  node count. Empty input remains one empty partition. Each partition executes
  its stage in node/partition order, and collected output concatenates in that
  order.
- `MapOp`, `FilterOp`, and `FlatMapOp` resolve the named callback on every
  application. Flat-map handlers must return a portable List.
- The local provider has no remote failures: successful results expose
  `failures = Nil`. Handler and shape failures propagate immediately; retries
  and `allowPartial` are accepted for source compatibility but cannot hide a
  deterministic local error.

## Shuffle execution

- `preShuffle` runs before key extraction.
- Keys retain first-seen order across the concatenated partition output.
- Group-by returns `(key, values)` pairs with values in source order.
- Reduce-by-key folds values left-to-right with the named combine handler and
  returns `(key, reducedValue)` pairs in first-key order.
- An empty combine name is valid only for group-by. Missing key/combine
  handlers and invalid operation/result shapes are explicit errors.

## Isolation and packaging

- The provider depends only on plugin SPI/v2 core, Scala runtime, and JDK.
- VM/direct ASM/build-jvm use the same provider and portable values.
- Standard/slim distributions and deterministic artifacts require the provider
  jar and ServiceLoader entry.
- Scala 3 seed/image files remain unchanged.

## Out of scope

- Actor/network transport, remote workers, discovery, cluster membership,
  retries after remote failure, partial remote results, durable queues, and
  code/closure shipping. These require an explicit advanced provider and never
  trigger transparent compatibility fallback.
