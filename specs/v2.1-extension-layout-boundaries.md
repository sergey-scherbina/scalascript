# ScalaScript 2.1 — extension layout boundaries

## Overview

An indented `extension` declaration owns only the definitions in its layout
body. The native 2.1 frontend must close that body at dedent, including a
Markdown code-block boundary, so a following top-level `def` does not inherit
the extension receiver. This restores source-declared function arity and
unblocks the standard parser-combinator runtime without changing constructor or
match semantics.

## Interface

```scalascript
extension [A](p: Parser[A])
  def map[B](f: A => B): Parser[B] = PMapped(p, f)
  def many(): Parser[List[A]] = PMany(p, 0)

def runParser(p: Any, input: String, pos: Position, ctx: Any): Any =
  p match
    case PRegex(pattern) => input.matchPrefix(pattern)
    case _               => None
```

The extension members receive the declared receiver `p` as their owned first
parameter. The dedented `runParser` declaration has exactly its four written
parameters; no extension receiver is prepended. Ending the fenced code block
that contains the extension body is also a body boundary because the next code
unit resumes at the enclosing indentation.

## Behavior

- [ ] Every indented member of one extension declaration receives its declared
      receiver and remains callable through selected extension syntax.
- [ ] Dedent closes the extension body before a following top-level `def`.
- [ ] A Markdown code-block boundary cannot extend receiver ownership into the
      next top-level code unit.
- [ ] The following top-level function retains its exact source arity in
      emitted CoreIR.
- [ ] Type parameters and nested parameter/function types in the extension
      receiver header do not close or open the body prematurely.
- [ ] Constructor tags, arities, and match arms remain unchanged; a correctly
      passed `PRegex/1` value selects the existing `PRegex` arm.
- [ ] A focused fixture produces identical exact output on native VM and direct
      ASM for multiple extension members plus a dedented constructor match.
- [ ] The calculator, JSON, and YAML-like parser-combinator examples leave the
      former `match: no arm for PRegex/1` boundary on both lanes; any later
      independent failure is tracked separately.
- [ ] Native-entry, corpus/parity/taxonomy, release, and fresh affected
      conformance gates remain green.

## Out of scope

- Changing pattern matching or the `PRegex` data constructor. Their emitted
  CoreIR is already structurally correct.
- General implicit search or extension-overload resolution.
- Adding a host parser-combinator implementation or example-specific globals.
- Redesigning braced extension syntax beyond behavior already accepted by the
  frontend.

## Design

The self-hosted parser currently keeps the active extension receiver in a
mutable cell and clears it when a non-`def` statement is parsed. That is
insufficient for significant indentation: both member definitions and a later
dedented top-level definition begin with `def` after physical newlines have
been normalized.

The layout pass must recognize the receiver clause that belongs to an
`extension` header and open a virtual layout body only after its closing
delimiter is followed by a newline. Existing offside handling then emits a
virtual close at dedent. The extension parser consumes the virtual opener,
keeps receiver state while body definitions are parsed, and clears it at the
matching virtual close. Nested delimiters in receiver types remain ordinary
delimiter stack entries and cannot masquerade as the receiver-clause close.

This keeps the current flat definition IR and receiver-prepending lowerer; only
the lexical lifetime of receiver state changes.

## Decisions

- **Layout owns the boundary** — derive extension lifetime from the same
  indentation machinery as other optional-brace declarations. Rejected:
  relying on the next non-`def` statement, because a valid top-level definition
  is indistinguishable from a member after newlines are discarded.
- **Receiver-clause close is contextual** — mark only the outer receiver
  delimiter as the possible layout opener. Rejected: treating every `)` after
  `extension` as an opener, which breaks nested function/tuple types.
- **Fix arity, not matching** — preserve the already correct constructor arm
  and pass the intended scrutinee. Rejected: weakening match failure or adding
  a special `PRegex` runtime case, which would hide the shifted argument list.
