# ScalaScript 2.1 native `doc` / `render`

## Overview

The standard ScalaScript 2.1 native path must provide the long-standing core
`doc(...)` builder and `render(...)` output helper without loading the v1
interpreter's `DocV`, `PluginBridge`, Scalameta, or any parser/renderer library.
They are general runtime helpers: unlike `contentDocument` and
`contentToMarkdown`, they do not inspect or parse Markdown content.

This slice deliberately separates provider ownership from the independent
`md"..."` language-lowering fix while another active worktree owns the
self-hosted lowerer.

## Interface

```scalascript
def doc(parts: Any*): Doc
def render(value: Any): Unit
```

`Doc` is opaque at source level. On the native runtime boundary it is an
ordinary reserved-tag value containing the supplied `ssc.Value` parts in
source order. The representation contains no v1 value or host parser product.

The native host provider registers `doc` and `render` as provider handlers.
The existing self-hosted `App(Global(name), args)` fallback reaches a handler
only when the program has no top-level definition, runtime global, or other
lexical owner for that name. This supplies any-arity `doc` without inserting
either helper into the shared plugin-global registry. Both VM and direct-ASM
calls resolve through the existing `ssc.plugin.NativePluginContext`; neither
helper becomes a kernel primitive or built-in global.

The v2 kernel exposes its existing user-display operation as a read-only
runtime utility so the host provider and `println` cannot drift. This is not a
new global, primitive, parser, or SPI mutation surface.

## Behavior

- [x] `doc()` and `doc(part1, ..., partN)` preserve zero or more ordinary
      runtime values in source order without eagerly converting them to text.
- [x] `render(docValue)` displays each part with the established user-visible
      scalar/container formatting, joins adjacent parts with exactly one LF,
      and writes exactly one final LF.
- [ ] A `Doc` nested directly or transitively inside another `Doc` is flattened
      recursively in source order. The reserved `NativeDoc` representation is
      never passed to the generic ADT display function or exposed in output;
      empty nested documents contribute no text and no extra separator.
- [x] `render(nonDocValue)` has the same visible output as `println(value)` and
      returns `Unit`.
- [x] Focused assembled programs produce byte-identical output on native VM,
      direct ASM, and deterministic `build-jvm` artifacts.
- [x] User-defined functions named `doc` or `render` retain ordinary lexical
      ownership; plugin globals are only fallback globals when the program has
      no definition of that name.
- [x] The host provider and its artifact contain no v1 `DocV`, `PluginBridge`,
      Scalameta, CommonMark/Flexmark, ujson/upickle, or other parser/renderer
      dependency.

## Out of scope

- Parsing, validating, or rendering Markdown syntax.
- Structural content lookup and `contentToMarkdown`; those remain owned by
  `v2/runtime/std/content-plugin`.
- The built-in `md"..."` interpolator; it is an independently landed
  self-hosted language feature rather than provider behavior.
- Removing or rewriting the permanent Scala 3 seed.
- Adding `doc`/`render` to the v2 kernel's primitive/global table.

## Design

1. Reuse `v2/runtime/std/host-plugin`, which already owns standard process and
   output-adjacent runtime behavior and is present in VM/direct-ASM/artifacts.
2. Encode a document as a reserved ordinary `ssc.Value.DataV` whose fields are
   the unevaluated-after-call (but already strict) part values.
3. Recursively flatten only provider-owned `NativeDoc` values. Reuse the
   kernel's existing display function for every non-document leaf, join the
   resulting leaf strings with LF, then emit through the process stdout owned
   by the JVM host provider. An empty document has no leaves; rendering it
   still follows the public one-final-LF contract.
4. Prove the plugin boundary with focused provider tests and the assembled
   dependency/class-load gates; do not add a new module or dependency.

## Decisions

- **Existing host provider owns the helpers** — chosen because `render` performs
  process output and `doc` exists only to feed that general output contract.
  Rejected: structural content provider (wrong API ownership), and kernel
  intrinsic/global (violates plugin ownership).
- **Handler fallback, not plugin globals** — chosen because the runtime already
  redirects unresolved ordinary calls to provider handlers after checking
  lexical/top-level/global ownership. Rejected: registering `doc`/`render` in
  the shared plugin-global registry; a full release gate proved that adding the
  generic `render` name there perturbs existing reactive execution despite the
  focused content case passing.
- **Ordinary reserved-tag value** — chosen because it is portable across VM and
  ASM and requires no v1/host hierarchy. Rejected: v1 `DocV`/`ForeignV`, which
  would reintroduce the compatibility graph and make artifacts host-shaped.
- **One shared display implementation** — chosen to preserve `println` parity
  for strings, numbers, ADTs, lists, maps, and exact decimals. Rejected: a
  second renderer in the plugin, which would drift and duplicate semantics.
- **Provider-local recursive flattening** — chosen because only the provider
  owns and can distinguish the reserved `NativeDoc` representation. Rejected:
  teaching the kernel display function about it (would make a core primitive
  out of provider state) and displaying nested docs as arbitrary ADTs (leaks
  `NativeDoc(...)` in the public `examples/content.ssc` output).
- **No parser involvement** — chosen because `doc` composes values and `render`
  displays them. Rejected: interpreting strings as Markdown, which belongs to
  neither helper and would violate the self-hosted parser boundary.

## Results

Implemented 2026-07-11. The existing host provider owns lexical-safe `doc` and
`render` handlers; the VM compiler and direct-ASM runtime resolve handlers only
after program/global ownership. Both lanes reuse the kernel display function.

The fresh-install release gate also exposed a pre-existing K62.33 regression:
`Signal` constructors bypassed the core-free reactive provider and built legacy
cells. VM and ASM now prefer the registered provider and retain the raw cell
only for bare-kernel execution.
