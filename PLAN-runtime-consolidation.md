# Runtime consolidation + v1.17.4 Http/Ws JVM transports

## Problem

Today there are TWO independent implementations of the HTTP/WS server stack
that ssc-generated JVM code relies on:

| | `backend-interpreter/src/main/scala/scalascript/server/*` | `backend-jvm/src/main/scala/scalascript/codegen/JvmGen.scala` `serveRuntime` |
|---|---|---|
| LOC | ~4 500 across 20 files | ~2 250 in one string literal |
| Shape | Real Scala classes | String template emitted into generated `.scala` |
| Tests / IDE | Yes (sbt test target) | None (string literal) |
| Runtime | Interpreter, in-process | scala-cli, recompiles per run |
| Drift risk | High — bug fixes must be duplicated |

The JS backend (`JsRuntimeMcp.scala`, etc.) is intrinsically different (Node http)
and stays out of scope of this consolidation.

## Goal

ONE implementation of the server stack used by both
`backend-interpreter` and code generated by `backend-jvm`.  After this lands,
v1.17.4 (MCP `Transport.Http` + `Transport.Ws` on JVM) becomes ~50 LOC of
adapter, not a new HTTP/WS stack.

## Phase 1 — Extract pure primitives

Files in `backend-interpreter/src/main/scala/scalascript/server/` that have
**no interpreter coupling** (no `import scalascript.interpreter.*`, no
`Value`/`Computation` references):

- `WsFraming.scala`   — pure RFC 6455 frame codec
- `Password.scala`    — Argon2 / bcrypt helpers
- `RateLimit.scala`   — token bucket
- `Totp.scala`        — RFC 6238 TOTP
- `Jwt.scala`         — HMAC JWT sign/verify
- `JwtRsa.scala`      — RSA JWT sign/verify
- `SessionCookie.scala` — cookie codec
- `SessionStore.scala`  — in-memory session store
- `Metrics.scala`     — counters / gauges
- `RestValidationError.scala` — exception type

Total ~1 100 LOC.

### Action

1. Create new sbt module `runtime-server-common`:
   ```
   lazy val runtimeServerCommon = project
     .in(file("runtime-server-common"))
     .settings(name := "scalascript-runtime-server-common", scalacOptions ++= sharedScalacOptions)
   ```
2. Move the files above (preserve `package scalascript.server`).
3. `backend-interpreter` adds `.dependsOn(runtimeServerCommon)` — its `import`s
   keep working unchanged.
4. `backend-jvm/JvmGen` reads the source of each file at codegen time
   (from the `runtime-server-common` jar's source attachment OR from a
   resource directory packaged into the jar) and inlines them in place of
   the corresponding section of `serveRuntime` (string template).
5. Drop the duplicated copies inside `serveRuntimePart1` / `Part1b` / `Part2`.

### Tests

Existing sbt tests in `backend-interpreter/src/test/.../scalascript/server/*`
cover the moved files; they keep running unchanged.
Conformance tests in `conformance/` exercise the generated JVM output;
re-run the suite to verify no regression.

## Phase 2 — Split interpreter-coupled server classes

Files that today reference `Interpreter`, `Value`, `Computation`:
`WebServer`, `WsConnection`, `WsRoutes`, `WsProxy`, `WsClientSession`,
`BlockingWsSession`, `OAuth`, `WebAuthn`, `TlsProxy`.  Total ~3 400 LOC.

The shape of these is: ~80% pure protocol logic (HTTP request parsing,
route matching, WS upgrade handshake, OAuth flow, WebAuthn registration,
TLS handshake), ~20% dispatch into user code (call a `Value.Closure` for
the interpreter, call a Scala `Function1` for the codegen output).

### Action

1. Introduce a `RouteDispatcher` trait in `runtime-server-common`:
   ```scala
   trait RouteDispatcher:
     def dispatch(method: String, path: String, request: Request): Response
     def dispatchWebSocket(path: String, session: WsSession): Unit
   ```
2. Refactor `WebServer.start(...)` to take a `RouteDispatcher`.  The
   interpreter provides one that evaluates `Value.Closure` handlers; the
   generated code provides one that calls plain Scala closures.
3. Move the refactored `WebServer`, `WsConnection`, etc. into
   `runtime-server-common`.
4. `backend-interpreter` reduces to:
   - `Request`/`Response`/`UploadedFile` case classes (already pure)
   - `InterpreterRouteDispatcher` (~150 LOC) bridging the `Value` model
5. `JvmGen.serveRuntime` reduces to:
   - emit the `Request` / `Response` case classes
   - emit a `CodegenRouteDispatcher` (~80 LOC) calling user closures
   - emit `WebServer.start(port, dispatcher)` call

Expected dedup: `serveRuntime` drops from ~2 250 LOC to ~300 LOC of glue.

## Phase 3 — MCP transports (v1.17.4)

With the runtime consolidated, add:

- `runtime-server-common/src/main/scala/scalascript/runtime/mcp/HttpSseMcpTransport.scala`
  - Implements `io.modelcontextprotocol.spec.McpServerTransportProvider`
  - On `setSessionFactory(f)`, starts a `WebServer` (from the consolidated runtime)
    with two routes: `GET <path>` (SSE upgrade) and `POST <path>?sessionId=X`
    (JSON-RPC inbound)
  - Per-connection `HttpSseMcpSession` implements `McpServerTransport`,
    writes outbound messages as SSE `data: <json>\n\n` frames
- `runtime-server-common/src/main/scala/scalascript/runtime/mcp/WsMcpTransport.scala`
  - Same shape, uses `WebServer` + `onWebSocket(path)` from the consolidated runtime
  - Each WS frame (text opcode) is one JSON-RPC message
- In `JvmRuntimeMcp.scala`, replace the two `throw McpError(...)` with:
  ```scala
  case "Http" => new HttpSseMcpTransport(port, path)
  case "Ws"   => new WsMcpTransport(port, path)
  ```
- Wire into `_mcpTransport` so the SDK's `JMcpServer.sync(spec).connect(provider)`
  picks them up.

### Smoke test

- `conformance/mcp-server-http.ssc` — start `serveMcp(Transport.Http(0, "/mcp"))`
  on a random free port, spawn a client via `mcpConnect(Transport.Http(port, "/mcp"))`,
  call a tool, assert response.
- `conformance/mcp-server-ws.ssc` — same shape for WebSocket.

## Sequencing

1. Phase 1 — own PR, ~4 hours, low risk.
2. Phase 2 — own PR (or two), ~1-2 days, higher risk: needs interpreter
   regression testing on every conformance test that uses `route` / `serve` / `onWebSocket`.
3. Phase 3 — own PR, ~3 hours.

Total ~3 days of focused work.  Each phase is independently shippable and
reviewable.

## Out of scope

- JS backend HTTP/WS implementation (separate codebase, no dedup possible).
- Publishing `runtime-server-common` to Maven Central (could be a v2.0
  follow-up if external consumers want it).
- Migrating non-server code (e.g., `Effects` runtime) — same dedup pattern
  could apply but is a separate effort.
