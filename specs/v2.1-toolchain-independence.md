# ScalaScript 2.1 — toolchain independence

Status: **in progress** (2026-07-10)

This specification defines the release-wide ScalaScript 2.1 production-path
contract. It complements the feature-specific `v2.1.x` Distributed Streams
documents; those version labels remain valid and do not define the toolchain
packaging contract.

## 1. Purpose

The standard JVM path must execute and package ordinary `.ssc` programs without
Scalameta, Scala CLI, the Scala compiler, javac, or the `java.compiler` module:

```text
.ssc source
  -> native Markdown/code frontend
  -> native type checker
  -> CoreIR
  -> v2 VM or direct ASM JVM bytecode
  -> result / deterministic JAR
```

Scala remains the implementation language of the host components in 2.1. A
prebuilt distribution may therefore contain the Scala runtime and ASM. Building
ScalaScript itself from source still requires a Scala compiler. This milestone is
about the installed compiler/runtime path, not a source-bootstrap rewrite.

## 2. Dependency tiers

The distribution has two explicit tiers. A standard command must never discover
and silently enter the tools tier.

| Tier | Required contents | Forbidden dependencies | Intended commands |
|---|---|---|---|
| **standard** | staged launcher, native frontend tower, native checker/lowerer, CoreIR VM, direct ASM emitter, scalameta-free plugin runtime, selected runtime plugins, Scala runtime, ASM | Scalameta family, Scala CLI, `scala3-compiler`, compiler driver, `javax.tools`, `jdk.compiler`/`java.compiler` | default `ssc run`, `ssc run --bytecode`, `ssc build-jvm` |
| **tools / compatibility** | everything needed by an explicitly selected legacy or platform compiler route | none beyond the normal security/reproducibility policy | `--v1`, `--compat-frontend`, Scala/Java fences, legacy `compile-jvm`, Spark, Scala.js/WASM source compilation |

Normative rules:

1. `PATH` lookup for `scala-cli`, `scalac`, or `javac` is forbidden in the
   standard tier.
2. Loading a Scalameta or compiler class reflectively is equivalent to a direct
   dependency and is forbidden in the standard tier.
3. A tools-tier feature requested from a slim installation fails before partial
   execution with a diagnostic naming the missing tier and installation remedy.
4. Standard execution must remain functional after physically deleting all
   Scalameta-family JARs and the installed compiler directory.
5. The standard launcher's reachable module graph must not require
   `java.compiler` or `jdk.compiler`.

## 3. Architecture and ownership

### 3.1 Standard path

The native frontend is the `mira-md -> ssc1-front -> ssc1-check -> ssc1-lower`
tower. The installed launcher runs the tower through a prebuilt kernel JAR and
staged `.ssc0`/runtime resources; it does not invoke Scala CLI over `v2/src`.

`ssc1-check` is **mandatory** before the native frontend becomes the default.
Dropping Scalameta while also bypassing type checking would replace a dependency
but regress the language contract. The checker may deliberately use `Dyn` at
documented plugin/FFI boundaries, but it must:

- accept every valid deterministic default-lane corpus program;
- reject the negative type-conformance corpus;
- produce source-located diagnostics;
- never emit runnable CoreIR containing the parser's `_err` sentinel.

Imports are resolved before type checking and lowering. Resolution is DFS,
canonical-path deduplicated, relative to the importing module first, and then to
the staged standard-library root. Multiple command-line source files participate
in one module closure. Program arguments after `--` are data, never source paths.

### 3.2 Compatibility frontend

The current v1-parser/Typer `FrontendBridge` remains an optional migration and
rollback component. Its AST and `BlockForm` adapters belong in the tools tier and
must not be referenced by standard runtime classes. It may be removed after one
release with a green native corpus and no unique compatibility cases.

### 3.3 Plugin runtime boundary

The standard v2 runtime owns a Scalameta-free value and intrinsic SPI. Runtime
values, callable values, effects, records, collections, errors, and native
contexts must be expressible without `scala.meta.Tree` or a v1 frontend AST.

Existing v1 `NativeImpl` plugins may be supported by an adapter in the tools tier.
That adapter is not allowed on the classpath of standard plugin tests. New plugin
work continues to live under `runtime/std/`; this specification changes the value
boundary, not the plugin ownership rule.

### 3.4 VM and ASM backends

Both standard backends consume the same checked CoreIR:

- the VM compiles CoreIR to closures and executes it;
- `v2JvmBytecode` emits JVM bytecode directly with ASM.

The old v1 `AsmJitBackend` is not the 2.1 artifact backend: it consumes the
Scalameta-coupled v1 runtime graph. The v1 `JavacJitBackend` and that legacy ASM
JIT remain tools-tier compatibility implementations only.

## 4. CLI and artifact contract

### 4.1 Execution

During migration:

```text
ssc run --native file.ssc [more.ssc ...] -- [program args ...]
ssc run --native --bytecode file.ssc [more.ssc ...] -- [program args ...]
ssc run --compat-frontend file.ssc    # tools tier: Scalameta bridge
ssc run --v1 file.ssc                 # tools tier: complete v1 rollback
```

At cutover:

- plain `ssc run` is identical to `ssc run --native` and uses the v2 VM;
- `--bytecode` selects direct CoreIR-to-ASM execution without changing frontend
  or program semantics;
- `--native` remains an accepted, idempotent assertion for CI and deployment;
- `--v2` remains a deprecated alias for the default v2 runtime and no longer
  implies the Scalameta bridge;
- the old bridge is selected only by `--compat-frontend`.

The selected route is observable with `ssc info --execution-plan ...` (or an
equivalent stable machine-readable command). Tests must not infer the route from
timing or classpath accidents.

### 4.2 JVM artifacts

The new direct-ASM artifact command is intentionally distinct from the existing
Scala-source-backed `compile-jvm [--bytecode]` path:

```text
ssc build-jvm file.ssc [more.ssc ...] -o app.jar
```

`build-jvm` must:

- run the native frontend and native checker;
- emit `.class` files directly through `v2JvmBytecode`;
- include a deterministic manifest/main entry, required v2 runtime metadata,
  linked module definitions, and selected standard plugins;
- preserve `.ssc` source/line mapping in JVM diagnostics;
- produce byte-identical JARs for identical inputs, options, and dependency set;
- run through `java -jar app.jar -- ...` without Scala CLI, scalac, or javac.

Legacy `compile-jvm`, `.scjvm`, Scala facades, and Scala-source passthrough remain
tools-tier compatibility surfaces until separately migrated. The word
`--bytecode` on those commands does not make them part of the native ASM path.

## 5. Compatibility policy

Every corpus row is classified as exactly one of:

1. **standard deterministic** — VM and ASM observable output must match;
2. **standard server/long-running** — readiness and bounded interaction are
   tested instead of process completion;
3. **standard nondeterministic** — normalized semantic assertions replace byte
   comparison;
4. **backend-specific tools tier** — Scala/Java fences, Spark, Scala.js/WASM, or
   another explicitly compiler-backed target;
5. **document/non-code** — no executable ScalaScript block;
6. **temporarily unsupported compatibility case** — named in the baseline with
   a stable diagnostic and owner; no silent `_err`, empty output, or success.

An increase in category 4–6 requires a spec change. A parser/lowerer exit code of
zero is not success when the emitted IR contains `_err` or unresolved parser
sentinels.

## 6. Portable gates

All gates resolve the repository from their own script path or Git worktree;
absolute developer paths are forbidden. Each gate writes a stable JSON or TSV
summary in addition to the human report.

Required gates:

- **native-front corpus:** extract/import/parse/check/lower/run classification for
  every `examples/*.ssc` input, including sentinel detection;
- **VM/ASM parity:** checked CoreIR executed by both standard backends, with the
  classifications in §5;
- **artifact reproducibility:** two clean `build-jvm` runs compared byte-for-byte
  and executed with `java -jar`;
- **slim-filesystem negative gate:** delete compiler and Scalameta JARs, hide
  Scala CLI/scalac/javac from `PATH`, then run hello/import/argv/plugin/JAR smokes;
- **JRE-module negative gate:** run the standard launcher/runtime with
  `java.compiler` and `jdk.compiler` unavailable and inspect it with `jdeps`;
- **classpath gate:** fail when a standard-tier JAR contains or references
  `scala.meta`, v1 frontend AST, compiler-driver, or `javax.tools` symbols.

## 7. Audited baseline (2026-07-10)

Baseline is recorded against `origin/main` through `1cc748f5c`, with native
module-loading work through `4815915ba`:

| Surface | Current state | 2.1 requirement |
|---|---|---|
| Direct v2 ASM emitter | CoreIR-to-ASM exists; execution is in-memory and reached from the Scalameta bridge in the public CLI | native checked CoreIR input plus deterministic JAR output |
| Native parse/lower | 194/195 files emit IR, but partial parses can still carry `_err` | no sentinel-bearing success; every row classified |
| Native checker | 162/195 pass, with 32 known false-positive rejections; standard run bypasses it | mandatory, corpus-valid, negative-type green |
| Native plugin-enabled run | 41/195 OK or bounded server timeout in the latest documented sweep | all standard rows green/classified on VM and ASM |
| Plugin boundary | `PluginBridge` reaches v1 `core`, interpreter `Value`, and Scalameta-coupled callable state | standard runtime/plugin graph has no v1 frontend or Scalameta edge |
| Public ASM run | `ssc run --bytecode` uses the v1 Scalameta frontend bridge | native frontend/checker by default |
| JVM artifact path | legacy `compile-jvm --bytecode` generates Scala and invokes compiler tooling | new direct-ASM `build-jvm` path |
| Installation | compiler and Scalameta families are staged with the full CLI | physically separable standard/tools tiers |
| Java compiler | v1 default JIT is javac-backed; direct v2 emitter itself needs only JVM base modules plus library dependencies | no `java.compiler`/`jdk.compiler` in standard reachable graph |
| Portable parity script | `scripts/bc-parity-sweep` contains a developer-specific worktree path | repository-relative, machine-readable gate |

This baseline is deliberately stricter than the earlier claim that the backend is
"almost ready": ASM emission is advanced, but the public frontend, plugin value
graph, artifact writer, packaging, and negative gates still define substantial
release work.

## 9. Decisions and rejected alternatives

- **Keep Scala as implementation/runtime language in 2.1.** Removing
  `scala-library` is independent of removing runtime compiler dependencies and
  would obscure the critical path.
- **Require the native checker.** Parser-only replacement was rejected because it
  would silently weaken the typed-language contract.
- **Use the v2 CoreIR ASM emitter.** Promoting the old v1 `AsmJitBackend` was
  rejected because its input/runtime graph remains coupled to v1 frontend values.
- **Add `build-jvm`.** Reusing `compile-jvm --bytecode` unchanged was rejected
  because that command currently means generated Scala plus compiler tooling.
- **Split the distribution.** Deleting every compiler-backed feature in one
  release was rejected; explicit tools-tier Scala fences, Spark, Scala.js/WASM,
  facades, and rollback remain useful and honest.
- **No transparent fallback.** Falling back to Scalameta or javac after a native
  failure was rejected because it makes production dependency guarantees
  untestable and deployment-specific.

## 10. Non-goals

- compiling the ScalaScript repository from source without scalac;
- removing the Scala runtime from produced applications;
- rewriting Spark, Scala.js/WASM, or arbitrary Scala/Java fences to ASM in this
  milestone;
- removing the compatibility tier before the native path has a release of field
  evidence;
- changing target-independent `.ssc` semantics to accommodate an ASM limitation.
