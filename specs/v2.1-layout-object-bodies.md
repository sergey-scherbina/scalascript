# ScalaScript 2.1 — layout object bodies

## Overview

ScalaScript accepts Scala-style significant indentation for singleton-object
bodies. The native 2.1 frontend must preserve every member of `object O:` and
lower it under the object's owner, just as it already does for an explicitly
braced object. This closes a frontend gap that currently removes the first
member and emits later members as unowned globals, blocking the pure
ScalaScript parser-combinator examples before the runtime is reached.

## Interface

Both declaration forms describe the same object-member scope:

```scalascript
object Parser:
  def char(c: Int): Int = c + 1
  def twice(c: Int): Int = char(char(c))

object Braced {
  def char(c: Int): Int = c + 1
  def twice(c: Int): Int = char(char(c))
}
```

A selector such as `Parser.char(1)` resolves to the owned member emitted by the
frontend (`Parser_char` in CoreIR). An unqualified call from one member to a
sibling remains in the same object scope and resolves to the same owner.

The trailing colon is a layout opener only while scanning a declaration header
that began with `object` or `trait`. Colons in type ascriptions, parameters,
return types, and other expressions are not layout openers.

## Behavior

- [x] `object O:` retains both its first member and every later member.
- [x] Layout-object members are emitted with the object owner and selectors
      resolve to `O_<member>` CoreIR globals.
- [x] A member can call a sibling member without an explicit owner qualifier.
- [x] Methods and parameterless properties in a layout object have the same
      behavior as their braced equivalents.
- [x] Explicitly braced object bodies remain unchanged.
- [x] Type-ascription and return-type colons do not open layout blocks.
- [x] A focused fixture produces identical exact output on the v2 VM and direct
      ASM lanes.
- [x] The calculator, JSON, and YAML-like parser-combinator examples advance
      past the former `unbound global: Parser_regex` / `Parser_char` failure;
      any newly exposed independent failure is classified separately.
- [x] The affected conformance and ScalaScript 2.1 release gates remain green.

## Out of scope

- Generalizing layout handling for `class`, `enum`, `given`, or extension
  declarations beyond behavior already supported by the native frontend.
- Adding a host-side regex or parser-combinator provider. The affected DSLs are
  ordinary portable ScalaScript code over the existing `String.matchPrefix`
  intrinsic.
- Changing the naming convention used by CoreIR for object-owned members.

## Design

The self-hosted frontend's layout pass converts physical newlines into virtual
braces and semicolons before parsing. It already tracks whether a trailing colon
belongs to a `trait` header so it can open a virtual body without treating every
colon as significant. Extend that declaration-header state to recognize
`object` as well. A newline after the contextual trailing colon opens one
virtual layout block; dedent closes it through the existing offside machinery.

The parser and lowerer then receive the same structural token shape as a braced
object. No object-specific recovery, member renaming, or runtime registration is
needed.

## Decisions

- **Contextual declaration colon** — extend the existing header-state mechanism
  because it preserves type-ascription syntax. Rejected: treating every colon as
  a layout opener, which would turn ordinary type syntax into blocks.
- **Frontend fix, not provider emulation** — preserve the user's object and
  member structure because parser combinators are pure language code. Rejected:
  registering `Parser_*` host globals, which would hide a general parsing bug
  and couple standard execution to example-specific names.
- **Layout/braced structural equivalence** — reuse the parser's established
  braced-object path rather than maintain a second object-body parser.
