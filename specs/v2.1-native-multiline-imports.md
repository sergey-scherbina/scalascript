# ScalaScript 2.1 native multiline Markdown imports

## Overview

Markdown import paragraphs are source syntax, and Markdown permits a link label
to wrap across physical lines. The ScalaScript 2.1 self-hosted module loader
currently scans one line at a time, so a valid wrapped selective-import label is
dropped before dependency resolution. The missing declarations later look like
unknown runtime providers; in `dsl-sql-recovery.ssc`, `Parser.regex` becomes an
unhandled fallback effect even though its pure companion is defined in
`std/parsing/core.ssc`.

The native loader must recognize wrapped import labels as one bounded Markdown
link, load their `.ssc` targets through the existing DFS, and preserve the same
declaration/static-dispatch semantics as a one-line import. This remains pure
self-hosted source processing with no compatibility frontend or host parser.

## Interface

These declarations are equivalent:

```markdown
[Position, Parser, PRegex](std/parsing/core.ssc)

[Position, Parser,
 PRegex](std/parsing/core.ssc)
```

The continuation is bounded by the matching `]` and immediate `(target)` of the
same standalone link. After that close, ordinary paragraph/fence scanning
resumes. Multiple complete import links in one pure physical line retain source
order exactly as before.

## Behavior

- [x] A standalone `.ssc` import whose selective label spans two or more
      physical lines enters the native DFS module closure exactly once.
- [x] The wrapped label does not change selected declaration names or target
      resolution; imported objects, constructors, definitions, and extension
      ownership behave exactly like their one-line form.
- [x] `Parser.regex` from a wrapped import resolves statically to the imported
      `Parser_regex` definition and returns the existing `PRegex` value; it is
      never converted to a fallback `Op`.
- [x] Existing multiple-links-per-line imports preserve source order.
- [x] Links inside fenced code and prose paragraphs that merely begin with `[`
      remain non-import content; an unterminated label cannot consume a later
      heading, fence, or unrelated paragraph.
- [x] A focused multi-file fixture produces exact identical VM/direct-ASM output
      for a wrapped companion import plus an existing same-line import.
- [x] Installed `examples/dsl-sql-recovery.ssc` advances identically on VM and
      direct ASM through its four wrapped imports and contains no
      `Parser.regex` fallback effect.
- [x] Module-loading, native-entry, corpus/parity, both taxonomies,
      standard/slim/JRE/dependency/build-jvm, and fresh affected conformance
      gates pass before its reviewed blocker is retired.
- [x] Scalameta, the v1 frontend, source rewriting, and transparent compatibility
      fallback are absent from the implementation and standard artifact.

## Out of scope

- General CommonMark inline parsing or changing mixed prose/link paragraphs
  into imports.
- URL/`dep:` fetching behavior, aliases, or selective-import enforcement beyond
  the existing module-loader contract.
- Host implementations of parser combinators, regular expressions, or recovery
  semantics.
- Treating every unknown uppercase selected call as an imported companion.

## Design

Keep the existing outside-fence line scanner and add a small explicit pending
link-label state. A trimmed line may begin a pending candidate only when it is a
standalone link prefix beginning with `[`. Subsequent outside-fence lines are
joined only until the first closing `]`; that close must be immediately followed
by a complete `(target)` before the accumulated text is offered to the existing
`sscImportPaths` parser. A heading, fence opener, blank paragraph boundary, or
invalid close clears the candidate instead of swallowing later content.

The completed import still flows through `sscLoadImps`/`sscLoadMod`, lexical
path normalization, load-once `seen`, `sscDefsOnly`, and the combined lowerer's
object/extension registries. No new runtime dispatch exists.

## Decisions

- **Bounded scanner state** — chosen because the self-hosted loader already owns
  fence exclusion, pure-paragraph recognition, and multiple-link ordering.
  Rejected: concatenating the entire document, which could turn prose or fenced
  examples into imports.
- **Restore module ownership before lowering** — chosen because `Parser` is an
  ordinary `.ssc` object and `PRegex` is an ordinary constructor. Rejected:
  registering `Parser.regex` as a provider or teaching effect handling about
  missing imports.
- **Language contract, not example repair** — wrapped link labels are valid
  Markdown source syntax and therefore apply uniformly to every module.

## Baseline

On installed standard main before this slice:

- native VM exits 1 with `ssc: unhandled runtime effect: Parser.regex`;
- direct ASM exits 1 with `ssc: match: no arm for Op/3`;
- structural compilation contains no `Parser_*` definitions and the source
  closure contains only `examples/dsl-sql-recovery.ssc`;
- explicit `ssc-tools run --compat-frontend` completes four cases and prints one
  `unknown parser node` error per case. This is a comparison baseline only, not
  an allowed fallback.
