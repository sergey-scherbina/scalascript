# ScalaScript 2.1 — extension receiver scope through nested layout

## Overview

Every extension member receives the declared receiver as its first runtime
parameter, including a receiver-only member with no explicit parameter list.
The self-hosted frontend currently clears receiver ownership when it encounters
any residual virtual `}`. A nested layout body inside one member can therefore
end the whole extension early: later members lower as zero-argument functions
whose bodies still reference an unbound receiver global.

This slice makes the extension's real body close distinct from nested layout
closes. It generalizes the existing extension-layout contract and does not add
parser- or YAML-specific behavior.

## Interface

```scalascript
extension [A](p: Parser[A])
  def guarded: Parser[Any] =
    Parser.readCtx {
      case Context(level) => Guard(level)
      case _              => Fallback
    }

  def block: Parser[List[A]] = p.many1()
  def withLevel(level: Int): Parser[A] = PWithLevel(p, level)

def ordinary(value: Int): Int = value + 1
```

`guarded`, `block`, and `withLevel` have runtime arities 1, 1, and 2. The
nested match/lambda layout in `guarded` closes only that expression. It cannot
remove `p` from later members. The actual extension dedent closes receiver
ownership before `ordinary`, which remains arity 1.

## Behavior

- [ ] A receiver-only extension member after a nested layout body retains the
      receiver and lowers as `lam 1`.
- [ ] A later member with one explicit parameter lowers as `lam 2`.
- [ ] Nested match, lambda, block-argument, and other virtual layout closes do
      not emit an extension-end marker or clear receiver state.
- [ ] The real indented extension dedent emits exactly one extension-end marker
      and clears receiver state.
- [ ] An explicit-brace extension body has the same ownership boundary as its
      indented form, including nested layout inside a member.
- [ ] A following dedented top-level definition keeps its written arity and is
      not registered as an extension member.
- [ ] Extension start/end identity survives named multi-file import assembly.
- [ ] A focused multi-file fixture has exact identical VM/direct-ASM output for
      a nested-layout member followed by receiver-only and parameterized calls.
- [ ] `examples/dsl-yaml-like.ssc` leaves `arity: 0 expected, 1 given`; all
      parser DSLs either complete or advance identically to separately tracked
      language/runtime gaps.
- [ ] Native-entry, corpus/parity/taxonomy, release, and fresh affected
      conformance gates remain green.

## Out of scope

- Changing extension overload resolution or symbolic primitive precedence.
- Rewriting the parser-combinator or YAML libraries.
- Adding implicit receiver lookup, reflection, compiler fallback, or a host
  implementation of `block`, `line`, or `deeperIndent`.
- Repairing later independent parser evaluator or tuple-pattern gaps.

## Design

The layout stack already distinguishes explicit braces, ordinary layout,
parentheses, square brackets, and the outer extension receiver delimiter. Add
extension-specific body frames for both indented and explicit-brace forms. A
close of one of those frames emits a dedicated internal extension-close token;
ordinary nested layout closes remain ordinary expression delimiters.

`parseStmts` clears `extensionParamsCell` and emits `extension_end` only for the
dedicated token. A residual ordinary close is skipped without changing
extension ownership. The extension header still emits `extension_start`, and
the assembled-statement scanner continues rebuilding imported method identity
from those durable markers.

The dedicated token is parser-internal and never appears in source AST or
CoreIR. This keeps receiver arity a structural frontend property and preserves
the established lowerer/runtime contract.

## Decisions

- **Tag the owning frame, not the next definition** — the layout pass knows
  which delimiter opened the extension body. Rejected: guessing from a later
  `def`, because nested member bodies and dedented top-level defs both produce
  ordinary close tokens after newline normalization.
- **One close token for layout and braces** — both source forms have identical
  receiver lifetime. Rejected: fixing only the YAML-style indented form and
  leaving braced extensions vulnerable to the same nested-layout shape.
- **Preserve flat member statements plus durable markers** — imported extension
  dispatch already depends on this representation. Rejected: introducing a
  new nested extension AST solely for scope repair.
