# ScalaScript 2.1 native tuple-field patterns

Status: TI-8.2d3k2a implementation contract (2026-07-12)

## Overview

Constructor and tuple patterns on the standard self-hosted frontend must retain
the semantics of every field sub-pattern. The current lowering binds ordinary
names and recursively checks nested constructors, but silently discards literal
and typed field constraints. This makes distinct tagged tuples indistinguishable
and can bind a field name at the wrong runtime position.

## Interface

This slice adds no public API. It completes existing pattern syntax:

```scalascript
row match
  case ("customer", id: String, name: String, _) => name
  case ("order", id: String, customerId: String, amount: String) => amount
  case _ => ""

value match
  case Some(("ready", item: Entry)) => item
  case _                            => fallback
```

A literal field uses structural source equality. A typed field uses the same
portable nominal `__isTag__` test as a top-level typed binder; primitive aliases
retain the established `String`/`Int`/`Boolean` behavior. A successful typed
binder denotes the complete field value.

## Behavior

- [ ] A literal field is compared at its exact constructor position; mismatch
      continues with the next source arm.
- [ ] A typed binder field keeps its source name at the exact constructor
      position and tests the field with target-neutral nominal semantics.
- [ ] Wildcards continue occupying one field position without introducing a
      visible binding.
- [ ] Literal, typed, and nested-constructor obligations compose left-to-right;
      an arm body or guard runs only after every obligation succeeds.
- [ ] Failure reuses the once-evaluated original scrutinee and cannot expose
      names from the rejected arm.
- [ ] Pair/Tuple2 source compatibility and strict matching of all unrelated
      CoreIR constructor tags remain unchanged.
- [ ] A focused imported multi-file fixture is exact and byte-identical on
      native VM/direct ASM.
- [ ] `examples/distributed-join.ssc` produces the canonical three rows on both
      native lanes with the fixed CSV fixtures.
- [ ] Native-entry, parity/taxonomy, release, deterministic build-jvm, and fresh
      affected conformance gates remain green.

## Design

`goSubPats` retains a field's balanced type annotation as `tpat` instead of
skipping it. `fldBinder` unwraps typed binders so field scopes preserve their
source names. The lowerer treats nested constructors, literals, and typed fields
as ordered obligations carrying the CoreIR local position of the field.

Literal obligations lower to `__eq__`; typed obligations lower to `__isTag__`.
They add no locals. Nested-constructor obligations retain the existing inner
`IrMatch` and scope-shift behavior. All failure branches delegate to the next
source arm through `lowerOrderedGuardArms`.

## Decisions

- **Own refinement in the self-hosted frontend** — chosen because source
  patterns, not the VM/ASM matcher, know which `Pair` values are source tuples
  and which field annotations/literals apply. Rejected: making CoreIR tags or
  fields globally permissive, which would alter compiler-internal ADTs.
- **Use portable primitives only** — chosen for identical VM, ASM, JS, and
  future backend semantics. Rejected: JVM class tests or host reflection.
- **Preserve ordered fallthrough** — chosen to match source arm order and guard
  behavior. Rejected: emitting duplicate same-tag flat arms, whose winner is
  backend-sensitive and cannot express field refinements.

## Out of scope

- extractor/unapply protocols, structural subtyping, JVM platform classes;
- alternative patterns inside a constructor field;
- changing tuple value representation or compatibility frontend output;
- distributed networking, retries, or remote worker semantics.
