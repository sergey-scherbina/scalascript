# ScalaScript 2.1 core-free native Generator provider

Status: TI-8.2d3h implementation contract (2026-07-11)

## Goal

The standard ScalaScript 2.1 launcher must execute pull-based
`generator { body }`, `suspend(value)`, and local `Generator[T]` combinators on
the shared native VM/direct-ASM runtime. The provider must not load the
compatibility frontend, `PluginBridge`, v1 interpreter, Scalameta, Scala
compiler, or Java compiler.

The release target is the complete `examples/generators.ssc` contract: thirteen
ordered output lines covering finite pull, repeated `next`, `foreach`, fluent
map/filter/take, an infinite Fibonacci source bounded by `take`, drop, nested
flatMap, zip, and zipWithIndex.

## Baseline

Installed `origin/main` through `7404b41bc` has no standard native Generator
owner. VM and direct ASM both exit 1 at the first `generator` expression with
`unbound global: generator` and no stdout. The runtime taxonomy contains one
blocking standard-provider row.

## Ownership and representation

The provider lives in `v2/runtime/std/generator-plugin`, depends only on the v2
native plugin SPI/core graph, and is a required ServiceLoader provider in the
standard launcher and `build-jvm` artifacts. It registers the global closures
`generator` and `suspend` directly. No fallback or stub name is allowed.

A Generator is an opaque `Value.ForeignV(Value.NamedMethodObj)` backed by:

- one JDK 21 virtual producer thread;
- one zero-capacity synchronous event queue;
- a single-consumer lifecycle state;
- terminal events for completion, producer failure, and cancellation.

The zero-capacity queue is binding: `suspend(value)` cannot run ahead of the
consumer and an infinite producer cannot accumulate values in host memory.

## Pull and lifecycle contract

- `generator(body)` starts a virtual producer whose dynamic `suspend` target is
  isolated from every other generator.
- `suspend(value)` performs one blocking handoff and returns Unit only when a
  consumer accepts the value. Outside a live generator body it fails explicitly.
- `next()` returns `Some(value)` for one item and `None` after completion.
  Repeated calls after completion remain `None` without blocking.
- One Generator has one logical consumer. Combinators consume their upstream;
  using the same upstream through competing consumers is outside the contract.
- A producer exception is delivered to the next consumer operation with its
  bounded Generator context; it is never silently converted to completion.
- `take(n)` with `n <= 0` cancels upstream immediately. After its nth item it
  interrupts the abandoned upstream producer, so infinite sources do not leak a
  blocked virtual thread.
- `zip` stops at the shorter input and cancels the unused peer. Cancellation is
  idempotent and never becomes visible as a user error.

## Method-object operations

The provider supplies `next`, `toList`, `foreach`, `map`, `filter`, `take`,
`drop`, `flatMap`, `zip`, and `zipWithIndex`.

- All combinators are lazy pull pipelines and preserve input order.
- `filter` requires Boolean.
- `flatMap` requires another native Generator and drains each inner generator
  completely before advancing the outer generator.
- `zip` emits portable `Tuple2` values and stops at the shorter side.
- `zipWithIndex` starts at zero and emits portable long indices.
- `toList` builds a ScalaScript list iteratively; large finite streams must not
  recurse through the host stack.
- Callback arity/result errors identify the owning Generator method.

The self-hosted lowerer's structural List/Option/Either selector helpers must
fall through to the opaque method object for a chained Generator receiver, just
as required by the Dataset provider. No provider value may be pattern-matched
as a List.

## Dataset boundary

`Dataset.fromGenerator(generator)` may consume the public `next()` method and
materialize once at its own terminal boundary without a binary dependency on
the Generator provider. `Dataset.toGenerator()` remains explicitly unavailable
until a shared factory contract is specified; it must fail rather than return a
stub or compatibility value.

## Isolation

- The provider may reference only the plugin SPI, v2 core, Scala runtime, and
  JDK.
- VM, direct ASM, and `build-jvm` use the same provider class and portable
  values.
- Standard/slim distributions require the provider jar and service entry.
- The Scala 3 seed remains unchanged and no compiler-backed regeneration occurs.
