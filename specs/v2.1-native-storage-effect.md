# ScalaScript 2.1 core-free Storage effect

## Overview

Provide the established `Storage` key/value effect on the compiler-free 2.1 JVM
route through a dedicated native standard provider. The provider reuses the v2
dynamic effect host and preserves the existing ephemeral/file-backed behavior
without loading the v1 interpreter, frontend, bridge, or compiler graph.

## Interface

The provider registers the empty receiver value `Storage` with these methods:

- `Storage.get(key: String): Option[String]`;
- `Storage.put(key: String, value: Any): Unit` (persisted as its ScalaScript
  display string, matching compatibility behavior);
- `Storage.remove(key: String): Unit`;
- `Storage.has(key: String): Boolean`;
- `Storage.keys(): List[String]` in insertion order.

It also registers two runners:

- `runEphemeralStorage(body)` starts with an empty in-memory map and discards it
  when the body returns;
- `runStorage(body, path?)` loads and flushes a deterministic JSON object. The
  path is the explicit second argument, then `SSC_STORAGE_PATH`, then
  `./ssc-storage.json`.

## Behavior

- [x] ephemeral put/get/overwrite/remove/has/keys preserve insertion order and
  return standard portable `Some`/`None`/`List` values;
- [x] every ephemeral invocation starts empty, while nested runners restore the
  outer dynamic handler after the inner body exits;
- [x] file-backed state loads escaped JSON string keys/values, writes after each
  mutation, and is readable by a later runner invocation;
- [x] malformed persisted JSON fails explicitly rather than silently creating a
  partially trusted state;
- [x] `storage-demo.ssc` has stable exact output on native VM/direct ASM without
  writing into the repository during tests;
- [x] ServiceLoader, dependency/class-load, module-limited JRE, native-entry,
  corpus/parity/taxonomy/portable, and fresh conformance gates pass with no
  mismatch or one-sided row.

## Out of scope

- Multi-process locking, transactions, encryption, expiry, or non-String stored
  representations.
- Browser storage and non-JVM provider implementations.
- Changing the source API or the compatibility interpreter implementation.

## Design

Create `v2/runtime/std/storage-effect-plugin` depending only on the v2 native
plugin SPI. It registers an empty `DataV("Storage")`; the kernel's existing
empty-receiver effect dispatch sends method calls to the active `Storage`
handler. Each runner installs a `NativePluginContext.withEffect` scope around
the body closure and owns a private insertion-ordered map.

The small JSON string-object codec is ported from the compatibility
`StorageRuntime` into the provider. This keeps persistence deterministic and
avoids adding a JSON/parser or bridge dependency to a low-level effect module.
The provider is staged in both full and standard distributions and included in
the standard CLI ServiceLoader graph.

## Decisions

- **Dedicated provider rather than kernel special cases** — follows the binding
  rule that standard intrinsics live in `runtime/std/*-plugin` and keeps the
  reusable effect host generic.
- **Port the bounded string-object codec** — chosen because it is the existing
  wire behavior and has no external dependency. Rejected: loading the JSON
  compatibility plugin/bridge for a five-operation effect.
- **Flush each mutation** — preserves the established durability point and
  makes abrupt later failure less likely to lose a successful put/remove.

## Results

Landed in `55aae9abe`; runtime taxonomy tightened in `98b0d0976`. The dedicated
provider depends only on the native SPI, is loaded from the full and standard
ServiceLoader graphs, and is embedded in deterministic `build-jvm` artifacts.
Unit coverage is 3/3. `storage-demo.ssc` has exact portable container output on
native VM, direct ASM, and a standalone executable JAR while tests redirect
`SSC_STORAGE_PATH` into a temporary directory.

Since K62.22 (`d1a7b5451`), nested strings use the user-visible parity renderer
rather than diagnostic `Show.show`: `Some(alice)`, `List(user, role)`, and
`Some(hello world)` are intentionally unquoted. Conformance already owned this
format; release fixtures were reconciled in `befc249d4` without changing stored
values, key order, or JSON persistence.

The full 195-file native corpus is 194 frontend/checker successes plus one
non-code document, with 35 successful executions and 91 honest runtime errors.
Strict standard parity is 20 identical / 48 both-fail / 127 scoped skips, with
zero mismatch or one-sided error. Runtime taxonomy is 15 language-runtime / 18
standard-provider / 6 optional-provider / 3 example-contract / 6 tools-backend:
36 blockers and 48 total reviewed failures. The dependency gate reports 14
roots, 54 static edges, 11 reflective provider JARs, 28 staged dependency JARs,
and zero ownership or forbidden-parser violations. Native entry, plugin/class-
load, no-compiler JRE, slim distribution, build-jvm, portable/taxonomy smokes,
and fresh conformance 11/11 all pass.
