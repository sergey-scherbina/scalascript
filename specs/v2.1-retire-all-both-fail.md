# ScalaScript 2.1 retirement of all both-fail rows

## Overview

The final ScalaScript 2.1 release corpus must contain no VM/direct-ASM row for
which both executions fail. The frozen thirteen-row remainder is not promoted
into the standard runtime: seven examples belong to explicit optional
providers and six belong to compiler-backed or target-specific tools. Each row
is moved to a named, executable lane with an exact regression while plain
`ssc` remains the compiler-free `StandardMain` launcher.

This closes the reporting loophole in which backend agreement could be counted
even though neither backend successfully executed the program. A delegated
row is not a skip: it is present in an exact lane manifest, appears as a named
`provider-lane` or `target-lane` result in the exhaustive report, and is green
only when its real launcher regression passes.

## Interface

The exact lane manifest is stored at:

```text
tests/fixtures/v21-explicit-lanes/manifest.tsv
```

It contains exactly one row for each delegated source file and records its
family, lane kind, launcher regression, and dependency tier. The authoritative
gate is:

```text
tests/e2e/v21-explicit-lanes-gate.sh [--report FILE]
```

The generated report contains one successful row per exact manifest member.
The ordinary `scripts/bc-parity-sweep` report keeps every corpus member visible:

- `identical` for standard VM/direct-ASM successes;
- existing `skipped-*` categories for server, nondeterministic, and already
  declared backend/front-end documents;
- `provider-lane` or `target-lane` only for exact manifest members whose
  separate launcher regression is owned by the explicit-lane gate;
- `both-fail`, mismatch, or one-sided errors remain failures.

The negative-toolchain report adds exact delegated-lane counts but does not
copy or execute tools/provider dependencies inside the standard-only sandbox.
Its standard parity contract is `parity.both-fail=0`, mismatch/one-sided zero,
with the delegated membership validated against the same exact manifest and
the full-distribution explicit-lane gate run separately by CI and the
consolidated release gate.

## Behavior

- [x] All three PDF examples run through an explicit core-free native PDF
      provider, on native VM and direct ASM, with PDF dependencies absent from
      the plain standard classpath.
- [x] Both MCP examples run through the intended explicit MCP/tool launcher
      against the bundled deterministic MCP server and have exact output
      assertions.
- [x] The remote Graph example runs through an explicit graph-provider lane
      against a deterministic local RDF4J-compatible endpoint; plain Graph
      remains the local standard provider.
- [x] The SWIFT example runs through an explicit provider lane with a
      deterministic local SWIFT-compatible test endpoint and no secret or
      public-network dependency.
- [x] The NFC example runs through an explicit core-free native NFC provider
      on VM and direct ASM; the host-unavailable result is a successful,
      deterministic capability outcome rather than an unhandled intrinsic.
- [ ] Both quoted-macro examples run through the explicit tools launcher and
      preserve their documented exact outputs without a StandardMain fallback.
- [ ] Both WASM documents have declared target regressions through the WASM
      launcher; the deterministic pure example is executed and the HTTP
      example is compiled/validated without depending on the public network.
- [ ] The x402 client has an explicit tools/backend regression against a local
      deterministic x402 endpoint and never enters the standard native lane.
- [ ] The exact lane gate rejects missing, duplicate, unknown, stale,
      reclassified, or newly added delegated members.
- [ ] Exhaustive ordinary and negative-toolchain reports contain zero
      `both-fail`, mismatch, one-sided, and blocking rows; the release freeze
      rejects a reintroduced member or count drift.
- [ ] Plain `ssc`, `ssc-standard`, and `build-jvm` retain the compiler-free,
      Scalameta-free dependency graph; optional provider and tools artifacts
      are reachable only through explicit lane selection.
- [ ] The consolidated release gate, CI, and fresh affected conformance all
      pass with the new exact reports.

## Design

### Exact lane ownership

The thirteen current members are frozen by family:

| Family | Files | Required lane |
|---|---:|---|
| PDF | 3 | core-free opt-in native provider, VM + ASM |
| MCP | 2 | explicit MCP provider/tools launcher against local server |
| Graph | 1 | explicit remote graph provider against local endpoint |
| SWIFT | 1 | explicit payment provider against local endpoint |
| NFC | 1 | core-free opt-in native provider, VM + ASM |
| quoted macros | 2 | compiler-backed `ssc-tools` regression |
| WASM | 2 | explicit WASM target compile/run regression |
| x402 | 1 | compiler-backed tools/client regression against local endpoint |

The manifest is the only exception list. `bc-parity-sweep` must not infer
delegation from source spelling, failure text, or file-name patterns.

### Provider isolation

Native optional providers implement `ssc.plugin.NativePlugin` and depend on
the v2 plugin SPI plus their own feature libraries only. They do not depend on
the v1 interpreter, frontend, Scalameta, compiler, or compatibility bridge.
Provider JARs and their third-party dependencies are staged outside
`bin/lib/standard/jars` and are selected by an explicit provider launcher or
option. `NativePluginHost` receives only explicitly selected provider paths;
ordinary ServiceLoader discovery remains restricted to standard JARs.

Direct-ASM execution uses the same selected provider set as VM execution.
When a provider-backed `build-jvm` regression is required, the artifact embeds
only that explicitly selected provider and its runtime dependencies; it never
widens the default artifact graph.

### Tools and target lanes

Compiler-backed syntax is never partially reimplemented in StandardMain.
Quoted macros and x402 use `ssc-tools`; WASM uses its explicit target launcher.
Their regressions are local and deterministic. A compile-only target test is
allowed only where the observable operation inherently requires browser or
device facilities, and it must inspect the produced target artifact rather
than accepting command exit alone.

### Release accounting

The former runtime-taxonomy manifest and 13-row exact freeze become empty and
pin zero across every category. Delegated rows are accounted for by the new
exact lane manifest/report, not by converting them to `skipped-*`. The
consolidated release summary publishes both standard parity and explicit lane
metrics so the total corpus remains auditable.

## Decisions

- **Explicit delegated categories, not skips** — preserves every source row in
  the exhaustive report while distinguishing successful out-of-tier ownership
  from unexecuted sources. Rejected: extending the existing skip regexes.
- **One exact manifest shared by accounting and execution** — prevents the
  standard report and lane regressions from drifting independently. Rejected:
  per-test hard-coded file lists.
- **Core-free native providers for optional JVM capabilities** — keeps feature
  dependencies out of the language core and standard classpath. Rejected:
  loading v1 `.sscpkg` providers through the compatibility bridge.
- **Local deterministic services** — exercises real protocol paths without
  credentials, hardware, or public network. Rejected: success-by-stub and
  environment-dependent CI.
- **Keep target-specific syntax in tools** — protects the self-hosted standard
  frontend from compiler APIs and platform dialects. Rejected: transparent
  fallback or accepting platform types in ordinary `.ssc` execution.

## Out of scope

- Removing the permanent Scala 3 bootstrap seed or Scala runtime.
- Making PDF, MCP, remote Graph, SWIFT, NFC, quoted macros, WASM, or x402 part
  of the default standard dependency graph.
- Requiring public services, production credentials, Apple hardware, or NFC
  hardware in release CI.
- Removing `ssc-tools` or compiler-backed target emitters from the optional
  full distribution.
