# ScalaScript 2.1 standard launcher cutover

Status: **planned** (2026-07-12)

## Overview

TI-8.3 makes the already-frozen compiler-free ScalaScript 2.1 path the default
installed experience. Plain `ssc` and the descriptive `ssc-standard` alias run
the native frontend/checker plus the v2 VM or direct ASM from the physical
standard tier. Scalameta, the v1 frontend, Scala/Java compilers, and legacy
backend commands remain available only after the user explicitly invokes
`ssc-tools`; the standard entry never discovers or delegates to that launcher.

The permanent prebuilt Scala 3 seed is retained. This is a launcher and release
ownership cutover, not a source-bootstrap rewrite.

## Interface

The staged and self-installed launchers have this contract:

```text
bin/ssc          -> lib/standard/{ssc.jar,jars/*} -> scalascript.cli.StandardMain
bin/ssc-standard -> lib/standard/{ssc.jar,jars/*} -> scalascript.cli.StandardMain
bin/ssc-tools    -> lib/{ssc.jar,jars/*}          -> scalascript.cli.ssc
```

The standard entry accepts only the standard surface:

```text
ssc run [--native|--v2] [--bytecode] file.ssc [more.ssc ...] -- [args ...]
ssc file.ssc [args ...]
ssc build-jvm file.ssc [more.ssc ...] -o app.jar
ssc info --execution-plan [--bytecode]
ssc help
```

Requests for `--v1`, `--compat-frontend`, `check`, `compile-jvm`, emit/source
backends, install/package commands, or another tools-owned command fail before
source execution with a bounded diagnostic naming `ssc-tools`. Compatibility
remains explicit through commands such as:

```text
ssc-tools run --v1 file.ssc
ssc-tools run --compat-frontend file.ssc
ssc-tools compile-jvm file.ssc
ssc-tools install --prefix <dir>
```

The `ssc-tools install` result repeats the same three-launcher split: its plain
`ssc` is standard and its `ssc-tools` remains the explicit full-tier entry.

## Behavior

- [ ] Checked-in `bin/ssc`, `installBin`, contributor `install.sh`, and
      `ssc-tools install` all produce the launcher/classpath contract above.
- [ ] Plain `ssc run` and file shorthand use `StandardMain` and execute the
      native VM without compiler, Scalameta, v1, or tools classes.
- [ ] Plain `ssc run --bytecode` and `ssc build-jvm` use direct ASM and remain
      green after the full/tools JAR trees are physically absent.
- [ ] Plain `ssc` remains green with `java.compiler` and `jdk.compiler`
      unavailable in a module-limited runtime.
- [ ] `ssc --v1`, `ssc --compat-frontend`, and tools-owned commands fail with a
      bounded `ssc-tools` remedy and never spawn/delegate to the tools launcher.
- [ ] Explicit `ssc-tools run --v1` and `ssc-tools run --compat-frontend`
      preserve the compatibility frontend in the full distribution.
- [ ] The exhaustive 195-row frontend/parity/taxonomy release stays at zero
      mismatch, one-sided failure, language/standard gap, and blocking row;
      the exact 7 optional / 6 tools / 0 blocker freeze remains unchanged.
- [ ] Fresh `v2-*` conformance stays green.

## Design

`StandardMain` remains the single standard command dispatcher. TI-8.3 removes
its `ProcessBuilder` compatibility delegation entirely: recognising an explicit
compatibility flag is useful only to emit the bounded remedy. This prevents a
future classpath/layout change from turning an explicit-looking standard command
into a transparent transition to the full graph.

The two standard shell launchers intentionally share the same classpath and
main class. The short `ssc` name remains the user-facing executable;
`ssc-standard` is retained as a diagnostic/packaging alias and for one release
of script compatibility. AppCDS, when enabled, caches the standard classpath.

Release tests exercise the short name from a copied slim distribution and from
a runtime whose derived module set excludes both compiler modules. Static
checks also assert the generated launcher bodies, because a green
`ssc-standard` alias alone would not prove the default cutover.

## Decisions

- **No delegation from `ssc` to `ssc-tools`** — explicit selection is visible
  in the executable name and cannot be triggered by fallback. Rejected: spawning
  `ssc-tools` for `--v1`/`--compat-frontend`, because that keeps the standard
  launcher coupled to the compatibility layout.
- **Keep `ssc-standard` as an alias** — it is useful for layout diagnostics and
  existing gates while `ssc` becomes canonical. Rejected: deleting it during
  the same cutover, which would add unrelated script churn.
- **Preserve the full tools installation** — 2.1 separates default ownership;
  it does not remove legacy backends. Rejected: deleting compiler/scalameta JARs
  from the full distribution.
- **Preserve the Scala 3 seed** — the frozen seed is an allowed runtime
  implementation boundary. Rejected: coupling the release cutover to a new
  bootstrap implementation.

## Out of scope

- Removing the v1 frontend, Scalameta, scala-cli, or compilers from source and
  the optional full/tools distribution.
- Changing `.ssc` language semantics, provider ownership, or the frozen runtime
  taxonomy.
- Replacing the permanent Scala 3 seed or the Scala runtime.
- Making tools commands available on the standard classpath.
