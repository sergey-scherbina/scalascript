# ScalaScript 2.1 core-free native Coroutine provider

Status: implemented and verified (2026-07-21)

## Overview

The standard ScalaScript 2.1 launcher must execute the low-level asymmetric
`Coroutine[Y, R, T]` API on the shared native VM/direct-ASM runtime without the
compatibility frontend, v1 interpreter, Scalameta, Scala compiler, or Java compiler.
The native implementation preserves the already shipped v1/JS observable contract:
lazy start, zero-capacity two-way suspend/resume handoff, terminal error values,
explicit cancellation, and dynamic composition with the native Generator provider.

## Baseline

Installed `origin/main` through `d6e551b33` passes the historical INT/JS
`coroutine-*` conformance cases, but neither case opts into the native lane. The
real standard paths both fail before producing stdout:

```text
$ bin/ssc run tests/conformance/coroutine-basic.ssc
ssc: unbound global: coroutineCreate

$ bin/ssc run --bytecode tests/conformance/coroutine-basic.ssc
ssc: unbound global: coroutineCreate
```

The standard native Generator provider already registers global `suspend`. The
native plugin host rejects duplicate global ownership, so adding an independent
Coroutine registration for the same name is not a valid implementation.

## Interface

The logical public contract is:

```scalascript
enum Step[+Y, +T]:
  case Yielded(value: Y)
  case Returned(value: T)
  case Errored(message: String)
  case Cancelled

extern def coroutineCreate[Y, R, T](body: () => T): Coroutine[Y, R, T]
extern def coroutineResume[Y, R, T](co: Coroutine[Y, R, T], in: R): Step[Y, T]
extern def suspend[Y, R](out: Y): R
extern def coroutineCancel[Y, R, T](co: Coroutine[Y, R, T]): Unit
```

`v1/runtime/std/coroutine.ssc` currently erases the opaque `Coroutine` result and
parameter to `Any`; this is a type-surface limitation, not permission to accept an
arbitrary runtime value. The native functions accept only their own opaque handle.
The self-contained public walkthrough is
[`examples/coroutine-demo.ssc`](../examples/coroutine-demo.ssc); it runs unchanged
on the native VM, direct ASM, and `build-jvm` paths.

## Behavior

- [x] `coroutineCreate` allocates an opaque process-local handle without executing
      the body or starting a producer thread. The body starts on the first
      `coroutineResume`; that first input releases the start gate and is otherwise
      ignored because no `suspend` is waiting for it yet.
- [x] Each `suspend(out)` returns `Step.Yielded(out)` to exactly one resumer and
      blocks the body until the next resume input arrives; that input becomes the
      return value of the suspended call. No output or input is buffered ahead.
- [x] Normal body completion produces one `Step.Returned(value)`. An unhandled body
      exception produces one `Step.Errored(message)`. Both terminal steps invalidate
      the handle, and every later resume fails with a bounded completed/cancelled
      diagnostic instead of blocking.
- [x] Nested coroutines bind `suspend` to the innermost active body. Native Generator
      and Coroutine bodies coexist in one process, and each producer thread sees only
      its own dynamic suspend target.
- [x] `coroutineCancel` is idempotent before start, while suspended, and after normal
      completion. It interrupts an active body, invalidates the handle, and makes a
      later resume fail. `Step.Cancelled` remains in the portable ADT for higher-level
      controlled-shutdown protocols; cancellation does not keep a dead handle alive
      solely so it can be resumed to observe that case.
- [x] Calling `suspend` without a live Coroutine or Generator body fails immediately
      with a diagnostic naming both valid contexts.
- [x] The same provider values and behavior execute on native VM, direct ASM, and a
      deterministic `build-jvm` artifact. No compatibility fallback, generated host
      source, provider stub, or platform value crosses the language boundary.

## Design

### One owner for suspension

Ownership remains in the existing required
`v2/runtime/std/generator-plugin` project and its `59-generator` provider. Its
private Generator-only emitter thread-local becomes one private dynamic
`SuspendTarget` slot:

- a Generator target performs the existing synchronous item handoff and returns
  `Unit`;
- a Coroutine target publishes `Yielded(out)` and waits for the next resume value;
- the single registered `suspend` global delegates to the current target.

The provider also registers `coroutineCreate`, `coroutineResume`, and
`coroutineCancel`. Keeping one provider id means the ServiceLoader count, plugin
ownership graph, standard JAR allowlist, and build-jvm dependency roots do not
change. The implementation is still a standard-library plugin, never a core
intrinsic.

### Coroutine state

Each handle is a `Value.ForeignV` around private state containing:

- the body closure and native invocation context;
- one zero-capacity body-to-resumer handoff;
- one zero-capacity resumer-to-body handoff;
- lifecycle flags and the lazily-created JDK 21 virtual thread.

Resume is single-consumer and serialized per handle. Queue handoff supplies the
required happens-before edge for body side effects. A terminal step is retained only
long enough for the active resume to receive it; the public handle then becomes
invalid. `ForeignV` makes the mutable handle non-portable and non-serializable by
construction.

## Decisions

- **Extend the existing standard suspension owner** — this keeps one global owner and
  introduces no cross-plugin suspend ABI or extra packaging root. Rejected: a separate
  `coroutine-plugin` plus Generator-to-Coroutine project dependency (more module,
  ServiceLoader, allowlist, and artifact wiring for identical runtime behavior).
- **Use virtual-thread synchronous handoff** — it matches the shipped JVM contract and
  the already proven native Generator mechanism. Rejected: a CoreIR CPS transform
  (substantially larger compiler work and a second continuation representation).
- **Use an opaque state handle directly** — identity and lifecycle stay target-private
  without a global numeric-handle registry. Rejected: portable `DataV` ids (forgeable,
  serializable-looking, and require mutable global lookup state).
- **Return `Errored(message)` for body failure** — this preserves the public std module
  and conformance behavior. Rejected: throwing from `coroutineResume` for the first
  body failure (observable incompatibility); invalid/terminal handles still throw.

## Out of scope

- Rebuilding Generator, Async, actors, WebSockets, or algebraic effects on Coroutine.
- Changing the v1 interpreter or JavaScript coroutine implementations.
- Concurrent resumers, cross-thread transfer, a public scheduler, timeouts, or
  symmetric coroutine jumps.
- Durable/network serialization, save/resume, or replay of mutable coroutine handles.
- Exposing `Step.Cancelled` by making a cancelled handle resumable again.
