# ScalaScript 2.1 — self-hosted core and parser independence

Status: **in progress** (2026-07-10)

## Overview

ScalaScript 2.1 keeps a permanent Scala 3 seed while moving every normative
parser above that seed into ScalaScript itself. The standard language core must
not require third-party JSON, YAML, Markdown, regular-expression, serialization,
or parser libraries. Pure portable behavior belongs in `.ssc`/`.ssc0`; external
libraries are permitted only behind an explicit plugin/backend boundary or in
build/test tooling.

This contract strengthens `specs/v2.1-toolchain-independence.md`. Its current
references to `SimpleYaml`, ujson, and host-side manifest extraction describe the
migration baseline, not the final 2.1 dependency boundary.

## Definitions and boundary

The 2.1 distribution uses five dependency layers:

| Layer | Implementation | Allowed dependencies | Parser ownership |
|---|---|---|---|
| **Scala 3 seed** | `v2/src` plus the minimal host/SPI needed to start it | JDK and Scala runtime | May decode only bootstrap `ssc0`/CoreIR/image formats; it must not parse JSON, YAML, Markdown, or ordinary `.ssc` syntax |
| **Self-hosted language core** | staged `.ssc0` compiler image and `.ssc` pure modules | seed primitives only | Owns Markdown profile, ScalaScript lexer/parser/checker/lowerer, front-matter profile, and portable text codecs |
| **Pure standard library** | `runtime/std/**/*.ssc` | other pure `.ssc` modules | Owns canonical JSON/YAML/regex/parser-combinator semantics; no `extern def` is required for a pure parser |
| **Plugins/backends** | `v2/runtime/std/*-plugin`, backend plugins | explicit feature-specific external libraries | May accelerate or integrate, but may not silently replace canonical semantics or become an undeclared core dependency |
| **Tools/tests** | sbt, compatibility frontend, reference/fuzz harnesses | unrestricted within normal policy | Reference parsers may be used only for differential tests, never by a standard command |

The Scala 3 seed is permanent and intentionally trusted. Self-hosting does not
mean removing Scala 3 from the repository or building the seed with ScalaScript.
It means that the seed stays small, stable, and ignorant of user-facing data and
document grammars.

## Interface

The public language API remains source-compatible:

```scalascript
jsonValue(text)       // tolerant, total navigation
jsonParse(text)       // strict
jsonRead(text)        // strict navigable value
jsonStringify(value)  // canonical compact encoding

parseYaml(text)
toYaml(value)
```

The native frontend returns one structural compilation result rather than
requiring the host to reparse text produced by the frontend:

```text
NativeCompilation(
  program: CoreProgram,
  manifest: ManifestValue,
  runtimeConfig: RuntimeConfig,
  markdown: List[MarkdownDocument],
  sources: List[SourceIdentity]
)
```

The concrete bridge may use `ssc.Value` ADTs or a frozen seed image ABI. The
Markdown field carries the canonical structural parse product (or a
source-located `MarkdownError`) for every root source. The permanent Scala seed
may validate/map this frozen ADT, but it must not reparse Markdown text. The
bridge must not serialize CoreIR, runtime configuration, or Markdown to a text
format and parse it again in Scala host code.

## Format profiles

### JSON

The canonical parser follows RFC 8259 behavior required by the existing API:

- strict and tolerant entry points share one scanner and AST;
- objects, arrays, strings, booleans, null, numbers, escapes, Unicode escapes,
  and surrogate pairs are supported;
- strict parsing rejects trailing input, malformed escapes, unescaped control
  characters, invalid numbers, and incomplete containers;
- serialization is deterministic and compact;
- exact decimal text is preserved for `asDecimal` rather than round-tripped
  through `Double`;
- the implementation is hand-written ScalaScript over character/string
  primitives and does not depend on host regex semantics.

### YAML/front-matter

ScalaScript defines a bounded **Frontmatter YAML Profile**, not all YAML 1.2:

- block/flow mappings and sequences;
- quoted/plain strings, null, booleans, integers, decimal values;
- comments and literal/folded blocks needed by existing manifests;
- deterministic ordered mapping representation;
- duplicate manifest keys are rejected with source position;
- anchors, aliases, tags, merge keys, directives, and multi-document streams are
  rejected with a bounded diagnostic.

`std.yaml` uses the same self-hosted parser/profile by default. A full YAML
implementation may be an explicit plugin, but standard front-matter never needs
that plugin.

### Markdown

The self-hosted Markdown frontend owns the ScalaScript Markdown Profile required
by `SPEC.md`: headings/scopes, pure-link import paragraphs, prose, fenced blocks,
lists, images, tables, metadata directives, and inline interpolation source.
Every standard native compilation evaluates that canonical parser and crosses
the seed boundary as a structural `MarkdownDocument`; a `MarkdownError` aborts
before plugin installation. Unsupported CommonMark extensions remain inert
content rather than triggering a host Markdown parser. Full CommonMark/GFM
reference libraries are test or tools dependencies only.

### Regular expressions

Core JSON/YAML/Markdown/language parsing must not depend on `java.util.regex` or
another target regex engine. Parsers use explicit scanners. `std.parsing` may
later expose a self-hosted portable regex engine with specified semantics;
platform regex remains an optional plugin/FFI capability until then.

## Bootstrap and compiler image

The installed path is:

```text
Scala 3 seed
  -> load frozen compiler image
  -> self-hosted Markdown/front-matter/language frontend
  -> checked CoreIR + structural manifest/config
  -> VM or backend plugin
```

Release `N` may build the compiler image for release `N+1`. A stage-2 rebuild
must reproduce the canonical compiler image or its canonical semantic digest.
The production launcher may keep the seed `ssc0` and CoreIR decoders; those are
bootstrap decoders, not user-facing format parsers. No bootstrap fallback may
invoke ujson, SimpleYaml, CommonMark, Scalameta, or the compatibility frontend.

## Dependency policy

1. Standard seed/core JARs have a zero third-party parser/codec dependency
   budget. In particular they must not contain or reference `ujson`, `upickle`,
   SnakeYAML, Jackson, Circe, CommonMark/Flexmark, Borer, Scalameta, or parser
   combinator libraries implemented outside ScalaScript.
2. The JDK and Scala runtime are allowed seed implementation dependencies in
   2.1. Removing `scala-library` is not part of this feature.
3. ASM is a JVM backend dependency, not a language-core dependency. It must be
   reachable only when the direct JVM backend is selected or packaged.
4. JDBC drivers, crypto providers, network clients/servers, compression engines,
   and optional accelerated codecs belong to named plugins.
5. A pure parser implemented through an `extern def` or hidden host global does
   not satisfy self-hosting.
6. Standard commands never discover and silently activate an accelerated parser
   plugin. Selection, when offered, is explicit and observable.
7. External reference implementations and corpora are allowed in `Test` scope.

## Design

### Shared parser products

Self-hosted parsers return ordinary target-independent ADTs. Host-facing plugins
consume those values through `ssc.Value`/the native SPI rather than importing a
library-specific JSON/YAML value hierarchy. HTTP transports own bytes and text;
they do not own JSON parsing. SQL configuration consumes the already-parsed
front-matter product.

### Migration order

1. Add a dependency classifier/gate and record the current standard graph.
2. Land the pure JSON scanner/AST/encoder in ScalaScript with conformance tests.
3. Switch the standard JSON surface and HTTP reuse to the self-hosted codec;
   remove ujson from the default standard runtime graph.
4. Land the Frontmatter YAML Profile parser in ScalaScript.
5. Make the frontend return manifest/runtime config structurally and retire
   `NativeFrontmatter`/`SimpleYaml` from the standard path.
6. Complete the self-hosted Markdown Profile and remove CommonMark from the
   standard path.
7. Isolate ASM and every remaining removable dependency behind backend/plugin
   boundaries; reconcile slim-distribution packaging.
8. Add stage-2 bootstrap, corpus, classpath, and runtime class-load release gates.

Each step must stay green independently. Compatibility/tools code may keep its
existing libraries until its own retirement; it cannot leak into standard tests.

## Behavior

- [ ] The permanent Scala 3 seed boundary is documented and mechanically
  allowlisted; it contains no JSON/YAML/Markdown parser.
- [x] A dependency gate classifies every standard runtime JAR as seed, pure core,
  backend plugin, feature plugin, or tools/test and rejects unclassified JARs.
- [x] JSON strict/tolerant parsing and deterministic stringify run from
  ScalaScript code on VM and direct ASM without loading ujson/upickle.
- [x] JSON covers malformed input, escapes, Unicode/surrogate pairs, numbers,
  exact decimals, nesting, total accessors, and trailing-input rejection.
- [x] Frontmatter YAML Profile parsing runs from ScalaScript and produces the
  same manifest/runtime config for VM and direct ASM.
- [x] Invalid/duplicate YAML keys and unsupported anchors/tags/multi-doc features
  fail with source-located diagnostics before plugin installation.
- [x] The native compiler returns CoreIR plus manifest/config structurally; the
  Scala host does not parse YAML or frontend-emitted textual CoreIR.
- [ ] The self-hosted Markdown Profile covers every first-class construct named
  in `SPEC.md` without CommonMark/Flexmark in the standard classpath.
- [ ] Pure core parsers do not call host regex engines.
- [ ] ASM and other unavoidable external libraries are reachable only from the
  backend/plugin that declares them.
- [ ] Standard slim execution, `build-jvm`, and representative JSON/YAML/HTTP/SQL
  programs pass with external parser libraries physically removed.
- [ ] A stage-2 self-host rebuild reproduces the canonical compiler image/digest.

## Verification

Every parser slice requires:

- focused `.ssc` conformance cases on native VM and direct ASM;
- malformed-input negative cases with stable source positions;
- differential tests against a reference implementation in `Test` scope where
  useful, with no reference JAR in the produced standard artifact;
- fuzz/property cases bounded for CI;
- `tests/conformance/run.sh --only '<affected globs>'` before push;
- `jdeps`, JAR-entry, and runtime class-load gates for forbidden families;
- `git diff --check`.

The final release gate additionally deletes every external parser JAR and the
tools tier, then runs native hello/import/JSON/YAML/HTTP/SQL and two deterministic
`build-jvm` builds.

## Decisions

- **Keep the Scala 3 seed permanently** — this is the trusted implementation
  base chosen by the project. Rejected: rewriting the seed in ScalaScript,
  because that confuses self-hosted language behavior with bootstrapping.
- **Pure parsers are standard-library/core code, not plugins** — they need no
  host capability and must behave identically on every target. Rejected: making
  ujson/SnakeYAML mandatory native plugins, because that retains target-specific
  semantics and dependencies.
- **Use explicit scanners in core parsers** — deterministic cross-target behavior
  and bounded errors matter more than parser-combinator concision. Rejected:
  platform regex primitives in canonical parsers.
- **Specify a YAML profile** — it covers ScalaScript manifests without YAML's
  unsafe and complex global features. Rejected: full YAML 1.2 in the 2.1 core.
- **Structural frontend result** — avoids parsing compiler output twice and
  makes manifest ownership unambiguous. Rejected: stdout JSON/YAML/CoreIR as the
  in-process compiler-host ABI.
- **Reference libraries stay in tests** — differential validation is useful but
  must not become production behavior.

## Out of scope

- rewriting or removing the Scala 3 seed;
- removing the Scala runtime or JDK from the JVM distribution;
- implementing every YAML 1.2 feature in the standard profile;
- parsing opaque `scala`, JavaScript, SQL, HTML, or CSS fenced languages in the
  language frontend;
- removing external dependencies from explicitly selected plugins or tools;
- immediately deleting the v1 compatibility tier.
