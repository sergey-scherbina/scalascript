# ScalaScript 2.1 native curried closures

## Overview

The self-hosted frontend preserves Scala-style parameter-list calls as nested
CoreIR applications while a definition's parameter clauses are represented by
one flattened `Lam(totalArity, body)`. Native VM and direct ASM therefore need
one shared closure under-application contract. Rejecting the first application
breaks ordinary curried helpers even though their CoreIR is structurally valid.

## Interface

For a closure `f` with remaining arity `n` and an application containing `k`
arguments:

- `k == n` invokes `f` once;
- `0 <= k < n` returns a closure of arity `n - k` that captures the supplied
  arguments in order;
- `k > n` is an arity error.

Applying the returned closure combines its captured arguments with the new
arguments and evaluates the original closure body in the original captured
environment. This applies to user functions and first-class closures; it does
not add a named-function or example-specific dispatch path.

## Behavior

- [ ] A two-argument closure applied as `f(a)(b)` produces the same result as
      `f(a, b)` on the native VM and direct ASM.
- [ ] A partially applied closure remains first-class and can be invoked later
      without re-evaluating already captured argument values.
- [ ] Captured arguments retain left-to-right order and compose with the
      original lexical environment.
- [ ] Supplying more arguments than the remaining arity retains the explicit
      arity failure instead of dropping or reinterpreting arguments.
- [ ] Installed `examples/algebraic-effects.ssc` advances through the curried
      `withLogger` helper identically on VM/direct ASM.

## Design

`Runtime` owns a small helper that creates a residual `ClosV` around the
original closure and already supplied arguments. Both the VM trampoline's
`Call` branch and the direct-bytecode `Emit.app` shim call the same helper when
an application is short. The residual closure returns the original closure's
`Step`, preserving tail calls and avoiding a second application convention.

## Out of scope

- changing parser or typer representation of parameter-clause boundaries;
- auto-applying a closure with too many arguments;
- compatibility-frontend fallback or v1 dispatch.

## Decisions

- **Shared runtime ABI** — chosen because VM and generated ASM already converge
  on `ClosV`; rejected compile-time name-based flattening because it would miss
  first-class/local closures and duplicate call semantics.
