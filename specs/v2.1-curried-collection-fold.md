# Curried collection folds on the native JVM path

## Overview

ScalaScript collection methods support both ordinary parenthesized currying
and Scala-style trailing blocks. The self-hosted frontend represents
`xs.foldLeft(z) { f }` as an application of the partial method result to `f`.
The portable runtime must therefore return a real closure after the first
argument list instead of fabricating an unresolved effect operation. Native VM
and direct ASM share this contract.

## Interface

```scalascript
xs.foldLeft(z) { (acc, value) => next }
xs.foldLeft(z)((acc, value) => next)
```

The first application, conceptually `xs.foldLeft(z)`, produces an arity-one
function. Applying that function to the binary fold closure is observationally
equivalent to the already supported uncurried runtime dispatch with arguments
`[z, f]`.

## Behavior

- [x] A list `foldLeft(List[Int]()) { ... }` running-total fixture prints

- [x] Parenthesized and trailing-block second argument lists use the same
      partial-dispatch closure semantics.
- [x] Mutable array receivers produced by the portable `Array` companion can
      use the same curried `foldLeft` contract.
- [x] Completing the partial call delegates to the existing effect-aware full
      fold implementation; no eager duplicate fold loop is introduced.
- [x] `examples/functional.ssc` prints its canonical running-total line and is
      byte-identical on native VM/direct ASM.
- [x] Full strict parity has zero mismatch and one-sided rows; runtime/sentinel
      taxonomy and fresh affected conformance remain valid.

## Out of scope

- Changing `foldLeft` iteration order, accumulator semantics, or effect
  threading.
- General automatic currying for every unknown method name.
- Rewriting curried source calls into one flattened call in the frontend.
- Repairing the separately tracked JSON/YAML placeholder values unless they
  are proven to share this exact partial-dispatch root cause.

## Design

Portable method dispatch recognizes the supported collection receiver plus
`foldLeft` with exactly `[z]`. It returns `ClosV(..., arity = 1, ...)`. The
closure takes the supplied fold function and calls the same `methodOp` path
with `[z, f]`; that path already selects list, mutable-array, or map traversal
and preserves `foldThreadOp` effect handling.

The bytecode emitter needs no special branch: it already applies `ClosV`
through `Emit.app`, which shares the VM closure ABI. The focused source fixture
must exercise both backends through the assembled distribution.

## Decisions

- **Runtime partial-method closure** — chosen because it matches the CoreIR
  already emitted for trailing blocks and reuses the full portable dispatcher.
  Rejected: ASM-only normalization (would leave VM semantics wrong).
- **Only known fold receivers** — chosen to keep miss diagnostics honest.
  Rejected: treating every one-argument method miss as a curried call (would
  turn typos into delayed failures).
- **Delegate, do not duplicate** — chosen so effect-aware traversal has one
  implementation. Rejected: a second eager loop inside the partial case.
