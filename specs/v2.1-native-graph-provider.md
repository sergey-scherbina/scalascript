# ScalaScript 2.1 core-free Graph provider

Status: TI-8.2d3l implementation contract (2026-07-12)

## Goal

The standard ScalaScript 2.1 launcher must execute the portable process-local
property/RDF graph facade without the v1 interpreter graph plugin, graph backend
implementations, Scalameta, or a compiler. Backend-specific remote query
languages remain explicit capabilities and must fail with bounded diagnostics
instead of being emulated or routed through compatibility.

## Baseline and ownership audit

Installed standard VM/direct ASM both fail before stdout:

- `graph-storage-interpreter.ssc`: `unhandled runtime effect:
  Graph.putVertex`;
- `graph-rdf4j-http-storage.ssc`: `unhandled runtime effect: Graph.putRdf`.

The first document declares `backend: in-memory` and explicit compatibility
prints `imports:b.ssc`. It is a required standard contract. The second declares
`backend: rdf4j-http` plus external URL/user/password configuration; even the
compatibility interpreter prints `Stored two books.` and then deliberately
rejects `Sparql.select` as unavailable. Its remote query is an optional/external
provider boundary, not a local standard success target.

## Interface

A required provider at `v2/runtime/std/graph-plugin` owns these exact structural
intrinsics:

```text
Graph.putVertex(graphName, value)
Graph.getVertex(graphName, id)
Graph.vertices(graphName)
Graph.putEdge(graphName, value)
Graph.edges(graphName)
Graph.neighborValues(graphName, from, edgeLabel?)
Graph.neighbors(graphName, from, edgeLabel?)
Graph.putRdf(graphName, value)
Graph.getRdf(graphName, subject)
Graph.triples(graphName, subject?, predicate?)
```

It also owns exact unavailable diagnostics for `Sparql.select`,
`Sparql.update`, `Cypher.query`, and `Gremlin.query` on this process-local
provider. Unknown `Graph.*`/query operations remain unhandled.

Portable values keep their source `DataV` identity. `StoredEdge` exposes
`id/from/to/label/value`; `RdfTriple` exposes `subject/predicate/obj`. Options
and lists use the standard portable `Some`/`None` and `Cons`/`Nil` values.

## Property graph behavior

- Stores are process-local, keyed by exact graph name, and start empty on every
  provider installation.
- Vertex IDs come from an exact `id` field. Re-putting the same ID replaces the
  value without changing insertion position.
- Vertex and edge iteration is insertion ordered.
- An edge requires exact `from` and `to` string fields and both endpoint
  vertices. Its default label is the decapitalized source constructor name with
  a trailing `Edge` removed.
- An explicit `id` field wins. Otherwise IDs are deterministic
  `<from>-<label>-<to>`, suffixed `-2`, `-3`, ... on collision.
- Neighbor traversal follows edge insertion order and may filter by exact
  optional label; it returns the stored destination values.

## RDF behavior

- RDF records are keyed by an exact `id` field, falling back to `subject`.
  Re-put replaces in place.
- `getRdf` accepts a string, portable `Iri(value)`, or `Some`-wrapped subject.
- `triples` walks records and source fields in stable insertion/declaration
  order, excludes `id`/`subject`, and applies optional exact subject/predicate
  filters.
- The local provider does not parse SPARQL/Cypher/Gremlin or contact a remote
  endpoint. Exact query operations fail before returning a value and name the
  required explicit backend path.

## Errors and isolation

Every operation validates arity and value shape. Missing IDs/endpoints,
non-string graph/edge identifiers, invalid option values, absent vertices, and
remote query attempts fail explicitly; none becomes Unit, `Stub`, `Op`, an
empty fabricated result, or a compatibility value. Reinstall clears all graph
state so batch programs cannot leak values to one another.

## Packaging

- The provider depends only on native plugin SPI/v2 core, Scala runtime, and
  JDK.
- VM/direct ASM/build-jvm share one implementation and ServiceLoader entry.
- Standard/slim images and deterministic artifacts require its jar.
- Scala 3 seed/image sources remain unchanged.

## Out of scope

- TinkerGraph/RDF4J/Neo4j/JanusGraph dependencies, persistence, network I/O,
  credentials, discovery, remote transactions, query parsing, and browser-local
  graphs;
- deriving codecs or converting source values into backend-native models;
- transparent fallback to the v1 graph plugin, compatibility frontend, or JVM
  compiler.
