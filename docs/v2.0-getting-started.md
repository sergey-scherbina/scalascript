# v2.0 Separate Compilation — Getting Started

This is a hands-on, copy-pasteable walkthrough of ScalaScript's v2.0
separate-compilation pipeline.  Goes from "I have an `.ssc` file" to
"I have a JAR + source maps + LSP integration" in ~15 minutes.

For the design rationale + wire format, see
[`docs/separate-compilation-plan.md`](separate-compilation-plan.md)
and [`docs/v2.0-artifact-format.md`](v2.0-artifact-format.md).

---

## What v2.0 gives you

The default `ssc compile foo.ssc` parses, type-checks, normalises, and
emits the entire reachable module-tree in **one pass**.  Re-compile any
file in the tree and the whole thing re-emits.

v2.0 makes each `.ssc` an independent compilation unit:

```
foo.ssc  ──ssc compile-jvm──▶  foo.scim   (interface — what foo exports)
                              foo.scir   (IR — foo's compiled body)
                              foo.scjvm  (per-module JVM bytecode)
```

Re-compile `foo.ssc` → only `foo.*` re-emit; consumers of `foo` stay
cached.  Per-module .scjvm is ~10 KB; the JVM runtime preamble lives
in a shared `_runtime.scjvm-runtime` artifact (~570 KB, compiled once).

Real production numbers on a trivial 2-module fixture:

| Metric | Before v2.0 | After v2.0 Phase 4 |
|---|---|---|
| `compile-jvm` per module (warm) | 1410 ms (via `scala-cli`) | **119 ms** (in-process Driver) |
| `.scjvm` size per module | ~515 KB | **10 KB** (51× smaller) |
| `.scjs` size per module | 80–150 KB | **500 bytes** (200× smaller) |
| Linked JAR (2 modules) | 554 KB | 301 KB |
| Stack-trace line in IDE | `.scala` synthetic | `.ssc` original |

## Step 1 — single file, JVM

Create `a.ssc`:

````markdown
---
name: a
---

# Module A

```scalascript
def add(x: Int, y: Int): Int = x + y
def boom(): Int = throw new RuntimeException("kaboom")
println("a.add(2, 3) = " + add(2, 3))
```
````

Compile to a JVM artifact:

```bash
ssc compile-jvm --bytecode a.ssc -o artifacts/a.scjvm
```

What just happened:
- The compiler parsed `a.ssc`, type-checked it, ran `JvmGen` on the
  parsed module, and ran the **in-process** Scala 3 compiler driver
  (Phase 3 — no subprocess).
- The `.class` files were base64-encoded into a ZIP and stored in the
  `classBundle` field of `artifacts/a.scjvm`.
- A companion `_runtime.scjvm-runtime` was emitted alongside (it carries
  the JVM runtime preamble — `effectsRuntime`, etc. — compiled once and
  shared by all modules in this artifact dir).

Inspect:

```bash
ssc info artifacts/a.scjvm
# format: .scjvm (JVM-backend cached source)
# magic: SSCART
# abiVersion: 2.0
# moduleId: a
# sourceHash: 0d2a3e...
# size: 10238 bytes
# scalaSourceBytes: 312
# imports: 0

ssc info artifacts/_runtime.scjvm-runtime
# format: .scjvm-runtime
# capabilities: [Core]
# ...
```

Link to a JAR and run:

```bash
ssc link --backend jvm --bytecode artifacts/ -o out.jar
java -cp out.jar:$(coursier fetch -p org.scala-lang:scala3-library_3:3.7.2) a_sc
# → a.add(2, 3) = 5
```

The `$(coursier ...)` bit gives `java` the Scala 3 standard library on
its classpath — ScalaScript artifacts don't bundle it.  In CI / dev:

```bash
export SCALA_STDLIB=$(coursier fetch -p org.scala-lang:scala3-library_3:3.7.2)
java -cp out.jar:$SCALA_STDLIB a_sc
```

## Step 2 — single file, JS

Same source, JS backend:

```bash
ssc compile-js a.ssc -o artifacts/a.scjs
ssc link --backend js artifacts/ -o out.js
node out.js
# → a.add(2, 3) = 5
```

Same shape — `.scjs` (~500 bytes) + `_runtime.scjs-runtime` (~142 KB
shared) → `out.js`.

## Step 3 — two modules, cross-imports

Create `b.ssc` that uses `a`:

````markdown
---
name: b
---

# Module B

[add](./a.ssc)

```scalascript
val sum = add(7, 8)
println("b: a.add(7, 8) = " + sum)
```
````

`[add](./a.ssc)` is the import: bind `add` from `a.ssc` into b's scope.

Compile + link both (note: only `b.ssc` is the target; `a.ssc` is
auto-resolved as a dep):

```bash
ssc compile-jvm --bytecode b.ssc -o artifacts/b.scjvm --iface-dir artifacts/
```

What happened:
- `compile-jvm` walked b's import graph: found `a.ssc` as a dep.
- Auto-resolved: compiled `a.ssc` first (writing `artifacts/a.scjvm`),
  then `b.ssc` against `a`'s interface (`artifacts/a.scim`).
- `--iface-dir artifacts/` told the consumer's typer where to find
  imported interfaces.

Want to see the dep graph without compiling?

```bash
ssc deps b.ssc --graph
# target: /path/to/b.ssc
# resolved 2 module(s) in topo order:
#   /path/to/a.ssc
#   /path/to/b.ssc (target)
# edges:
#   b.ssc → a.ssc
```

Link and run:

```bash
ssc link --backend jvm --bytecode artifacts/ -o out.jar
java -cp out.jar:$SCALA_STDLIB b_sc
# → b: a.add(7, 8) = 15
```

## Step 4 — incremental build

Got a whole directory of `.ssc` files?  Use `build --incremental`:

```bash
ssc build --incremental --backend jvm src/
# Discovered 5 module(s) in src
#   [compile]  a.ssc        ... OK
#   [compile]  b.ssc        ... OK
#   [compile]  c.ssc        ... OK
#   [compile]  d.ssc        ... OK
#   [compile]  e.ssc        ... OK
# Done: 5 compiled, 0 up-to-date, 0 failed
```

Re-run without changes:

```bash
ssc build --incremental --backend jvm src/
# Done: 0 compiled, 5 up-to-date, 0 failed
```

Touch one file:

```bash
touch src/c.ssc
ssc build --incremental --backend jvm src/
# Done: 1 compiled, 4 up-to-date, 0 failed
```

Behind the scenes: every artifact carries a SHA-256 of its source.
Re-running compares hashes; matches are skipped.

### Section-level caching (opt-in)

If you have ONE big `.ssc` with many code blocks, you can cache at
the **section** level too:

```bash
ssc build --incremental --backend jvm --section-cache src/
```

Section caching uses a cumulative hash chain: edit the LAST section
of a 5-section module → only the last section re-emits.  Edit the
FIRST → everything cascades (the safe, conservative default).

Want body-only edits to NOT cascade?  Use interface-based caching
(Phase 5):

```bash
ssc build --incremental --backend jvm --section-cache=interface src/
# Edit section 1's def body but not its signature →
# sections 2-5 stay fresh.
```

## Step 5 — strict type-checking

By default, `ssc compile` and `ssc check` are **permissive** on
undefined names — the typer leaves them as `Any` (matches the
interpreter, where intrinsics get injected late).

For interface-aware code review, opt into `strict`:

```bash
ssc check-with-iface --iface-dir artifacts/ b.ssc
# exit 0 — all references resolve through the iface dir
```

Misspell a name:

```bash
# b.ssc references `addd(7, 8)` (typo)
ssc check-with-iface --iface-dir artifacts/ b.ssc
# Reference to undefined name: addd
# at b.ssc:7:11
# exit 1
```

Strict mode catches:
- Bare-name undefined refs (`addd`)
- `pkg.member` where `pkg` is an imported module but `member` isn't
  in its `.scim` exports
- Deep `a.b.c` chains where any link is missing from the chain
  (`std.dsl.nonExistent` → diagnostic about `std.dsl`)

It deliberately does NOT flag:
- Bare names that look like interpreter intrinsics (`println`,
  `runActors`, `effect`, etc. — there's a 60-entry prelude allowlist)
- Method selectors on values whose type the typer doesn't know
  (e.g. `someValue.toString` where `someValue: Any`)

## Step 6 — source maps

By default a JVM stack trace from a v2.0 JAR shows `.scala` synthetic
line numbers (from the emitted Scala source).  Useful, but a step
removed from your `.ssc`.

Opt into source maps:

```bash
ssc link --backend jvm --bytecode --source-map artifacts/ -o out.jar
```

This injects JSR-45 `SourceDebugExtension` attributes into every
`.class` (via ASM).  Then:

- **IntelliJ / VS Code Metals / `jdb` / any JDI-aware tool**: stack
  frames resolve to `.ssc` lines automatically.
- **Raw `java -cp ...` (no debugger)**: still shows `.scala` lines.
  This is a JVM platform limit — `Throwable.printStackTrace` doesn't
  consult `SourceDebugExtension`.  Kotlin, Scala 2, and every other
  language built on JSR-45 has the same caveat.

For JS:

```bash
ssc link --backend js --source-map artifacts/ -o out.js
# → emits out.js + out.js.map (V3 spec); browsers and Node devtools
#   resolve frames to .ssc automatically.
```

## Step 7 — health check + cleanup

`ssc verify` walks an artifact dir and validates every artifact:

```bash
ssc verify artifacts/
# [OK] a.scim         envelope OK, sourceHash OK, 2 exports
# [OK] a.scjvm        envelope OK, sourceHash OK, classBundle 10 KB, caps [Core]
# [OK] b.scim         envelope OK, sourceHash OK, 1 exports, imports a
# [OK] b.scjvm        envelope OK, sourceHash OK, classBundle 11 KB, caps [Core]
# [OK] _runtime.scjvm-runtime   envelope OK, classBundle 570 KB, caps [Core]
# verify: 5 OK, 0 WARN, 0 FAIL
```

`--strict` also checks source freshness (re-hashes every `.ssc`
referenced by an artifact):

```bash
ssc verify --strict --src-dir src/ artifacts/
# Adds [WARN] entries for any source whose hash diverged from the
# artifact's stored sourceHash.
```

`--json` for CI:

```bash
ssc verify --json artifacts/ | jq '.summary'
# { "ok": 5, "warn": 0, "fail": 0 }
```

Deleted a source file?  Garbage-collect its artifacts:

```bash
rm src/c.ssc
ssc clean artifacts/
# Removed: c.scim, c.scir, c.scjvm
# 3 file(s) removed.
```

Dry-run for CI safety:

```bash
ssc clean --dry-run artifacts/
# DRY-RUN c.scim
# DRY-RUN c.scir
# DRY-RUN c.scjvm
```

Wipe everything:

```bash
ssc clean --all artifacts/
# Removed: a.scim, a.scjvm, b.scim, b.scjvm, _runtime.scjvm-runtime, ...
```

## Step 8 — IDE integration (LSP)

ScalaScript ships a minimal Language Server.  It speaks JSON-RPC
over stdio and supports:

- `initialize` / `shutdown`
- `textDocument/didOpen` / `didChange` / `didClose`
- `textDocument/definition` (go-to-definition)
- `textDocument/hover` (type info on hover)
- `textDocument/publishDiagnostics` (live parse + type errors)

VS Code config (`.vscode/settings.json`):

```json
{
  "scalascript.lsp.command": "ssc",
  "scalascript.lsp.args": ["lsp"],
  "scalascript.lsp.initializationOptions": {
    "artifactDir": "${workspaceFolder}/artifacts"
  }
}
```

(You'll need an LSP client extension — generic-LSP clients work.)

What you get:
- **Go-to-definition** on imported symbols resolves through the
  `artifactDir`'s `.scim` files to the correct `.ssc` source line
  (Phase 4 — `ExportedSymbol.definitionLine` is populated by the
  extractor from scalameta positions).
- **Hover** shows the symbol's type.
- **Diagnostics** fire on `didOpen`/`didChange` — both parse errors
  (with line:col snippet) and strict-mode undefined-name diagnostics.

## Step 9 — consuming external packages

Library authors publish `.sscpkg` archives.  Consumers reference them
in their front-matter `dependencies:` map:

```markdown
---
name: my-app
dependencies:
  myLib: "dep:org.example/myLib:1.0"
---

[hello](myLib://myLib.ssc)

```scalascript
hello()
```
```

`ssc compile-jvm` auto-resolves the dep through the dep cache
(`~/.cache/scalascript/deps/`).  When the cached `.sscpkg` carries
pre-compiled artifacts (Phase 5: `ssc package --with-artifacts`),
the consumer uses those directly — no re-parse.

## CLI reference

```
ssc compile-jvm <file.ssc> [-o out.scjvm] [--bytecode] [--iface-dir DIR]
                           [--artifact-dir DIR] [--no-auto-deps]
ssc compile-js  <file.ssc> [-o out.scjs]  [--iface-dir DIR]
                           [--artifact-dir DIR] [--no-auto-deps]
ssc compile-runtime --capabilities <list> --backend {jvm|js}

ssc emit-interface <file.ssc> [-o out.scim | -]
ssc emit-ir        <file.ssc> [-o out.scir | -]

ssc check-with-iface --iface-dir DIR <file.ssc>

ssc link --backend {jvm|js} <artifact-dir>
         [--bytecode] [--source-map] [-o out.{jar|js|scir}]

ssc build --incremental --backend {jvm|js} <src-dir>
          [--section-cache[=interface]]

ssc info   <artifact> [--json] [--sections]
ssc verify <artifact-dir> [--strict] [--src-dir DIR] [--json]
ssc clean  <artifact-dir> [--dry-run] [--all]
ssc deps   <file.ssc> [--graph]
ssc lsp                  # stdio server
```

All commands accept `--help`.
