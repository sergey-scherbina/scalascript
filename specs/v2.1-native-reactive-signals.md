# ScalaScript 2.1 core-free reactive signals

## Overview

Provide the established general-purpose `Signal`, `computed`, and `effect`
runtime on the compiler-free 2.1 JVM route through a dedicated native standard
provider. This runtime is separate from the provenance-rich NativeUi ABI signal
store: general reactivity has no view/root ownership and must remain usable in
ordinary scripts and standalone `build-jvm` artifacts.

## Interface

- `Signal(initial)` returns a mutable signal supporting `get`, zero-argument
  `apply`, `set(value)`, and zero-argument tagged application `signal()`;
- `computed { body }` returns a read-only signal whose value is recomputed when
  any signal read by its updater changes;
- `effect { body }` runs once immediately, tracks every signal read, and reruns
  after a dependency changes; it returns `Unit`.

## Behavior

- [ ] mutable signals return the current value through `get`/`apply`/`signal()`
  and synchronously expose successful `set` writes;
- [ ] computed signals track dependencies and update before a downstream effect
  observes the transaction;
- [ ] effect dependencies are replaced on every run, so conditional reads do
  not retain stale subscriptions;
- [ ] one write flushes subscribers in insertion order and runs a diamond
  consumer at most once even when both a source and its computed derivative are
  read;
- [ ] an effect may write a signal it currently reads without recursively
  scheduling itself, while other subscribers still observe the write;
- [ ] `signals-demo.ssc` has stable exact output on native VM/direct ASM and a
  standalone `build-jvm` JAR;
- [ ] provider ownership, dependency/class-load, no-compiler JRE, slim,
  corpus/parity/taxonomy/portable, and fresh conformance gates pass without a
  mismatch or one-sided row.

## Out of scope

- asynchronous/batched scheduling, explicit disposal, weak subscriptions, or
  cross-thread mutation;
- UI root/component lifecycle, fetch descriptors, persistence, or Apple/JVM UI
  observation stores;
- changing compatibility interpreter or browser signal implementations.

## Design

Create `v2/runtime/std/reactive-plugin` depending only on the native plugin SPI.
Public handles are portable `DataV("ReactiveSignal", [id])` values. Provider-
local maps own mutable signal/effect state; no `ForeignV`, v1 interpreter value,
or UI ABI object crosses the boundary. Tagged apply/method registrations resolve
the id and implement reads/writes.

Each effect owns an insertion-ordered dependency set. A read while an effect is
active records both sides of the subscription. Before rerunning an effect, the
provider removes its previous dependencies and collects the new ones. A
provider-wide insertion-ordered pending set forms the synchronous flush queue.
Computed values reuse the same mechanism: an internal updater effect writes a
private read-only signal, so it is subscribed before later consumers and their
diamond paths deduplicate naturally.

## Decisions

- **Dedicated general provider** — keeps non-UI semantics independent from
  NativeUi root/provenance lifecycle. Rejected: reusing `NativeUiSignal`, which
  would couple ordinary scripts to UI installation and ownership.
- **Portable id handle plus provider-local state** — works through VM, direct
  ASM, and packaged JARs without embedding JVM objects in CoreIR values.
- **Synchronous insertion-ordered flush** — preserves deterministic established
  behavior and exact output. Rejected: a thread pool/event loop, which would add
  nondeterminism and a new host capability.
