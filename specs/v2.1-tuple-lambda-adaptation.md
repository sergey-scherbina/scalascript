# ScalaScript 2.1 tuple-lambda collection adaptation

## Overview

The standard native runtime must preserve the existing ScalaScript collection
contract in which a multi-parameter callback can consume one tuple element.
Tuple literals lower to the portable internal `Pair/2` representation while
some host-backed collection operations produce `Tuple2/2`; callback behavior
must not depend on which equivalent representation supplied the element.

## Interface

```scalascript
val pairs = List(("parse", 1), ("check", 2))
val labels = pairs.map((name, value) => s"$name=$value")
val values = pairs.flatMap((name, value) => List(value))
```

For `List.map` and `List.flatMap`, a callback whose declared arity equals the
field count of the single tuple element receives those fields in declaration
order. Both portable `Pair/2` and runtime `Tuple2/2` are source-level
two-element tuples for this rule.

An arity-one callback still receives the complete tuple value. A callback used
as an ordinary function and called with two explicit arguments remains an
ordinary arity-two function.

## Behavior

- [x] `List.map` spreads one `Pair/2` element into an arity-two callback once,
      in field order, on native VM and direct ASM.
- [x] `List.flatMap` applies the same rule and preserves its ordinary result
      flattening and effect sequencing.
- [x] Runtime-produced `Tuple2/2` and portable/imported `Pair/2` elements are
      observationally identical at the callback boundary.
- [x] Arity-one callbacks receive the tuple intact; direct two-argument calls,
      folds, and binary comparators keep their current contracts.
- [x] A field-count mismatch follows the ordinary arity diagnostic and is not
      retried, reflected, or silently padded.
- [x] `dsl-mini-language.ssc` prints the exact 13-line compatibility output on
      VM, direct ASM, and `build-jvm` together with the parameterless-def fix.
- [x] Strict parity retires only the mini-language row; release and fresh
      affected `v2-*` conformance gates remain green with zero mismatches and
      one-sided backend errors.

## Design

Keep adaptation at the collection-callback invocation seam already owned by
the shared v2 runtime. Before `map` or `flatMap` calls a multi-arity closure,
inspect the one element: spread fields only for `Pair` or `TupleN` when the
field count exactly equals the closure arity; otherwise make the ordinary
single-argument call and retain its exact arity failure.

The frontend continues lowering `(a, b) => body` as an ordinary arity-two
closure. No tuple-specific closure kind enters CoreIR, so direct calls and
other binary operations retain the same ABI.

## Decisions

- **Adapt at the existing collection seam** — chosen because the runtime
  already implements this contract for `TupleN`; adding `Pair` reconciles the
  representation boundary without changing CoreIR. Rejected: globally
  treating every arity-two closure as a tuple lambda (would change direct
  calls and obscure real arity errors).
- **Exact tag and arity check** — chosen to keep arbitrary two-field ADTs
  distinct. Rejected: spreading every `DataV` with two fields.
- **No failure retry** — chosen to preserve effects and deterministic errors.
  Rejected: call with one argument, catch an arity error, then retry with
  fields (could execute user code twice and conflate body failures with ABI).

## Out of scope

- changing tuple serialization or replacing `Pair/2` with `Tuple2/2` globally;
- adding adaptation to unrelated collection families or binary operations;
- repairing derives/Mirror, direct-do, optics, or SQL blockers.

## Results

`12d3d9cf2` extends the existing exact `TupleN` callback seam to the portable
`Pair` tag for `List.map` and `List.flatMap`; the closure arity and field-count
guard are unchanged. The shared VM/direct-ASM runtime therefore performs one
effect-aware call with the two fields, while every non-tuple or mismatched
value follows the existing single-argument path and exact arity diagnostic.

The complete public mini-language pipeline is byte-identical across VM, direct
ASM, and `build-jvm`. Strict focused parity is 1 identical / 0 both-fail, and
the exhaustive 195-file release gate is 48 identical / 18 both-fail / 129
skipped with zero mismatch/one-sided rows and fresh conformance 11/11.
