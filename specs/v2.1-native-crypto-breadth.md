# ScalaScript 2.1 core-free crypto breadth

## Overview

Complete the standard native JVM crypto provider so the compiler-free 2.1 route
implements the established `std.crypto` surface without loading the v1
interpreter bridge, Scalameta, Scala CLI, a Scala/Java compiler, or third-party
crypto libraries. The behavior is a direct port of the existing v1
`crypto-plugin` contracts and tests, not a new cryptographic design.

## Interface

In addition to the already-native hash, HMAC, PBKDF2, random-byte, and Base64
functions, `ssc.plugin.crypto.CryptoNativePlugin` owns these standard globals:

- AES: `aesGenKey`, `aesGenIv`, `aesGcmEncrypt`, `aesGcmDecrypt`,
  `aesGcmEncryptBytes`, `aesGcmDecryptBytes`, `aesCbcEncrypt`, and
  `aesCbcDecrypt`;
- public-key/certificate: `x509PublicKey`, `rsaOaepEncrypt`,
  `verifyEd25519`, `verifyEd25519Url`, `verifyRsaSha256`, `ed25519Sign`,
  `ed25519SignUrl`, and `rsaSignSha256`;
- portable utilities: `totp`, `totpValidate`, `shamirSplit`, and
  `shamirRecover`.

String and byte payloads retain the existing Base64 framing documented by
`std.crypto`. Ed25519 accepts raw 32-byte public keys/seeds as well as standard
SPKI/PKCS#8 encodings. RSA signatures support `PKCS1` and `PSS`.

## Behavior

- [ ] AES-256-GCM round-trips UTF-8 and arbitrary bytes with a fresh 12-byte IV
  framed before ciphertext/tag; authentication failures are not swallowed.
- [ ] AES-256-CBC uses a caller-supplied 16-byte IV, PKCS#7 padding, and returns
  ciphertext without framing the IV.
- [ ] X.509 extraction produces Base64 SPKI DER and RSA-OAEP uses SHA-256 for
  both OAEP and MGF1.
- [ ] Ed25519 signing reproduces RFC 8032 vectors; standard/Base64url
  verification returns `false` for malformed or hostile input rather than
  throwing.
- [ ] RSA PKCS#1/PSS sign and total-verify contracts match the existing plugin.
- [ ] RFC 6238 TOTP matches fixed SHA-1/256/512 vectors and validation honors the
  requested symmetric window.
- [ ] Shamir splitting validates `2 <= threshold <= total <= 255`; any threshold
  subset recovers the original bytes, while malformed/inconsistent shares fail
  explicitly.
- [ ] `crypto-encrypt-demo.ssc`, `crypto-verify-demo.ssc`, and
  `totp-shamir-demo.ssc` complete with stable observable output on native VM and
  direct ASM.
- [ ] the provider module remains core-free and every native-entry/corpus/parity/
  taxonomy/portable/conformance release gate passes with no mismatch or
  one-sided row.

## Out of scope

- Browser/WebCrypto or non-JVM provider implementations.
- New algorithms, key formats, password vaults, or protocol-level key storage.
- Replacing the JDK JCA/JCE runtime; only compiler modules are forbidden.
- Changing the existing `std.crypto` declarations or compatibility behavior.

## Design

Port the proven helpers from the v1 crypto plugin into the v2 provider using
JDK `Cipher`, `Signature`, `KeyFactory`, `Mac`, `CertificateFactory`, and
`SecureRandom`. Keep conversion and GF(256) Shamir helpers private to the native
plugin. Register each operation as both intrinsic and global through the
existing `NativePluginContext`; do not introduce a bridge dependency.

Unit tests reuse published RFC vectors and deterministic round trips. Assembled
tests run the three existing user examples through both native execution lanes.
Only after those programs are identical may their three standard-provider rows
leave runtime taxonomy.

## Decisions

- **Port established code into the core-free provider** — chosen because its
  semantics and adversarial cases are already exercised. Rejected: delegating
  to the v1 plugin, which would reintroduce the forbidden compatibility graph.
- **Use JDK JCA/JCE** — chosen because it is present in the allowed JVM runtime
  and is independently maintained. Rejected: a new crypto dependency or
  hand-written AES/RSA/Ed25519.
- **Keep verifiers total, decrypt/sign operations strict** — preserves the
  public contract: hostile verification input yields `false`, while trusted
  authoring/decryption mistakes remain explicit errors.
