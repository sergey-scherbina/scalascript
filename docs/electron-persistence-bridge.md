# Electron Persistence Bridge

> Status: Phase 1 skeleton landed - May 2026.

This spec describes the planned replacement for the current
localStorage-backed Electron SQL fallback documented in
[`electron-sql.md`](electron-sql.md).

The target outcome: `sqlite:<path>` in an Electron desktop bundle should mean a
real SQLite database stored under Electron's app data directory, while keeping
the renderer sandboxed (`nodeIntegration: false`, `contextIsolation: true`).

## Goals

- Provide durable SQLite persistence for Electron desktop apps generated by
  `ssc run --frontend electron` and `ssc build --target desktop`.
- Keep filesystem access in the Electron main process, not in renderer code.
- Preserve the existing `.ssc` source surface:
  - `databases:` front-matter;
  - `sql` fenced blocks;
  - `Db.query(name, sql, params)`;
  - `Db.execute(name, sql, params)`.
- Keep same-app `fetch("/api/...")` route dispatch working inside the renderer,
  so `fetchAction` and `fetchTable` continue to call `.ssc` REST handlers.
- Restrict the bridge to databases declared in the bundle manifest. Renderer
  code must not be able to open arbitrary filesystem paths.
- Add restart-level smoke coverage for persistence, not only in-memory behavior.

## Non-Goals

- Do not expose general Node APIs to renderer code.
- Do not enable `nodeIntegration`.
- Do not expose arbitrary filesystem reads/writes through preload.
- Do not implement PostgreSQL/MySQL/remote database clients in Electron main.
- Do not replace the browser SQL runtime for non-Electron web builds.
- Do not make the localStorage fallback disappear immediately; it remains the
  web/browser fallback and a transition path.

## Current State

Today, Electron bundles are generated by `ElectronBundleBuilder`:

- `main.js` creates a `BrowserWindow` and loads `index.html`.
- For bundles with `databases:` front-matter, `main.js` registers the Phase 1
  database IPC handlers and `preload.js` exposes `window.__sscElectron.db`.
  For pure UI bundles, `preload.js` remains inert.
- `app.js` contains the generated ScalaScript JS runtime, the SPA patch, and
  embedded `backend-sql-runtime-js`.
- The renderer has no Node filesystem access.

As a result, `sqlite:<path>` in Electron renderer code currently uses a narrow
localStorage-backed fallback. This is useful for demos, but it is not real file
SQLite and should not be presented as durable desktop persistence.

## Architecture

### Module Layout

The implementation should stay in the existing Electron frontend module:

```text
frontend/electron/
  src/main/scala/scalascript/frontend/electron/
    ElectronBundleBuilder.scala
    ElectronEmitter.scala
    ElectronPersistenceBridge.scala      # proposed generator helpers
  src/test/scala/scalascript/frontend/electron/
    ElectronEmitterTest.scala
```

`tools/cli` should remain a thin caller of `ElectronBundleBuilder`.

### Generated Files

The generated Electron bundle gains an active preload bridge:

| File | Change |
| --- | --- |
| `main.js` | Registers IPC handlers for SQL connection/query/execute/close. Owns SQLite connections and filesystem paths. |
| `preload.js` | Exposes a narrow `window.__sscElectron.db` API via `contextBridge`. |
| `app.js` | Detects `window.__sscElectron.db`; for declared `sqlite:<path>` databases, routes `Db.query` / `Db.execute` through the bridge. |
| `package.json` | Includes the chosen SQLite dependency once implementation selects it. |

The bridge should be generated only when the parsed module declares at least one
database URL that needs Electron persistence. Pure UI apps keep the inert preload.

### Database Path Resolution

`sqlite:<path>` is resolved in main process:

| Source URL | Main-process file path |
| --- | --- |
| `sqlite::memory:` | In-memory connection; no file. |
| `sqlite:` | In-memory connection; no file. |
| `sqlite:./todos.db` | `path.join(app.getPath("userData"), "todos.db")` |
| `sqlite:todos.db` | `path.join(app.getPath("userData"), "todos.db")` |
| `sqlite:/absolute/path.db` | Rejected by default. |
| `sqlite:../outside.db` | Rejected after path normalization. |

Initial policy: bundle-local database names map to app-data files. Absolute and
parent-relative paths are rejected to avoid turning the renderer bridge into a
filesystem escape hatch. A future explicit `allow-absolute-sqlite-paths` build
flag could relax this for developer tools, but that is out of scope for the
first implementation.

### Main Process Contract

The main process owns a registry derived from front-matter:

```json
{
  "default": { "url": "sqlite:./todos.db" },
  "audit":   { "url": "sqlite:./audit.db" }
}
```

Only keys present in that registry may be opened.

IPC channels:

| Channel | Direction | Payload | Result |
| --- | --- | --- | --- |
| `ssc:db:query` | renderer -> main | `{ dbName, sql, params }` | `{ ok: true, rows }` or `{ ok: false, error }` |
| `ssc:db:execute` | renderer -> main | `{ dbName, sql, params }` | `{ ok: true, count }` or `{ ok: false, error }` |
| `ssc:db:close` | renderer -> main | `{ dbName }` | `{ ok: true }` or `{ ok: false, error }` |
| `ssc:db:list` | renderer -> main | `{}` | `{ ok: true, names }` |

`rows` is an array of plain objects keyed by result column names. `params` is a
JSON-compatible array. Binary values, dates, and bigint need a later encoding
policy; Phase 1 can reject them with a clear error.

### Preload Contract

`preload.js` exposes only this surface:

```javascript
contextBridge.exposeInMainWorld('__sscElectron', {
  db: {
    query(dbName, sql, params)   { return ipcRenderer.invoke('ssc:db:query', { dbName, sql, params }) },
    execute(dbName, sql, params) { return ipcRenderer.invoke('ssc:db:execute', { dbName, sql, params }) },
    close(dbName)                { return ipcRenderer.invoke('ssc:db:close', { dbName }) },
    list()                       { return ipcRenderer.invoke('ssc:db:list', {}) }
  }
})
```

No raw `ipcRenderer` object is exposed.

### Renderer Integration

`backend-sql-runtime-js` should keep the existing provider split:

- browser/web: existing localStorage fallback for `sqlite:<path>`;
- Node: current `fs/promises` sql.js path;
- Electron renderer: bridge provider when `window.__sscElectron?.db` exists.

The bridge provider should implement the same `Connection` contract used by
`ConnectionRegistry`, so `sql` blocks and `Db.query`/`Db.execute` do not need
source-level changes.

### SQLite Engine Choice

Two implementation options are acceptable:

1. **`better-sqlite3` in main process**
   - Pros: synchronous API, simple prepared statements, mature file persistence.
   - Cons: native dependency; packaging needs electron-builder native rebuild
     handling.
2. **`sql.js` in main process**
   - Pros: pure JS/wasm; closer to existing JS runtime.
   - Cons: explicit export/writeback lifecycle; persistence is easier to get
     wrong.

Decision for Phase 2: use **`better-sqlite3`** in the main process. The
synchronous API keeps IPC handlers simple and maps directly to prepared
statement query/execute semantics. The native dependency and packaging rebuild
path are accepted risks for Phase 2/4 because this bridge targets desktop
Electron bundles, not zero-install browser output.

## Migration

The migration should be additive:

1. Keep current localStorage fallback unchanged for browser and web builds.
2. Generate bridge-enabled Electron bundles for modules with `databases:`.
3. In Electron renderer, prefer bridge provider over localStorage fallback.
4. Update `electron-sql.md` once bridge lands:
   - current fallback section becomes "legacy/browser fallback";
   - `sqlite:<path>` in Electron becomes app-data file persistence.

Existing `.ssc` source files do not change.

## Phases

### Phase 0 - Spec And Contract

- Land this design document.
- Keep current runtime behavior unchanged.
- Decide `better-sqlite3` vs `sql.js` before Phase 1 implementation.

### Phase 1 - Bridge Skeleton

- Generate active `preload.js` only when the bundle has declared databases.
- Expose `window.__sscElectron.db.list()`.
- Register main-process IPC handlers from the declared database manifest.
- Add a smoke test that asserts declared names are visible through preload.

This phase does not execute SQL yet.

Landed in May 2026:

- `ElectronPersistenceBridge` generates the main/preload bridge.
- `ElectronBundleBuilder` passes parsed `manifest.databases` into the Electron
  emitter, so `ssc run --frontend electron` and desktop bundle generation get
  the bridge for `.ssc` files that declare databases.
- `ssc:db:list`, `ssc:db:query`, `ssc:db:execute`, and `ssc:db:close` handlers
  are registered; query/execute deliberately return a clear "not implemented"
  result until Phase 2 wires `better-sqlite3`.
- `ToolkitElectronSmokeTest` verifies `window.__sscElectron.db.list()` exposes
  the `default` database before exercising the existing Add flow.

### Phase 2 - SQLite Query/Execute

- Implement `query` and `execute` for `sqlite:<path>` and `sqlite::memory:`.
- Add renderer provider selection in `backend-sql-runtime-js`.
- Preserve current localStorage fallback when bridge is absent.
- Extend `ToolkitElectronSmokeTest` to assert Add still works through the bridge.

### Phase 3 - Restart Persistence

- Add an Electron smoke that:
  1. starts the app;
  2. inserts a row;
  3. exits;
  4. starts the same bundle again;
  5. verifies the row is still present.
- Use a temporary app-data root for tests so the developer profile is not
  polluted.

### Phase 4 - Packaging Hardening

- Ensure `ssc build --target desktop` includes the SQLite dependency and any
  native/wasm assets.
- Verify packaged macOS/Linux/Windows artifacts can open the database.
- Document any platform-specific build prerequisites.

## Testing Strategy

- Unit tests:
  - path normalization rejects absolute and parent-relative paths;
  - generated preload exposes only `__sscElectron.db`;
  - generated main process includes handlers only when databases are present.
- Smoke tests:
  - bridge list smoke;
  - toolkit Add flow through bridge;
  - restart persistence smoke.
- Regression tests:
  - no bridge for pure UI Electron bundles;
  - browser SQL fallback still works when no Electron bridge exists;
  - Node `sqlite:<path>` behavior remains file-backed.

Electron-dependent tests should continue to cancel when `electron` is not on
`PATH`, unless CI explicitly opts in to a required Electron smoke lane.

## Security Notes

- Renderer never receives `ipcRenderer`, `fs`, `path`, or arbitrary Node APIs.
- Renderer may only address database names declared in the generated bundle.
- Main process validates SQL params are JSON-compatible before execution.
- Main process rejects absolute/escaping SQLite paths.
- SQL injection remains the caller's responsibility for raw SQL strings; bind
  params continue to be the safe path.
- Error payloads should include message and error code, not arbitrary stack
  traces, unless a debug build flag is enabled.

## Open Questions

- Should the first implementation use `better-sqlite3` or `sql.js` in main?
- Should Electron app-data path be overrideable for tests through environment
  variable, CLI flag, or generated `package.json` script?
- Do we need transaction support in Phase 2, or can it wait until Phase 3/4?
- Should `sqlite:/absolute/path.db` ever be allowed for developer builds?
- How should non-JSON parameter/result types be encoded across IPC?
