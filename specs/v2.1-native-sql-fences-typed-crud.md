# ScalaScript 2.1 native SQL fences and typed CRUD

## Overview

The compiler-free standard frontend must retain executable `sql` fences in
document order, rewrite every `${expr}` to a JDBC bind, and expose the first
result in a section as `<Section>.sql`. The existing core-free SQL provider
then owns raw fence execution plus the bounded typed `RowCodec` CRUD surface on
VM, direct ASM, slim/JRE, and reproducible `build-jvm`, without loading the v1
SQL runner, Scalameta, PluginBridge, or a Scala/Java compiler.

## Interface

````scalascript
```sql @db=reports
SELECT id, name FROM users WHERE active = ${onlyActive}
```

val rows = Users.sql

case class Todo(id: Int, text: String, done: Boolean) derives RowCodec
Db.insert("default", "todos", Todo(1, "Buy milk", false))
Db.update("default", "todos", "id", 1, Todo(1, "Buy oat milk", true))
val todos = Db.query[Todo]("default", "SELECT id, text, done FROM todos", [])
````

Each SQL fence receives a stable source-order `_sqlBlock_N` binding. The first
SQL fence under a usable heading also installs one structural section value;
later SQL fences in that section do not replace the alias. `@db=name` selects a
declared front-matter database and defaults to `default`.

`Db.query[A]`, `Db.insert`, and `Db.update` support registered portable product
types whose fields are simple JDBC scalar values (`Boolean`, integral/floating
numbers, exact Decimal, String, bytes, and `Option`/null binds). Table, column,
and key names accepted by generated writes are plain SQL identifiers only.

## Behavior

- [x] SQL fences execute exactly once in document order before later
      ScalaScript statements consume their results.
- [x] Every `${expr}` becomes one `?` and one source-ordered bind expression;
      `$$` is a literal dollar and malformed bind syntax fails before JDBC.
- [x] `@db=name` and default database selection use only the structural native
      front-matter config and retain lazy connection creation.
- [x] `_sqlBlock_N` stores every result and the first fence in a usable section
      is reachable as `<Section>.sql` on VM, ASM, and build-jvm.
- [x] Raw SQL results are ordinary portable lists/maps for row statements and
      `Int` update counts for DDL/DML.
- [ ] `derives RowCodec` creates a structural descriptor from portable Mirror
      metadata without reflection or compiler-generated host classes.
- [ ] `Db.query[A]` retains the requested nominal product and reconstructs
      fields in declared order using case-insensitive JDBC labels.
- [ ] `Db.insert` emits an identifier-validated, fully bound insert from
      registered product fields; `Db.update` excludes the key column from SET
      and binds the explicit key value in the WHERE clause.
- [ ] Missing fields, unsupported product/bind values, malformed binds, bad
      identifiers, unknown databases, and client-only SQL fences produce
      bounded standard-tier diagnostics with no fallback.
- [ ] `sql-h2-quickstart.ssc` prints its two canonical lines and
      `typed-sql-crud.ssc` prints `1/1:Buy oat milk:true` exactly on standard
      VM, direct ASM, and reproducible build-jvm.
- [ ] Provider/unit, dependency/class-load, slim/JRE, 195-row parity/taxonomy,
      and fresh `v2-*` conformance gates pass with blocking both-fail equal to
      zero.

## Design

### Self-hosted document lowering

`ssc1-run.ssc0` already walks headings and fences to produce the executable
ScalaScript projection. It will recognize the exact `sql` language token and
its `@db`/`@side` attributes, scan bind delimiters without parsing SQL, and
inject ordinary ScalaScript statements at the fence position. The injected
surface calls provider-owned `Db.sql(db, rewrittenSql, binds)` and
`sqlSection(result)`; the lowerer emits no example/heading-specific globals.

The scanner follows the global SQL safety contract: it is deliberately not
SQL-string-aware, so `${...}` binds even between SQL quotes. It tracks balanced
ScalaScript braces inside the expression, converts `$$` to one literal dollar,
and emits a bounded sentinel for an unterminated bind. `@side=client` remains a
JS-family backend surface and is rejected by the standard JVM lane rather than
executed server-side.

### Core-free provider

`v2/runtime/std/sql-plugin` adds one result-polymorphic raw operation and the
typed product operations. `SqlRuntime.execute` remains the only JDBC statement
executor. The provider converts JDBC rows to portable values, derives column
order from `V2PluginRegistry` field metadata, and constructs parameterized
INSERT/UPDATE SQL only after validating identifiers. It does not parse SQL,
generate host source, or import v1 AST/interpreter/value types.

`Db.query[A]` retains `A` through a dedicated frontend marker and lowers to a
provider operation carrying the nominal tag. `RowCodec_derived` consumes the
already portable `Mirror(label, elemLabels, elemTypes)` value; no Scala
reflection or runtime compiler participates.

## Decisions

- **Inject portable calls while walking the document** — chosen because this
  is the only self-hosted stage that still owns exact fence/statement order.
  Rejected: reconstructing order from content metadata in the Scala seed or
  reopening the source with a host parser.
- **One raw result-polymorphic provider operation** — chosen because JDBC
  already reports rows versus update count. Rejected: parsing the leading SQL
  keyword in the frontend or maintaining separate query/execute heuristics.
- **Bound values only** — required by SPEC §3.3.1. Rejected: interpolation by
  text substitution or any raw escape.
- **Bounded portable RowCodec subset** — chosen to close the declared standard
  examples truthfully using existing product/Mirror metadata. Rejected:
  importing the full annotation/reflection-driven v1 codec stack.

## Out of scope

- `transaction` fences, multi-statement scripts, PostgreSQL LISTEN/NOTIFY, and
  browser/client SQL execution;
- dynamic SQL identifiers, string-substitution interpolation, SQL grammar
  validation, migrations, ORM relationships, or generated host classes;
- advanced `RowCodec` annotations/aliases/defaults/key inference and unknown-
  column policy beyond the bounded native product contract;
- changing the explicit `ssc-tools --compat-frontend` implementation.

## Audit baseline

Before implementation, both standard engines fail before stdout:

- `sql-h2-quickstart.ssc`: `unbound global: ActiveUsers`; checked CoreIR has no
  SQL fence operation and retains only `ActiveUsers.sql` / `Headcount.sql`
  consumers. Explicit compatibility prints two canonical lines.
- `typed-sql-crud.ssc`: `unbound global: RowCodec_derived`; checked CoreIR
  drops the schema fence, initializes the derived cell, retains insert/update
  product arguments, and erases `Db.query[Todo]` to untyped `Db.query`.
  Explicit compatibility prints `1/1:Buy oat milk:true`.
