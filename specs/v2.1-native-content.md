# ScalaScript 2.1 native structural content

## Overview

The standard ScalaScript 2.1 native path must expose the existing
`std.content` API without loading the v1 content bridge, Scalameta,
CommonMark/Flexmark, or another host parser. The self-hosted frontend already
produces a canonical `MarkdownDocument`, but the Scala 3 seed currently
validates that value and discards everything except its block count. This
feature retains a self-hosted `DocumentContent` projection for every module in
the linked source closure and exposes those immutable values through a
core-free native content provider.

The permanent seed remains Scala 3. It validates and stores frozen structural
ADTs; it does not parse Markdown, heading attributes, fence info strings,
front-matter, or imported module text.

## Interface

### Frozen frontend product

`NativeCompilation/4` keeps its four-field outer tag, but its third field is
promoted from an unlabelled list of root-only Markdown results to a labelled
module-closure content product:

```text
NativeCompilation(
  program: IrProg,
  rootManifests: List[Option[YamlResult]],
  contentModules: List[NativeContentModule],
  roots: List[String]
)

NativeContentModule(
  source: String,
  explicitRoot: Boolean,
  directImports: List[String],
  namespace: String,
  document: DocumentContent
)
```

`source` and every `directImports` entry use the same normalized target-
independent identity as the self-hosted module loader. `contentModules` contains
each loaded source once. `explicitRoot` preserves command-line root order;
non-root modules keep deterministic loader order. Helper imports
`std/content.ssc` and `std/ui/content.ssc` remain in the compilation closure but
are excluded when the runtime builds the public imported-content namespace
table.

`namespace` is the module's non-empty `name:` front-matter value, otherwise its
`.ssc` path stem. Namespace selection, direct-import ownership, and the
`DocumentContent` projection are computed inside the self-hosted tower. The
Scala seed must not reproduce those rules by scanning source text.

The third-field schema change is intentional and atomic: the staged tower and
the seed ship together. A seed receiving the obsolete root-only list fails with
an `invalid native frontend structural ABI` diagnostic; it must not reparse the
roots as a compatibility fallback.

### Self-hosted content projection

A pure module under `runtime/std/` converts the already parsed
`MarkdownDocument` and `YamlResult` values into the public types declared in
`runtime/std/content.ssc`:

```scalascript
DocumentContent
SectionContent
ContentBlock
ContentInline
ContentValue
```

It owns all text-to-structure work needed by that model:

- stable generated heading ids and explicit `{#id ...}` attributes;
- the nested section tree formed by heading levels;
- metadata-directive attachment to the following block;
- fence language/info attributes and structured YAML fence values;
- front-matter conversion to `ContentValue`;
- source-order preservation for blocks, sections, rows, lists, and imports.

The module may reuse the pure self-hosted YAML and Markdown ADTs. It contains no
`extern def`, host regex, or platform type. The native provider receives only
the finished structural values and therefore never parses content strings.

### Core-free runtime boundary

The v2 plugin SPI gains an immutable content view:

```scala
final case class NativeContentModule(
  source: String,
  explicitRoot: Boolean,
  directImports: List[String],
  namespace: String,
  document: ssc.Value)

final case class NativeRuntimeConfig(
  databases: Map[String, NativeDatabaseConfig] = Map.empty,
  contentModules: List[NativeContentModule] = Nil)

trait NativePluginContext:
  def contentModules: List[NativeContentModule]
```

`document` is a validated ordinary `ssc.Value` with the public content tags;
no parallel v1 AST or host-specific content hierarchy is introduced. Providers
receive an immutable list and cannot replace frontend products.

### Native content provider

`v2/runtime/std/content-plugin` owns the standard native implementations of:

```scalascript
contentDocument()
contentSection(id)
contentBlock(id)
contentData(id)
contentMetadata(path)
contentPlainText(value)
contentToMarkdown(value)
contentModules()
contentModule(namespace)
contentModuleSection(namespace, id)
contentModuleBlock(namespace, id)
contentModuleData(namespace, id)
contentModuleMetadata(namespace, path)
```

The provider registers public content constructor field names and returns
ordinary `ssc.Value` constructors, `Some`/`None`, lists, and maps. It has no
dependency on `core`, `frontendCore`, `backendInterpreter`, `pluginApi`, the v1
content plugin, or an external parser/renderer library.

For a single-root run, `contentDocument()` selects that explicit root and the
`contentModule*` family sees only its direct imports. For multiple explicit
roots, the first command-line root is the deterministic current document; each
root remains present in the immutable module list for future source-aware
execution work. Missing ids return `None`. An empty or duplicate imported
namespace produces the established bounded diagnostic rather than silently
choosing a document.

`contentPlainText` and `contentToMarkdown` are structural renderers. They may
inspect validated tags and strings but must not parse Markdown or data formats.
Markdown output is deterministic semantic Markdown; byte-for-byte source
whitespace preservation is not promised.

### Pure content binding

`contentBind(value, bindings)` is an ordinary target-independent definition in
`runtime/std/content.ssc`, not a provider intrinsic. `bindings` must be a
`ContentValue.MapV`. The function recursively rebuilds `DocumentContent`,
`SectionContent`, paragraphs, nested bullet/ordered-list items, and every table
header/body cell. Image, embedded, text, and code payloads remain unchanged;
emphasis, strong, and link labels recurse into their children.

An `Expr(source)` is substituted only when `source` is a non-empty dotted path
whose segments match `[A-Za-z_][A-Za-z0-9_]*`. Resolution starts at the root
`MapV` and every non-final segment must resolve to another `MapV`. Missing,
invalid, or non-map paths preserve the original `Expr` node. A resolved value
becomes `Text` using the established deterministic content rendering: strings
verbatim, lowercase booleans, normalized numbers, empty null, comma-separated
lists, and key-sorted map entries. The Scala provider neither validates paths
nor performs substitution and must not install an identity fallback.

## Behavior

- [x] The self-hosted tower emits one labelled content product for every module
      in the linked closure, with normalized direct-import edges and no host
      source scan.
- [x] Heading ids/tree, attrs, metadata attachment, manifests, and structured
      YAML fences are projected to the public content ADT by pure `.ssc` code.
- [x] The Scala 3 seed validates and retains complete structural content values;
      it does not reparse source or reduce documents to `blockCount`.
- [x] `NativePluginContext` exposes immutable content modules without importing
      v1 AST, Scalameta, interpreter, or plugin-api types.
- [x] The native provider returns the current document and recursively resolves
      present/missing section, block, data, and metadata lookups.
- [x] Imported-content lookup exposes only direct imports, uses manifest name or
      path-stem namespaces, filters helper modules, and rejects duplicates
      deterministically.
- [x] Plain-text and semantic-Markdown rendering cover every bounded
      self-hosted Markdown Profile block used by the public examples.
- [x] `examples/content-to-markdown.ssc` and the focused multi-file imported-
      content fixture produce byte-identical output on native VM, direct ASM,
      and `build-jvm` artifacts. The public
      `content-linked-namespaces.ssc` now resolves and prints its imported
      section; its later `BigInt.toString` failure is independently tracked as
      TI-8.2d2x and is not content-provider behavior.
- [x] The standard/slim/JRE distributions load no Scalameta,
      CommonMark/Flexmark, v1 content bridge, or external parser while running
      the focused examples.
- [x] A malformed structural content product fails before provider installation
      with a bounded source-identifying ABI diagnostic.
- [x] `contentBind` is a pure `.ssc` definition, recursively binds every
      supported inline-bearing block shape, preserves unresolved expressions,
      and is byte-identical on native VM/direct ASM and `build-jvm`.

## Out of scope

- `contentCurrentSection()` source-aware dynamic selection. Until runtime
  source identity is carried into calls, it reports a bounded unavailable
  diagnostic rather than guessing from the last executable fence.
- UI toolkit conversion (`contentView`, `contentToolkit*`) and framework/native
  view emission; those remain UI-provider work.
- Full CommonMark/GFM beyond the bounded ScalaScript Markdown Profile.
- Byte-exact reconstruction of original Markdown whitespace.
- The `md"..."` interpolator and the core `doc`/`render` helpers used by
  `examples/content.ssc`. `md` is a language lowerer feature and `doc`/`render`
  are general runtime helpers, not structural-content APIs; they require an
  independent non-overlapping slice.
- Removing the permanent Scala 3 seed.

## Design

1. Extend the self-hosted loader result so direct import identities are durable
   data rather than re-discovered by `NativeSourceClosure` or a provider.
2. Evaluate the pure content projection beside the existing self-hosted
   Markdown/YAML evaluation and return `NativeContentModule` values in
   `NativeCompilation/4`.
3. Decode by tag/arity in `NativeV2Structural`, retaining validated documents as
   `ssc.Value` in `NativeRuntimeConfig`.
4. Add the content list to `NativePluginContext` and propagate it through both
   installed execution and artifact metadata/runtime reconstruction.
5. Install a zero-external-dependency native provider and stage it in standard,
   slim, JRE, and deterministic `build-jvm` images.
6. Define runtime-dependent content binding in `runtime/std/content.ssc`, so
   path recognition, traversal, rebuilding, and value rendering execute as
   portable ScalaScript code after the provider returns immutable values.
7. Retire only taxonomy rows proven by exact assembled VM/ASM/artifact output.

## Decisions

- **Project content in ScalaScript before crossing the seed boundary** — chosen
  because heading attrs, ids, fence info, and YAML data are parsing semantics.
  Rejected: rebuilding `DocumentContent` in the Scala plugin (would create a
  second host parser and violate the self-hosted-core contract).
- **Carry labelled closure modules and direct edges structurally** — chosen so
  imported content never opens or reparses files at runtime. Rejected: retain
  root documents only and let the provider load imports (runtime I/O plus host
  parsing, and artifacts would not be self-contained).
- **Reuse ordinary `ssc.Value` content constructors** — chosen to keep one
  target-independent runtime value boundary. Rejected: import the v1
  `scalascript.ast.DocumentContent` hierarchy (it pulls the compatibility/core
  graph into the standard tier).
- **Keep content helpers in a dedicated provider** — chosen because document
  availability is execution context. Rejected: add content globals to the v2
  kernel (violates feature-plugin ownership).
- **Keep binding in the public `.ssc` module** — chosen because binding values
  exist only at user runtime while path interpretation is language semantics.
  Rejected: parse dotted paths or rebuild content in the Scala provider (a
  second host implementation), and rejected: evaluate bindings during frontend
  projection (runtime values do not exist yet).
