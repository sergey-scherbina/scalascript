# ScalaScript 2.1 negative toolchain release gates

Status: **complete** (2026-07-12)

## Overview

The final TI-9 gate proves that ScalaScript 2.1 stays compiler-independent when
the forbidden toolchain is not merely unused but physically unreachable. It
runs the standard frontend/checker, VM, direct ASM, representative native
providers, and a real HTTP server from a copied standard-only distribution with
no full CLI, Scalameta/v1 bridge, compiler tree, scala-cli/scalac/javac command,
or `java.compiler`/`jdk.compiler` module.

This closes the gap between the existing focused slim/JRE tests and the
exhaustive 195-row release corpus. The permanent prebuilt Scala 3 seed remains
inside the allowed standard graph.

## Interface

The report-producing gate is:

```text
tests/e2e/v21-negative-toolchain-release-gate.sh [--report FILE]
```

It creates a private install containing only:

```text
bin/ssc
bin/ssc-standard
bin/lib/standard/**
```

The gate derives the standard module set with `jdeps`, constructs a `java`
wrapper using `--limit-modules`, and constructs a PATH containing only the
non-compiler host utilities needed by the corpus scripts. It rejects any
resolvable `scala-cli`, `scalac`, or `javac` command and proves both compiler
modules are unresolvable.

`scripts/native-front-corpus --standard --ssc PATH` is the standard-only
frontend/checker mode. It loads `ssc1-front`/`ssc1-check` from the standard v2
core classpath and uses the supplied StandardMain launcher for runtime results;
it never requires `v2-frontend-bridge`, `ssc.bridge`, or `bin/lib/jars`.

The negative report contains at least:

```text
default.launcher=standard
tools.present=false
compiler.jars=0
scalameta.jars=0
scala-cli.available=false
scalac.available=false
javac.available=false
java.compiler.available=false
jdk.compiler.available=false
forbidden.references=0
frontend.ok=194
checker.ok=194
parity.mismatch=0
parity.one-sided=0
runtime.blockers=0
provider.smoke=pass
server.smoke=pass
release.ready=true
```

A separate freeze/checker validates those metrics. Its synthetic smoke must
reject launcher drift, a forbidden JAR/reference count, a visible compiler
command/module, frontend/checker regression, one-sided parity, blocker growth,
and missing/duplicate metrics.

## Design

The negative gate reuses the production artifacts and existing corpus/taxonomy
scripts rather than building a third launcher. Setup-time `jar`/`jdeps` may come
from the full JDK; user-program execution uses only the module-limited `java`
wrapper. A standard-mode frontend corpus is necessary because the migration
version of `native-front-corpus` used `v2-frontend-bridge` only for its runtime
column even though front/check themselves were self-hosted.

The HTTP fixture binds an ephemeral process-local port and executes once on VM
and once on direct ASM. This makes ServiceLoader/provider/server reachability a
release property, not only a unit/static-scan property.

## Decisions

- **One combined negative release environment** — every proof observes the same
  copied files, PATH, and module set. Rejected: treating independent slim and
  JRE tests as sufficient, because their assumptions could diverge.
- **Explicit standard mode for native-front-corpus** — preserves the detailed
  front/check/sentinel report without a compatibility runtime. Rejected:
  copying `v2-frontend-bridge` into the negative distribution.
- **Freeze the report and self-test the validator** — a green happy path alone
  cannot prove CI would notice drift. Rejected: undocumented shell assertions
  with no stable report contract.
- **Retain focused existing gates** — they remain useful fast diagnostics even
  after the combined gate is authoritative. Rejected: replacing them and losing
  failure locality.

## Out of scope

- Removing Scala 3 or the Scala runtime from the permanent seed.
- Removing tools/compatibility artifacts from the full distribution.
- Running compiler-backed JS/JVM/Rust/Swift/Spark backends in the negative
  environment.
- Expanding the frozen optional-provider/tools-backend runtime taxonomy.

## Results

**Update (2026-07-19, `ci-negtc-gate`).** The corpus has since grown to 209 examples (frontend
208 code + 1 non-code; parity 63 identical / 0 both-fail / 131 skips / 15 delegated). Two new
JDBC-facade examples — `scljet-hello.ssc` and `scljet-jdbc.ssc` — run correctly on the VM lane but
overflow the JVM per-class size limit on the direct-bytecode lane (`Class too large: ssc/gen/Entry`,
because the whole scljet SQLite engine plus the JDBC facade inline into one generated class). Since
there is no bytecode output to compare, `bc-parity-sweep` now classifies them with an explicit,
named `skipped-oversized-bytecode` allow-list (never a heuristic — any other oversized program still
surfaces as a real `bytecode-error`), and the negative-toolchain freeze baselines were advanced
accordingly. Tracked in BUGS.md `scljet-jdbc-facade-bytecode-class-too-large`; un-skip once the v2
bytecode backend splits the monolithic `ssc/gen/Entry` class.

Landed in `43fded0f9`. The copied distribution has only standard launchers and
40 standard JARs; compiler and Scalameta JAR counts are zero. Sanitized PATH
resolves none of scala-cli/scalac/javac, and module-limited Java cannot resolve
`java.compiler` or `jdk.compiler`. The derived set explicitly adds
`jdk.crypto.ec`, a reflective JCA provider edge discovered when the first
negative run made Ed25519 unavailable; exact crypto VM/ASM coverage now guards
it. Standard-only frontend/checker is 194/194 plus one non-code row, parity is
53 identical / 13 reviewed both-fail / 129 skips, mismatch and one-sided counts
are zero, runtime blockers are zero, and JSON/SQL/state/crypto/HTTP server
smokes pass. The freeze self-test rejects every declared drift class. CI and
the consolidated release gate run the environment; final conformance is 11/11.
