# ScalaScript 2.1 — native case-object values

## Overview

The self-hosted 2.1 frontend must retain a top-level `case object` as one
stable portable nullary constructor value. It currently recognizes only
`case class`; `case object NoContext extends ParserContext` is discarded, so
the standard parser-combinator runtime reaches `unbound global: NoContext`.

## Interface

```scalascript
trait ParserContext
case object NoContext extends ParserContext

def classify(value: Any): String = value match
  case NoContext => "none"
  case _         => "custom"
```

The inheritance clause is type-level metadata on the native type-erased route.
At runtime `NoContext` is a single `NoContext/0` data value. A named `.ssc`
import retains the declaration, and every bare reference selects that value.

## Behavior

- [x] `case object Name` parses as a declaration rather than a match arm or

- [x] An optional `extends`/`with` clause is consumed without swallowing the
      following top-level declaration.
- [x] Lowering emits one stable `IrDef(Name, IrCtor(Name, Nil))` value.
- [x] Bare references, aliases, structural equality, and nullary constructor
      patterns observe that same value.
- [x] Import-closure filtering preserves the declaration across a named
      multi-file `.ssc` import.
- [x] A focused multi-file fixture has identical exact VM/direct-ASM output.
- [x] Calculator and JSON parser examples leave `unbound global: NoContext`;
      any later independent boundary is tracked separately.
- [x] Native-entry, corpus/parity/taxonomy, release, and fresh affected
      conformance gates remain green.

## Out of scope

- Static inheritance/subtyping checks on the type-erased CoreIR route.
- Singleton types such as `NoContext.type`.
- Stateful or member-bearing `case object` bodies; ordinary object member
  ownership remains a separate existing feature.
- Fixing the separately tracked portable `Unit` and tuple-collection gaps.

## Design

The parser introduces one explicit `caseobj` statement node containing the
declared name. It uses the same statement-boundary scan as `case class` for
optional inheritance syntax, but has no parameter list. The native module
loader treats `caseobj` as a declaration. The lowerer maps it directly to a
nullary constructor value definition; existing `IrMatch` and structural
equality already implement the required runtime semantics on both VM and ASM.

No host intrinsic, reflection, compiler fallback, or provider global is needed.
