# ScalaScript 2.1 — toolchain independence

Status: **in progress** (2026-07-10)

This specification defines the release-wide ScalaScript 2.1 production-path
contract. It complements the feature-specific `v2.1.x` Distributed Streams
documents; those version labels remain valid and do not define the toolchain
packaging contract.

## 1. Purpose

The standard JVM path must execute and package ordinary `.ssc` programs without
Scalameta, Scala CLI, the Scala compiler, javac, or the `java.compiler` module:

```text
.ssc source
  -> native Markdown/code frontend
  -> native type checker
  -> CoreIR
  -> v2 VM or direct ASM JVM bytecode
  -> result / deterministic JAR
```

Scala remains the implementation language of the host components in 2.1. A
prebuilt distribution may therefore contain the Scala runtime and ASM. Building
ScalaScript itself from source still requires a Scala compiler. This milestone is
about the installed compiler/runtime path, not a source-bootstrap rewrite.

## 2. Dependency tiers

The distribution has two explicit tiers. A standard command must never discover
and silently enter the tools tier.

| Tier | Required contents | Forbidden dependencies | Intended commands |
|---|---|---|---|
| **standard** | staged launcher, native frontend tower, native checker/lowerer, CoreIR VM, direct ASM emitter, scalameta-free plugin runtime, selected runtime plugins, Scala runtime, ASM | Scalameta family, Scala CLI, `scala3-compiler`, compiler driver, `javax.tools`, `jdk.compiler`/`java.compiler` | default `ssc run`, `ssc run --bytecode`, `ssc build-jvm` |
| **tools / compatibility** | everything needed by an explicitly selected legacy or platform compiler route | none beyond the normal security/reproducibility policy | `--v1`, `--compat-frontend`, Scala/Java fences, legacy `compile-jvm`, Spark, Scala.js/WASM source compilation |

Normative rules:

1. `PATH` lookup for `scala-cli`, `scalac`, or `javac` is forbidden in the
   standard tier.
2. Loading a Scalameta or compiler class reflectively is equivalent to a direct
   dependency and is forbidden in the standard tier.
3. A tools-tier feature requested from a slim installation fails before partial
   execution with a diagnostic naming the missing tier and installation remedy.
4. Standard execution must remain functional after physically deleting all
   Scalameta-family JARs and the installed compiler directory.
5. The standard launcher's reachable module graph must not require
   `java.compiler` or `jdk.compiler`.

Physical staged layout during migration:

```text
bin/ssc-standard
bin/lib/standard/ssc.jar          # class-filtered native entry only
bin/lib/standard/jars/            # explicit standard dependency allowlist
bin/lib/standard/native-front/    # checker/lowerer tower + std sources

bin/ssc                            # compatibility launcher until TI-8
bin/ssc-tools                      # explicit full compatibility entry
bin/lib/ssc.jar + bin/lib/jars/    # compatibility runtime
bin/lib/compiler/                  # optional compiler/plugins
```

TI-7.1 landed this standard layout in `c43d23e59`. TI-7.2 `be229a70d` added the
explicit tools launcher to `installBin`, `install.sh`, and self-install. The
standard entry delegates only explicitly selected `run --v1` and
`--compat-frontend`; other tools-backed surfaces return the bounded tier/remedy
diagnostic. TI-7.3 `65773c2fe` made the separation a CI release gate: it copies
the staged distribution, deletes the compatibility JAR/plugin/compiler trees,
the full CLI JAR, and both compatibility launchers, then executes the complete
representative standard surface from the surviving files. The TI-8.1 audit
`e4cd55b36` then made every dependency JAR a static-scan root and removed H2's
optional `SourceCompiler*` family from only the standard-tier copy; the original
driver remains intact in tools. The default launcher remains on the
compatibility graph until the rest of TI-8 because TI-4 frontend parity is still
open; separating physical ownership is independently shippable, while changing
the default before parity would be a semantic regression.

## 3. Architecture and ownership

### 3.1 Standard path

The native frontend is the `mira-md -> ssc1-front -> ssc1-check -> ssc1-lower`
tower. The installed launcher runs the tower through a prebuilt kernel JAR and
staged `.ssc0`/runtime resources; it does not invoke Scala CLI over `v2/src`.

`ssc1-check` is **mandatory** before the native frontend becomes the default.
Dropping Scalameta while also bypassing type checking would replace a dependency
but regress the language contract. The checker may deliberately use `Dyn` at
documented plugin/FFI boundaries, but it must:

- accept every valid deterministic default-lane corpus program;
- reject the negative type-conformance corpus;
- produce source-located diagnostics;
- never emit runnable CoreIR containing the parser's `_err` sentinel.

Imports are resolved before type checking and lowering. Resolution is DFS,
canonical-path deduplicated, relative to the importing module first, and then to
the staged standard-library root. Multiple command-line source files participate
in one module closure. Program arguments after `--` are data, never source paths.

### 3.2 Compatibility frontend

The current v1-parser/Typer `FrontendBridge` remains an optional migration and
rollback component. Its AST and `BlockForm` adapters belong in the tools tier and
must not be referenced by standard runtime classes. It may be removed after one
release with a green native corpus and no unique compatibility cases.

### 3.3 Plugin runtime boundary

The standard v2 runtime owns a Scalameta-free value and intrinsic SPI. Runtime
values, callable values, effects, records, collections, errors, and native
contexts must be expressible without `scala.meta.Tree` or a v1 frontend AST.

Existing v1 `NativeImpl` plugins may be supported by an adapter in the tools tier.
That adapter is not allowed on the classpath of standard plugin tests. New plugin
work continues to live under `runtime/std/`; this specification changes the value
boundary, not the plugin ownership rule.

#### 3.3.1 Native provider interface

The standard JVM tier uses a dedicated ServiceLoader contract under
`ssc.plugin`, built without `core`, `backendInterpreter`, `pluginApi`, or any
Scalameta/compiler dependency:

```scala
trait NativePlugin:
  def id: String
  def install(context: NativePluginContext): Unit

trait NativePluginContext:
  def argv: List[String]
  def databases: Map[String, NativeDatabaseConfig]
  def invoke(fn: ssc.Value, args: List[ssc.Value]): ssc.Value
  def withEffect(effectTag: String)(handler: (String, List[ssc.Value]) => ssc.Value)(body: => ssc.Value): ssc.Value
  def register(name: String)(fn: List[ssc.Value] => ssc.Value): Unit
  def registerGlobal(name: String, arity: Int)(fn: List[ssc.Value] => ssc.Value): Unit
  def registerValue(name: String, value: ssc.Value): Unit
  def registerFields(tag: String, fields: Vector[String]): Unit
```

`ssc.Value` is the 2.1 standard runtime value boundary. Closures and opaque host
handles already have first-class `ClosV`/`ForeignV` representations, so a second
v1-shaped value hierarchy is not introduced. `NativePluginContext` is the only
registry mutation surface exposed to providers; providers do not import
`V2PluginRegistry` directly. Providers also invoke user callbacks only through
`context.invoke`; this keeps trampoline/closure-environment details inside the
standard host and gives server, streaming, SQL, and UI providers one callback
contract.

Normative loading rules:

1. Providers register through
   `META-INF/services/ssc.plugin.NativePlugin` and are installed in stable
   `id` order.
2. Duplicate intrinsic/global ownership is an error naming both providers;
   last-writer-wins registration is forbidden in the standard tier.
3. `ssc run --native` loads only this provider set. It must not call or link
   `ssc.bridge.PluginBridge`; `--compat-frontend` continues to load the v1
   `NativeImpl` adapter from the tools tier.
4. Feature providers live under `v2/runtime/std/<feature>-plugin/` and may
   depend on a standalone feature runtime (for example the JDBC executor), but
   not on v1 frontend/interpreter/plugin-api modules.
5. The initial migration order is crypto/fs/os/json, followed by HTTP, SQL, and
   UI. A feature switches only after its native provider has an assembled
   VM/ASM smoke and a classpath-reference gate.
6. Runtime globals such as `args`, `cwd`, `sep`, and `platform` are installed
   by a small standard host provider, not by the compatibility bridge.
7. `invoke` accepts a callable v2 runtime value and explicit arguments, checks
   arity through the normal runtime path, and returns the callback's final
   value. A non-callable value is a bounded provider error, not an unchecked
   cast or compatibility fallback.

#### 3.3.2 HTTP migration boundary

HTTP migrates in two independently shippable provider slices:

1. The standard client/value slice owns outbound `httpGet`/`httpPost`/
   `httpPut`/`httpPatch`/`httpDelete`, response builders, cache helpers, timeout,
   and retry state. It uses the JDK HTTP client, represents `Response` as the
   ordinary v2 data constructor `(status, headers, body)`, and reuses the native
   JSON codec for non-string `Response.json` bodies. A deterministic loopback
   unit test plus assembled VM/ASM response-builder smoke are mandatory.
2. The server slice owns `route`, `serve`/`serveAsync`, `stop`, middleware,
   streaming responses, SSE, uploads, and WebSockets. These operations require
   an explicit native server-host SPI for route registration, callback
   invocation, and lifecycle. Any operation whose host hook has not landed must
   fail with a bounded "native HTTP server unavailable" diagnostic; calling the
   v1 interpreter/server bridge or silently treating registration as success is
   forbidden.

The client/value slice may land before the server slice. It does not by itself
complete HTTP migration or TI-5.

#### 3.3.3 SQL migration boundary

The standard launcher parses `databases:` declarations from every explicit
root source before provider installation and exposes immutable
`NativeDatabaseConfig` values through `NativePluginContext.databases`. Parsing
uses the standalone `SimpleYaml` module, not the v1 AST/manifest parser. An
entry preserves `url`, optional `user`, `password`, and `driver` strings so
`${env:...}` and other JDBC-runtime substitutions remain lazy. Identical
duplicate names across roots are accepted; conflicting definitions fail before
opening a connection or executing user code.

The native SQL provider lives under `v2/runtime/std/sql-plugin/` and may depend
on the standalone `backendSqlRuntime`. It converts the native configs into one
lazy `ConnectionRegistry` and owns the standard `DriverManager.getConnection`,
`Db.query`, and `Db.execute` surfaces. Queries return ordinary v2 lists of maps
with JDBC column labels as keys; update/DDL statements return `Int`. Bind values
are converted explicitly between `ssc.Value` and the executor rather than
passing v1 `PluginValue` or interpreter values through `ForeignV`.

The first independently shippable SQL slice covers named H2 front-matter,
parameter binding, DDL/DML, and row reads on both native VM and direct ASM. It
must include malformed/conflicting-config tests and the standard static/runtime
classpath gates. Typed `Db.insert`/`Db.update`, PostgreSQL LISTEN/NOTIFY, and
native lowering of fenced `sql`/`transaction` blocks are follow-up SQL slices;
they may not silently enter `PluginBridge` while pending.

#### 3.3.4 UI migration boundary

The standard UI provider must not reuse `frontendCore` runtime values because
that module currently depends on the Scalameta-coupled v1 `core`. The native
provider instead represents signals, event descriptors, and view nodes directly
on `ssc.Value`/`ForeignV` with named-method access. Mutable and derived signals
must expose the normal callable/get/set/update behavior, and user callbacks run
only through `NativePluginContext.invoke`.

The first UI slice owns `signal`, `computedSignal`, `eqSignal`, `textNode`,
`signalText`, `showSignal`, `fragment`, `element`, basic signal event builders,
and `emit`. `emit` performs deterministic UTF-8 static rendering to
`<outDir>/index.html`, escapes text and attributes, resolves current signal
values, and never invokes a frontend compiler. Unit tests plus assembled VM/ASM
must compare the emitted bytes, not merely accept a `ForeignV` or `Stub` result.

The standard JVM/static lane also constructs core-free declarative
`fetchUrlSignal` and `fetchAction` values so imported JSON/fetch helpers can be
checked and evaluated consistently. A fetch signal reads as empty until a
target renderer executes its descriptor; network execution remains the emitted
browser runtime's responsibility. Framework-specific SPA generation,
`serve(view, port)`, keyed reconciliation, data-table actions,
storage/WebAuthn, and desktop/mobile renderers remain follow-up UI slices. Until
their native host contracts land, they must fail with bounded native diagnostics
or unresolved-provider errors; loading the v1 frontend plugin/`frontendCore` as
a fallback is forbidden.

#### 3.3.5 Effect migration boundary

The v2 kernel already owns a core-free dynamically scoped effect-handler stack.
Providers access it only through `NativePluginContext.withEffect`: the host
pushes the named handler, evaluates the by-name body, and pops it in `finally`.
Providers never manipulate `V2EffectContext`, interpreter `Computation`, v1
`BlockForm`, `EffectHandler`, or `SpiValue` directly. User thunks and operation
callbacks continue to run through `context.invoke`.

The representative first provider owns `State` plus the curried
`runState(initial) { body }` runner. `State.get`, `State.set`, and
`State.modify` use the innermost active state; `modify` invokes its callback
through the host; the runner returns `(finalState, bodyResult)`. Nested runners
must restore the outer handler, and thrown callbacks must not leak a handler
into the next run. Unit tests and assembled VM/ASM cover both mutation and
nested restoration.

Logger formatting/collection, Random, Clock, Env, Retry, Cache, Async, and
stream handlers are follow-up provider slices. Until migrated, their native
runner globals may not be populated by the v1 block-form bridge.

### 3.4 VM and ASM backends

Both standard backends consume the same checked CoreIR:

- the VM compiles CoreIR to closures and executes it;
- `v2JvmBytecode` emits JVM bytecode directly with ASM.

The old v1 `AsmJitBackend` is not the 2.1 artifact backend: it consumes the
Scalameta-coupled v1 runtime graph. The v1 `JavacJitBackend` and that legacy ASM
JIT remain tools-tier compatibility implementations only.

## 4. CLI and artifact contract

### 4.1 Execution

During migration:

```text
ssc run --native file.ssc [more.ssc ...] -- [program args ...]
ssc run --native --bytecode file.ssc [more.ssc ...] -- [program args ...]
ssc run --compat-frontend file.ssc    # tools tier: Scalameta bridge
ssc run --v1 file.ssc                 # tools tier: complete v1 rollback
```

At cutover:

- plain `ssc run` is identical to `ssc run --native` and uses the v2 VM;
- `--bytecode` selects direct CoreIR-to-ASM execution without changing frontend
  or program semantics;
- `--native` remains an accepted, idempotent assertion for CI and deployment;
- `--v2` remains a deprecated alias for the default v2 runtime and no longer
  implies the Scalameta bridge;
- the old bridge is selected only by `--compat-frontend`.

The selected route is observable with `ssc info --execution-plan ...` (or an
equivalent stable machine-readable command). Tests must not infer the route from
timing or classpath accidents.

### 4.2 JVM artifacts

The new direct-ASM artifact command is intentionally distinct from the existing
Scala-source-backed `compile-jvm [--bytecode]` path:

```text
ssc build-jvm file.ssc [more.ssc ...] -o app.jar
```

`build-jvm` must:

- run the native frontend and native checker;
- emit `.class` files directly through `v2JvmBytecode`;
- include a deterministic manifest/main entry, required v2 runtime metadata,
  linked module definitions, and selected standard plugins;
- preserve `.ssc` source/line mapping in JVM diagnostics;
- produce byte-identical JARs for identical inputs, options, and dependency set;
- run through `java -jar app.jar -- ...` without Scala CLI, scalac, or javac.

The artifact is a deterministic, self-contained JAR rather than a launcher
that searches the installation at runtime:

1. `ssc/gen/Entry.class` is emitted directly by `v2JvmBytecode` and owns the
   JVM `main(String[])` method. The method initializes the standard artifact
   runtime, installs compiled definitions, evaluates the entry, and applies the
   same unresolved-effect/result validation as `ssc run --native --bytecode`.
2. Runtime classes are merged from an explicit standard-tier allowlist: v2
   core, native plugin SPI/providers, Scala runtime, and provider-specific leaf
   dependencies. The merge rejects conflicting duplicate entries, combines
   `META-INF/services/ssc.plugin.NativePlugin` deterministically, and excludes
   signatures, v1 frontend/interpreter/bridge, Scalameta, and compiler families.
3. `META-INF/scalascript/artifact.properties` records the artifact format,
   sorted source identities and SHA-256 digests, selected providers, and native
   database configuration. It contains no timestamps or absolute paths.
4. Every ZIP entry uses fixed metadata and lexical ordering. Manifest bytes,
   provider order, dependency order, and stored metadata are independent of the
   checkout path and wall clock.
5. Imported modules and multiple explicit roots are lowered into one linked
   `Program`; duplicate/conflicting module definitions follow the same native
   frontend rules as direct execution.
6. Generated methods carry `SourceFile`, line tables, and a
   `SourceDebugExtension` SMAP whose file table contains every explicit root.
   At minimum, entry statements and named definitions map to their originating
   `.ssc` line; generated helper frames may map to the owning statement/def.

TI-6 is delivered in independently green slices:

  the writer merges a strict runtime/provider allowlist, combines ServiceLoader
  descriptors, rejects conflicting entries, writes stored entries in lexical
  order at the ZIP epoch, and runs the native checker before ASM. Two staged
  argv+crypto builds were byte-identical (26,291,502 bytes; SHA-256
  `95590553b0174f3026a947fbb48f000a3cf878cf4e61c114d928c86a33b2d746`),
  executed with `java -jar` while compiler commands were absent, and passed
  entry/reference plus `jdeps` forbidden-dependency gates;
- [x] import/multi-root linking, embedded runtime configuration, and duplicate
  resource/definition diagnostics (2026-07-10, `147531fa7`). The native module
  closure is packaged as one checked `Program`; source hashes include explicit

  It writes a stable TSV baseline for release review.

Legacy `compile-jvm`, `.scjvm`, Scala facades, and Scala-source passthrough remain
tools-tier compatibility surfaces until separately migrated. The word
`--bytecode` on those commands does not make them part of the native ASM path.

## 5. Compatibility policy

Every corpus row is classified as exactly one of:

1. **standard deterministic** — VM and ASM observable output must match;
2. **standard server/long-running** — readiness and bounded interaction are
   tested instead of process completion;
3. **standard nondeterministic** — normalized semantic assertions replace byte
   comparison;
4. **backend-specific tools tier** — Scala/Java fences, Spark, Scala.js/WASM, or
   another explicitly compiler-backed target;
5. **document/non-code** — no executable ScalaScript block;
6. **temporarily unsupported compatibility case** — named in the baseline with
   a stable diagnostic and owner; no silent `_err`, empty output, or success.

An increase in category 4–6 requires a spec change. A parser/lowerer exit code of
zero is not success when the emitted IR contains `_err` or unresolved parser
sentinels.

## 6. Portable gates

All gates resolve the repository from their own script path or Git worktree;
absolute developer paths are forbidden. Each gate writes a stable JSON or TSV
summary in addition to the human report.

Required gates:

- **native-front corpus:** extract/import/parse/check/lower/run classification for
  every `examples/*.ssc` input, including sentinel detection;
- **VM/ASM parity:** checked CoreIR executed by both standard backends, with the
  classifications in §5;
- **artifact reproducibility:** two clean `build-jvm` runs compared byte-for-byte
  and executed with `java -jar`;
- **slim-filesystem negative gate:** delete compiler and Scalameta JARs, hide
  Scala CLI/scalac/javac from `PATH`, then run hello/import/argv/plugin/JAR smokes;
- **JRE-module negative gate:** run the standard launcher/runtime with
  `java.compiler` and `jdk.compiler` unavailable and inspect it with `jdeps`;
- **classpath gate:** fail when a standard-tier JAR contains or references
  `scala.meta`, v1 frontend AST, compiler-driver, or `javax.tools` symbols.

  `ssc run --native` handles hello, normalized relative/std imports, multiple
  sources, argv, explicit sentinel diagnostics, VM/direct ASM, and plugin
  intrinsics with Scala CLI absent.
- [ ] **TI-4 Frontend parity:** all corpus rows are classified, standard rows
  parse/check/lower without sentinels, and the mandatory checker passes both
  positive and negative suites.
  - [x] **Numeric/string checker inference (2026-07-10, `66b7c4ede`):** infer
    Float literals and mixed numeric arithmetic, String repetition, and
    substitution-aware String concatenation while retaining negative numeric,
    repetition-count, boolean, and condition checks. Direct corpus result:
    189 OK / 5 `TYPEERR` / 1 non-code after the companion prefix fix below;
    every remaining reject is a frontend parser-sentinel row.
  - [x] **Prefix/postfix precedence (2026-07-10, `6e8464ea8`):** postfix
    selection/application now binds inside unary `!`, `-`, and `~`. The
    sentinel-clear `!exists(path)` checker rejection is gone, and assembled
    native VM/direct-ASM tests cover all three operators on call operands.
  - [x] **Ordered immutable initialization (2026-07-10, `5db137a20`):** physical
    newlines now infer statement separators at top level and directly inside
    explicit braces when both adjacent tokens can terminate/start statements.
    Top-level immutable values and tuple bindings use empty global cells whose

    `val rows = Db.query(...)` after H2 DDL/DML and a nested `val inside =

    `17/20/2/101/101/2` output. The full native-front corpus completes 195 rows
    without crash/timeout (192 front success, 190 checker success), and the
    `v2-*` conformance slice remains 8/8.
  - [x] **Unresolved-result parity (2026-07-10, `7192cd6e4`):** shared VM/ASM
    result handling rejects top-level missing-dispatch `Stub` values and dotted
    unhandled runtime-effect `Op` values. The historical
    `Wallets.metaMask` bytecode lane now exits nonzero instead of printing the

## 9. Decisions and rejected alternatives

- **Keep Scala as implementation/runtime language in 2.1.** Removing
  `scala-library` is independent of removing runtime compiler dependencies and
  would obscure the critical path.
- **Require the native checker.** Parser-only replacement was rejected because it
  would silently weaken the typed-language contract.
- **Use the v2 CoreIR ASM emitter.** Promoting the old v1 `AsmJitBackend` was
  rejected because its input/runtime graph remains coupled to v1 frontend values.
- **Add `build-jvm`.** Reusing `compile-jvm --bytecode` unchanged was rejected
  because that command currently means generated Scala plus compiler tooling.
- **Split the distribution.** Deleting every compiler-backed feature in one
  release was rejected; explicit tools-tier Scala fences, Spark, Scala.js/WASM,
  facades, and rollback remain useful and honest.
- **Use `ssc.Value` as the native provider value boundary.** Reusing the v1
  `PluginValue` facade was rejected because its constructors/extractors are
  backed by `scalascript.interpreter.Value` and therefore pull `core` plus
  Scalameta into every provider. Inventing another closed value ADT was rejected
  because the v2 runtime already models closures, effects, records,
  collections, and opaque handles required by native providers.
- **No transparent fallback.** Falling back to Scalameta or javac after a native
  failure was rejected because it makes production dependency guarantees
  untestable and deployment-specific.

## 10. Non-goals

- compiling the ScalaScript repository from source without scalac;
- removing the Scala runtime from produced applications;
- rewriting Spark, Scala.js/WASM, or arbitrary Scala/Java fences to ASM in this
  milestone;
- removing the compatibility tier before the native path has a release of field
  evidence;
- changing target-independent `.ssc` semantics to accommodate an ASM limitation.
