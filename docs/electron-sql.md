# Electron SQL

> Status: current behavior note - May 2026.

This document describes what happens when a `.ssc` app that uses `databases:`,
`sql` blocks, `Db.query`, or `Db.execute` is built as an Electron desktop app
with:

```bash
ssc build --target desktop app.ssc
ssc run --frontend electron app.ssc
```

Electron uses a Chromium renderer loaded from `file://.../index.html`. That
renderer is browser-like: it has DOM APIs and `fetch`, but it does not expose
Node's `fs` module because generated windows run with `nodeIntegration: false`
and `contextIsolation: true`.

## Runtime Model

The Electron bundle is generated by `ElectronBundleBuilder`:

1. `index.html`, `main.js`, `preload.js`, and `package.json` come from
   `ElectronEmitter`.
2. `app.js` is generated by `JsGen` plus the browser/Electron SPA patch.
3. The SPA patch routes same-app `fetch("/api/...")` calls through registered
   `.ssc` routes instead of issuing network requests from a `file://` page.
4. SQL support comes from the shared `backend-sql-runtime-js`
   `sql-runtime.mjs` source, embedded into `app.js`.

The important consequence: a full-stack `.ssc` app can define REST routes and a
reactive UI in one file, and `fetchAction` / `fetchTable` can call those routes
inside the Electron renderer.

## `sqlite:` In Electron

On Node, `sqlite:<path>` means sql.js reads and writes the file through
`fs/promises`.

In an Electron renderer, that filesystem path is not available. The current
desktop behavior is a pragmatic browser fallback:

| URL | Electron renderer behavior |
| --- | --- |
| `sqlite::memory:` | In-memory database for the renderer session. |
| `sqlite:` | In-memory database for the renderer session. |
| `sqlite:<path>` | Browser fallback persisted in `localStorage` under a key derived from the URL. It is not a real file at `<path>`. |

This is enough for examples such as
[`examples/frontend/toolkit-demo/toolkit-demo.ssc`](../examples/frontend/toolkit-demo/toolkit-demo.ssc),
where the app needs a small local todo store behind REST routes.

## Supported SQL Subset In The Fallback

The fallback is intentionally small. It is not a general SQLite
implementation. Today it supports the statements needed by the toolkit demo:

| Shape | Supported |
| --- | --- |
| `CREATE TABLE [IF NOT EXISTS] <table> (...)` | Yes. Creates an in-memory/localStorage table record. |
| `INSERT INTO <table>(col, ...) VALUES (?, ...)` | Yes. Stores one row; `todos` gets an auto `id` when absent. |
| `SELECT col, ... FROM <table> [ORDER BY col]` | Yes. Returns rows by requested columns. |
| `DELETE FROM <table> WHERE col = ?` | Yes. Deletes matching rows. |

Everything else raises a `SqlRuntimeError` that names the unsupported SQL.
Browser/Electron async init failures are rendered into `document.body` instead
of producing a silent blank window.

## What This Is Not

This fallback is not:

- a full SQLite parser or query engine;
- a replacement for `sql.js`;
- a durable app-data file in Electron's `app.getPath("userData")`;
- shared with the JVM interpreter's `jdbc:sqlite:` database;
- shared with Node's file-backed `sqlite:<path>` behavior.

Electron bundles with `databases:` now use the Electron main/preload bridge
tracked in [`electron-persistence-bridge.md`](electron-persistence-bridge.md).
The bridge executes SQLite in the main process through `sql.js` and persists
file-backed databases under `app.getPath("userData")`. The fallback only exists
when the Electron bridge is absent, keeping browser-like renderer demos
functional and honest.

## Tested Path

`ToolkitElectronSmokeTest` covers the current production path:

1. builds the toolkit demo Electron bundle through `ElectronBundleBuilder`;
2. starts real Electron when `electron` is available on `PATH`;
3. verifies the initial window renders;
4. clicks the generated `Add` button;
5. checks that the todo appears in both the DOM and `fetch("/api/todos")`.

When Electron is not installed, the smoke test is cancelled rather than failed.

## Future Direction

There are three plausible ways to make Electron SQL more complete:

1. **Bundle sql.js assets.** Keep renderer-only execution, but make `import`
   resolution and wasm loading explicit in the generated bundle.
2. **Use an Electron main/preload bridge.** Keep filesystem access in the main
   process, expose a narrow `Db.query` / `Db.execute` IPC surface to the
   renderer, and persist under `app.getPath("userData")`. This is the preferred
   path; see [`electron-persistence-bridge.md`](electron-persistence-bridge.md).
3. **Promote the fallback.** Keep localStorage-backed SQL as an explicit,
   documented "demo/local browser store" with a broader but still bounded SQL
   subset.

Until one of those lands, treat Electron `sqlite:<path>` as localStorage-backed
demo persistence, not as real file SQLite.
