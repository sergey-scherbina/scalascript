# v2.0 Artifact Format — Wire Specification

This document is the authoritative wire-format spec for the seven on-disk
artifact files produced by the ScalaScript v2.0 compiler.  External tools
(IDEs, alternative linkers, dependency-graph viewers, content-addressable
caches) read this format directly; the rules below define what they can
and cannot rely on across compiler versions.

The companion test suite is
[`core/src/test/scala/scalascript/artifact/ArtifactAbiCompatibilityTest.scala`](../core/src/test/scala/scalascript/artifact/ArtifactAbiCompatibilityTest.scala) —
every claim here is locked down by an executable test there.

---

## 1. Envelope

Every artifact is a single pretty-printed (2-space-indent) JSON document.
The two outermost envelope fields are:

| Field        | Type     | Value                                       |
| ------------ | -------- | ------------------------------------------- |
| `magic`      | `String` | `"SSCART"` — the literal 6-character magic. |
| `abiVersion` | `String` | `"2.0"` — current ABI version.              |

Constants live in [`ir/src/main/scala/scalascript/ir/Ir.scala`](../ir/src/main/scala/scalascript/ir/Ir.scala)
lines 204-211 (`object ArtifactVersion`).

### Read-time validation

Readers MUST check the envelope BEFORE deserialising the payload.
Implementations:

- [`core/src/main/scala/scalascript/artifact/ArtifactIO.scala`](../core/src/main/scala/scalascript/artifact/ArtifactIO.scala) lines 111-124 (`checkEnvelope`)
- [`core/src/main/scala/scalascript/artifact/JvmArtifactIO.scala`](../core/src/main/scala/scalascript/artifact/JvmArtifactIO.scala) lines 150-163
- [`core/src/main/scala/scalascript/artifact/JsArtifactIO.scala`](../core/src/main/scala/scalascript/artifact/JsArtifactIO.scala) lines 147-160

### ABI version semantics

The version string is `major.minor`.  **Current policy: any mismatch is
fatal at read time.**  The check is `!=` (exact equality), not a
major-match.  This is intentional during v2.0 hardening — the matrix of
mixed minor versions is too large to verify by inspection, so any drift
forces a recompile.  See § 6 "Deprecation policy" for how the policy
would loosen if dual-support is ever needed.

### Envelope is shared across formats

All seven artifact formats use the same `SSCART` magic and the same
`2.0` version.  The envelope alone does **not** disambiguate them; the
consumer MUST select its reader based on the file extension:

| Extension          | Reader                          |
| ------------------ | ------------------------------- |
| `.scim`            | `ArtifactIO.readInterface`      |
| `.scir`            | `ArtifactIO.readIr`             |
| `.scjvm`           | `JvmArtifactIO.readJvm`         |
| `.scjs`            | `JsArtifactIO.readJs`           |
| `.scjvm-runtime`   | `JvmArtifactIO.readRuntime`     |
| `.scjs-runtime`    | `JsArtifactIO.readRuntime`      |

Feeding a `.scjvm` payload to `readInterface` parses the envelope
successfully but then fails on the payload schema mismatch — the test
suite pins this behaviour so consumers can't rely on a friendlier error.

## 2. Per-format payload schemas

All payload case classes live in `ir/src/main/scala/scalascript/ir/Ir.scala`.
Source line ranges below cite that file.

### 2.1 `.scim` — module interface

`ModuleInterface` (lines 268-280) — what a downstream module needs to
type-check against this one, without re-parsing source.

| Field                    | Type                          | Default     | Required key? |
| ------------------------ | ----------------------------- | ----------- | ------------- |
| `magic`                  | `String`                      | —           | yes           |
| `abiVersion`             | `String`                      | —           | yes           |
| `pkg`                    | `List[String]`                | —           | yes           |
| `moduleName`             | `Option[String]`              | (none)      | yes (may be `null`) |
| `moduleVersion`          | `Option[String]`              | (none)      | yes (may be `null`) |
| `sourceHash`             | `String` (SHA-256 hex)        | —           | yes           |
| `exports`                | `List[ExportedSymbol]`        | —           | yes           |
| `instances`              | `List[InstanceDecl]`          | `Nil`       | no            |
| `capabilities`           | `List[CapabilityDecl]`        | `Nil`       | no            |
| `externDefs`             | `List[ExportedSymbol]`        | `Nil`       | no            |
| `dependencies`           | `Map[String, String]`         | `Map.empty` | no            |
| `sectionHashes`          | `Map[String, String]`         | `Map.empty` | no            |
| `sectionOwnHashes`       | `Map[String, String]`         | `Map.empty` | no            |
| `sectionInterfaceHashes` | `Map[String, String]`         | `Map.empty` | no            |
| `scalaFacade`            | `Map[String, String]`         | `Map.empty` | no            |
| `apiDescriptorV3`        | `Option[String]`              | `None`      | no            |

`apiDescriptorV3` is an opaque additive carrier for the canonical UTF-8 JSON
text defined by [`ssc-api-descriptor-v3.md`](ssc-api-descriptor-v3.md). The
legacy `.scim` layer does not derive, parse, validate, or reinterpret descriptor
semantics. Pre-v3 JSON and MessagePack payloads decode with
`apiDescriptorV3 = None`; a present value round-trips unchanged, and readers
that ignore unknown additive fields retain their previous behavior. The field
is declared last, after `scalaFacade`, and does not change
`ArtifactVersion.current`, which remains `2.0`.

Slice A adds only the carrier and descriptor library. Compiler producers,
linker population, admission, facades, and runners are deferred to later slices.

`ExportedSymbol` (lines 219-241) carries `name`, `fqn`, `kind`, `tpe`,
`span`, and (optional) `nested: List[ExportedSymbol] = Nil`.  The
`nested` field was added in v2.0 Stage 5.6; the default keeps older
`.scim` files readable.

`InstanceDecl` (lines 246-251): `typeclass`, `typeParam`, `witnessName`,
`fqn`.  All required.

`CapabilityDecl` (lines 256-258): `name`.  Required.

### 2.2 `.scir` — module IR body

`ModuleIrArtifact` (lines 290-297) — envelope + a serialised
`NormalizedModule` body as a JSON string.

| Field        | Type             | Default | Required key? |
| ------------ | ---------------- | ------- | ------------- |
| `magic`      | `String`         | —       | yes           |
| `abiVersion` | `String`         | —       | yes           |
| `pkg`        | `List[String]`   | —       | yes           |
| `moduleName` | `Option[String]` | (none)  | yes (may be `null`) |
| `sourceHash` | `String`         | —       | yes           |
| `body`       | `String` (JSON)  | —       | yes           |

The `body` is `upickle.default.write(NormalizedModule)` — a JSON string
embedded inside the outer JSON, so the envelope can be validated
without parsing the (often large) body.

### 2.3 `.scjvm` — JVM cached source

`ModuleJvmArtifact` (lines 314-354).

| Field          | Type                | Default | Required key? |
| -------------- | ------------------- | ------- | ------------- |
| `magic`        | `String`            | —       | yes           |
| `abiVersion`   | `String`            | —       | yes           |
| `moduleId`     | `String`            | —       | yes           |
| `pkg`          | `List[String]`      | —       | yes           |
| `moduleName`   | `Option[String]`    | (none)  | yes (may be `null`) |
| `sourceHash`   | `String`            | —       | yes           |
| `scalaSource`  | `String`            | —       | yes           |
| `imports`      | `List[String]`      | —       | yes           |
| `classBundle`  | `Option[String]`    | `None`  | no            |
| `capabilities` | `List[String]`      | `Nil`   | no            |

`classBundle` is base64-encoded ZIP of `.class` + `.tasty` files.
Empty `capabilities` AND non-empty `classBundle` indicates a
pre-split-runtime artifact whose `classBundle` ships its own runtime
preamble.

## 3. Backward-compatibility rules

The rules below define which schema edits are safe (no ABI bump) and
which are breaking (require an ABI bump and a coordinated upgrade).
The test suite enforces all of these.

### 3.1 Additive — safe (no ABI bump)

You may, without bumping `ArtifactVersion.current`:

- **Add a new field with an explicit `= None` / `= Nil` / `= Map.empty`
  default.**  Older artifacts that don't carry the field still parse
  (upickle uses the default); newer artifacts include it.  This is the
  property locked down by the `optionalFields` test family and by the
  programmatic enumeration in `ArtifactAbiCompatibilityTest`.

- **Read an artifact carrying unknown extra fields.**  Readers silently
  ignore them — this is the forward-compat property the same test
  suite pins down for every format.

### 3.2 Breaking — require an ABI version bump

Each of these alters the read-time semantics and MUST be paired with a
bump of `ArtifactVersion.current`:

- **Renaming a field** — old artifacts have the old key, new readers
  expect the new key.  Even if the type is unchanged.

- **Changing a field's type** — JSON number vs. string vs. array
  produces a parse error on the other side.

- **Removing a field** — even if the field had a default.  External
  tools may still write it, and an unknown-field policy that ever
  flips strict will break them.

- **Adding a field WITHOUT a Scala-level default** — upickle's derived
  `ReadWriter` treats such a field as required at the JSON-key level.
  Older artifacts that lack the field will fail to parse.

  **Forensic note.** `ModuleInterface.moduleName: Option[String]`
  (and similar `Option[String]` fields) are declared WITHOUT an
  `= None` default.  These are NOT absent-tolerant — the JSON key
  must be present (as `null` for `None`).  The test suite splits
  optional fields into `optionalFields` (truly absent-tolerant) and
  `requiredOptionTypedFields` (key required, value may be `null`)
  to keep this distinction honest.  If a future change adds an
  `= None` default to one of these fields, move the entry to
  `optionalFields` in the test and re-state in the table above.

- **Changing the meaning of an existing field** — e.g. switching
  `sourceHash` from SHA-256 to BLAKE3.  Readers can't tell the
  difference from the byte stream alone.

- **Changing the envelope** — magic, version-string format, or the
  top-level pretty-print convention.

### 3.3 Soft-breaking — review case-by-case

- **Adding a NEW REQUIRED variant to a sealed-trait-with-`ReadWriter`**
  (e.g. a new `IrExpr` subtype).  Old readers will fail to discriminate
  the new variant.  In practice this happens often inside
  `NormalizedModule` bodies; the envelope ABI bump is the safety net.

- **Tightening a value-domain check inside a field** (e.g. requiring
  `kind` ∈ a smaller set).  Old artifacts may carry now-invalid values.

## 4. Round-trip guarantees

- **Byte-stable round-trip.** For every artifact, `write -> read ->
  write` yields a byte-identical JSON string.  This is the property
  external content-addressable caches rely on for hash stability.

- **`sourceHash` is preserved verbatim.**  Readers never recompute or
  normalise the hash on read; whatever was written comes back exactly.

- **Field order is fixed by the case-class declaration order.**
  upickle's derived `ReadWriter` emits fields in declaration order.
  Reordering case-class fields is therefore a breaking change for
  byte-stability (though not for value-level equality).

---

## 5. Cross-format ambiguity

Because every format shares `"SSCART"` + `"2.0"` envelope, a payload
read with the wrong reader will:

1. Pass envelope validation (magic + version OK).
2. Fail at payload deserialisation (schema mismatch on required
   payload-specific fields).

Consumers MUST select the reader by file extension.  Inspecting only
the magic/version is insufficient.  Three explicit tests in the suite
cover this (`.scjvm` -> `readInterface`, `.scim` -> `readJvm`,
`.scjs-runtime` -> `readJs`).

---

## 6. Deprecation policy

When a breaking change becomes necessary, the policy is:

1. **Bump `ArtifactVersion.current`.**  `2.0` -> `2.1` for an
   additive-but-incompatible change (e.g. a new required field);
   `2.0` -> `3.0` for any rename, removal, or semantic shift.

2. **One ABI version cycle of dual support if any external consumer
   needs to migrate.**  The reader code path can branch on the
   incoming `abiVersion` string for one release, accepting both N and
   N-1 and emitting only N.  This is opt-in and only worth it when a
   third-party tool ecosystem exists; for purely internal artifacts,
   the current strict-equality policy is simpler and safer.

3. **Update this document and the test matrix in the same commit as
   the constant bump.**  Specifically:
   - Update the `optionalFields` / `requiredOptionTypedFields` maps in
     `ArtifactAbiCompatibilityTest` to reflect the new schema.
   - Update the field tables in § 2.
   - Update the constants test
     `ArtifactVersion constants are the documented v2.0 values`
     (or add a new one for `2.1` / `3.0`).

4. **Add a regression test for the new behaviour.**  At minimum: a
   round-trip test on a canonical example of the new schema, and (if
   dual support is in play) a "read N-1 written by old compiler" test
   using a checked-in fixture.

There is currently no dual-support shim; the first ABI bump that needs
one should land it alongside the bump.

## 6.5 Phase 4 additive fields (in flight)

The following additive (default-defaulted) fields are being added in
Phase 4 of v2.0.  All are backward-compatible — old artifacts read with
the default value; new artifacts populate the real value.  No ABI bump.

- **`ExportedSymbol.definitionLine: Int = 0`** and
  **`ExportedSymbol.definitionColumn: Int = 0`** — source position of
  the symbol's defining `def`/`val`/`class`.  Used by LSP for cross-
  module go-to-definition.  Old artifacts surface as `(0, 0)`.
- **`Content.CodeBlock.lineOffset: Int = 0`** — line number where the
  fenced code block starts in the `.ssc` file.  Not on-disk in any
  artifact (it's in-memory in the parsed `Module` only), but mentioned
  here for completeness — LSP uses it to translate block-local scalameta
  positions to file-level positions.
- **`ModuleJvmArtifact.lineMap: Map[String, Int] = Map.empty`** —
  generated-Scala-line → original-`.ssc`-line.  Powers the `--source-map`
  JVM Option A (JSR-45 SMAP injection via ASM).  Old artifacts surface
  as `Map.empty`; the SMAP-injection step is then a no-op (i.e., stack
  traces resolve to the synthetic Scala source, the existing Option B
  behaviour).

All five fields are tested in the Phase 4 follow-up suites — see
`MILESTONES.md` "Phase 4 / honesty-pass follow-ups".

## 8. `.sscc` — Compiled Script Cache (CBOR v2)

The `.sscc` file is the **compiled script cache** written by `ssc compile` /
`ssc build` for `.ssc` source modules. It is a standalone binary format
distinct from the seven JSON IR artifacts above. Consumers MUST NOT use
`ArtifactIO`; the sole entry point is
[`core/src/main/scala/scalascript/ast/SsccFormat.scala`](../core/src/main/scala/scalascript/ast/SsccFormat.scala).

### 8.1 File header

Every `.sscc` file begins with a fixed 9-byte header:

| Offset | Length | Value | Description |
|--------|--------|-------|-------------|
| 0 | 4 | `73 73 63 63` (`sscc` in ASCII) | Magic bytes |
| 4 | 1 | `02` | Format version (current) |
| 5 | 4 | CRC32 big-endian | CRC32 of the CBOR payload bytes |

The CRC32 covers **only the payload** (bytes `[9, end)`); it does not
cover the header itself.

Read-time validation order:
1. File length ≥ 9 bytes — otherwise `Left("truncated .sscc header")`.
2. Magic bytes match — otherwise `Left("not a .sscc file (bad magic bytes)")`.
3. Version == 1 → `Left("obsolete .sscc format (v1/msgpack) — please rebuild with current ssc")`.
4. Version > 2 → `Left(s"unsupported .sscc version $version ...")`.
5. CRC32 match — otherwise `Left("corrupt .sscc file — CRC32 mismatch ...")`.
6. CBOR decode — otherwise `Left(<exception message>)`.

### 8.2 Payload encoding

The payload is a CBOR (RFC 7049) byte string produced by
[Borer 1.16.2](https://sirthias.github.io/borer/) using `MapBasedCodecs`
(field-name-keyed maps, not positional arrays). The root type is
`ModulePickle`.

```
ModulePickle {
  name:     String
  pkg:      [String, ...]
  imports:  [String, ...]
  sections: [SectionPickle, ...]
}

SectionPickle {
  heading:  bytes        # gzip-compressed UTF-8
  contents: [ContentPickle, ...]
}

ContentPickle: one of
  Prose   { text:   bytes }  # gzip-compressed UTF-8
  CodeBlock { lang: String, code: bytes }  # code = gzip-compressed UTF-8
  Import  { path:   String }
  DataList { items: [ListItemPickle, ...] }

ListItemPickle {
  label: String
  body:  bytes           # gzip-compressed UTF-8
}
```

All `bytes` fields are stored as CBOR **byte strings** (major type 2) — no
Base64 encoding. The bytes are raw gzip output (`java.util.zip.GZIPOutputStream`).

### 8.3 Text compression

Long text fields (heading, prose, code, list-item body) are compressed with
gzip before being stored as CBOR byte strings. Short fields (< ~20 chars)
still go through gzip; the overhead is negligible and the code path is uniform.

Decompress with `java.util.zip.GZIPInputStream`. Encoding is always UTF-8.

### 8.4 Version history

| Version | Encoding | Notes |
|---------|----------|-------|
| 1 | msgpack via upickle `writeBinary` | Obsolete; rejected on read |
| 2 | CBOR via Borer 1.16.2 | Current; gzip text fields; CRC32 header |

### 8.5 Implementation

- Write: `SsccFormat.write(module: Module): Array[Byte]`
- Read:  `SsccFormat.read(bytes: Array[Byte]): Either[String, Module]`

Both live in `core/src/main/scala/scalascript/ast/SsccFormat.scala`.
