# Imported Tuple Collection Matching

## Overview

ScalaScript two-element tuples must keep one source-level identity when they
cross module and collection-runtime boundaries. Native tuple literals currently
lower to the internal `Pair/2` constructor, while JVM collection operations such
as `zip`, `zipWithIndex`, map iteration, and grouping produce `Tuple2/2`. That
representation difference must not be observable through positional selectors
or tuple patterns.

## Interface

This slice adds no public API. Existing source forms retain their contracts:

```scalascript
pair._1
pair._2

pair match
  case (left, right) => ...
```

Both forms accept a two-element tuple produced by a literal, `->`, a collection
operation, or an imported module.

## Behavior

- [ ] `._1` and `._2` return the same fields for internal `Pair/2` and
      runtime-produced `Tuple2/2` values on native VM and direct ASM.
- [ ] A source `(left, right)` pattern matches either internal representation,
      including a tuple nested inside another constructor, without globally
      aliasing user-defined constructor tags.
- [ ] A focused multi-file fixture covers imported `zip` values through both
      selectors and tuple-pattern matching with byte-identical VM/ASM output.
- [ ] `examples/imports.ssc` and `examples/extensions.ssc` complete with
      byte-identical VM/ASM output and leave the blocking runtime taxonomy.
- [ ] Native-entry, corpus/parity, sentinel/runtime taxonomy, release, and fresh
      affected conformance gates remain green with zero mismatch or one-sided
      backend errors.

## Design

The self-hosted lowerer owns the compatibility seam. When it emits a
source-level two-element tuple pattern, it emits equivalent `Pair/2` and
`Tuple2/2` CoreIR arms with the same field scope and body. Nested tuple
obligations use the same expansion. The generated positional selector helpers
likewise accept both tags.

The shared CoreIR VM and ASM matchers continue comparing constructor tags
exactly. This keeps arbitrary ADTs distinct and avoids turning every internal
`Pair` match in the ScalaScript-written compiler into a `Tuple2` match.

## Decisions

- **Expand only source tuple lowering** — chosen because `Pair` is also an
  internal ScalaScript ADT used by the bootstrap compiler. Rejected: globally
  treating `Pair` and `Tuple2` as equal in every CoreIR match, which could make
  unrelated compiler data match user tuples.
- **Preserve both runtime representations** — chosen to keep the compatibility
  bridge and existing provider values stable. Rejected: rewriting every JVM
  collection result to `Pair`, which would create a broad migration unrelated
  to this standard-native correctness slice.

## Out of scope

- changing three-or-more-element `TupleN` representation;
- renaming serialized CoreIR constructor tags;
- changing compatibility/frontend-bridge tuple emission;
- implementing unrelated collection methods or provider surfaces.
