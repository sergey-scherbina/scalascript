# ScalaScript 2.1 — symbolic extension precedence

## Overview

When a symbolic infix name is provided by an in-scope extension, applying that
operator to non-primitive values must call the extension rather than forcing a
numeric primitive. The native 2.1 route currently hard-codes every `|` as
integer `i.or`, so the standard parser-combinator choice operator receives
`PChar` values and fails before parsing. The fix must preserve ordinary integer
bitwise OR in the same assembled program.

## Interface

```scalascript
case class ParserNode(label: String)

extension (left: ParserNode)
  def |(right: ParserNode): ParserNode =
    ParserNode(left.label + "|" + right.label)

val choice = ParserNode("a") | ParserNode("b") // extension call
val mask   = 6 | 3                              // integer bitwise OR = 7
```

Extension identity may originate in an imported `.ssc` module. Numeric
primitive behavior wins when both operands are integers; otherwise a registered
symbolic `|` extension receives the left operand followed by the right operand.
Without a registered `|` extension, existing primitive lowering and its bounded
type error remain unchanged.

## Behavior

- [ ] A locally declared symbolic `|` extension is selected for non-Int
      receiver/argument values.
- [ ] An imported symbolic `|` extension retains the same selected-call
      behavior after module closure assembly.
- [ ] Multiple calls construct the extension's declared result rather than
      entering numeric `i.or`.
- [ ] `6 | 3` remains integer `7` even in a program that imports the extension.
- [ ] Without a registered `|`, the existing integer primitive path and honest
      non-Int type failure remain unchanged.
- [ ] A focused multi-file fixture produces identical exact output on native VM
      and direct ASM.
- [ ] Calculator and YAML-like parser-combinator examples leave their former
      `expected Int, got PChar(...)` boundary on both lanes; any later
      independent failure is tracked separately.
- [ ] Native-entry, corpus/parity/taxonomy, release, and fresh affected
      conformance gates remain green.

## Out of scope

- Full static overload resolution for arbitrary symbolic operators over erased
  native CoreIR types.
- Changing the semantics of integer bitwise `&`, `|`, `^`, or shifts.
- Adding parser-specific runtime intrinsics or host implementations.
- Resolving the separately tracked `case object NoContext` gap.

## Design

The durable extension registry introduced by the extension-layout slice tells
the lowerer whether a symbolic `|` definition is in scope, but CoreIR has no
static receiver type with which to choose between that extension and integer
bitwise OR. When `|` is registered, lower it to one internal dynamic dispatch
primitive carrying the extension closure plus both evaluated operands.

The runtime primitive preserves the established `IntV | IntV` result. For any
other operand shape it invokes the supplied extension closure through the
ordinary closure-call contract. VM and direct ASM already share primitive
fallback execution, so this adds no compiler, reflection, or backend-specific
path. When no `|` extension exists, keep emitting `i.or` exactly as before.

## Decisions

- **Primitive Int wins, extension handles the rest** — type-erased runtime
  dispatch preserves existing numeric source semantics while enabling ADT
  extensions. Rejected: globally rewriting every `|` to the extension, which
  silently changes integer programs in the same module closure.
- **Pass the closure explicitly** — the dispatcher invokes the exact imported
  definition selected by lowering. Rejected: runtime lookup by global name,
  which would duplicate linker ownership and complicate standalone artifacts.
- **No parser special case** — the mechanism is defined for any non-Int value.
  Rejected: recognizing `PChar`/`Parser` tags in the host runtime.
