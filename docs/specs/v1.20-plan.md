# v1.20 — DSL primitives + `std/parsing` — Implementation Plan

Status: **in progress**.  Feature branch: `feature/v1.20-dsl`.
Design source: [`docs/specs/dsl.md`](dsl.md).

## Open questions (resolved before implementation)

- **StringContext shape for JsGen**: emit `{parts: [...]}` plain object
  (consistent with how JsGen treats all value types as plain JS objects).
  Extension method `extension (sc: StringContext) def sql(args: Any*)` compiles
  to `function sql(sc, ...args)` — user-defined; JsGen emits the call as
  `sql({parts: ["p1","p2"]}, arg1, arg2)`. ✅ locked.

- **INT dispatch for unknown interpolator**: construct a `Value.DictV(Map("parts" -> listOfParts))`
  and look up the method named by the prefix in the current environment,
  treating it as a regular function call. ✅ locked.

- **JvmGen unknown interpolator**: emit `StringContext("p1","p2").prefix(arg1, arg2)` — Scala 3 native. ✅ locked.

- **`std/parsing` runs on all 3 backends?**: Yes — pure `.ssc` interpreted by
  all backends. No intrinsic needed; the ADT + extension methods are plain ScalaScript. ✅ locked.

- **Reified by default**: locked per `docs/specs/dsl.md §2.5`. `Parser[A]` is an ADT;
  `.parse(input)` is the explicit run method.

## Phases

### Phase 1 — User-defined interpolators (~3 days) ✅ IN PROGRESS

**Goal**: `extension (sc: StringContext) def myDsl(args: Any*)` works on all backends.

**Changes per backend**:

| Backend | Change |
|---------|--------|
| Interpreter | `Term.Interpolate` unknown prefix → build `DictV(parts)` + call user fn |
| JsGen | `Term.Interpolate` unknown prefix → emit `prefix(_sc([p1,p2]), a1, a2)` where `_sc` builds `{parts:[...]}` |
| JvmGen | `Term.Interpolate` unknown prefix → emit `StringContext("p1","p2").prefix(a1, a2)` |

**Conformance**: `conformance/dsl-interpolator.ssc` + expected output.

**Acceptance criteria**:
- `sql"select $name from $table"` calls user-defined `sql` ext method
- Multi-arg, zero-arg, escaping (`$$`) all correct
- All 3 backends produce same output

### Phase 2 — `std/parsing/core.ssc` (~2 days)

Pure-data `Parser[A]` ADT + `Position`/`Span`/`ParseResult`/`ParseError`.
Primitive constructors: `Parser.char`, `Parser.string`, `Parser.regex`, `Parser.satisfy`,
`Parser.success`, `Parser.fail`.

No interpreter (`.parse`) yet — just data construction + pattern matching.

**Acceptance criteria**: each constructor builds the right ADT node; `pattern match` works.

### Phase 3 — `std/parsing/combinators.ssc` (~3 days)

Extension methods on `Parser[A]` producing `Parser.X` nodes (pure data):
`~`, `|`, `map`, `flatMap`, `*`, `+`, `?`, `~>`, `<~`, `named`.

Default recursive-descent `extension [A](p: Parser[A]) def parse(input: String): ParseResult[A]`.

**Acceptance criteria**: calculator from `docs/specs/dsl.md §5.4` round-trips.

### Phase 4 — `std/parsing/helpers.ssc` (~2 days)

Tokenization helpers: `whitespace`, `identifier`, `number`, `integer`,
`stringLit`, `keyword(s)`, `skip`.

**Acceptance criteria**: JSON parser written entirely from helpers parses
`{"a":1,"b":[2,3]}` → correct ADT.

### Phase 5 — `std/dsl/*` helpers (~3 days)

`std/dsl/ast.ssc` — `Span`, `HasSpan`, `Node[A]` wrapper (locked).
`std/dsl/pretty.ssc` — `Doc` pretty-printer combinators (`text`, `line`, `indent`, `beside`, `above`).
`std/dsl/builders.ssc` — phantom-type builder template (locked §3.5).

**Acceptance criteria**: typed `Calc` AST with pretty-printer round-trips
parse → AST → pretty → parse.

### Phase 6 — Examples + docs (~2 days)

- `examples/dsl-sql-interpolator.ssc`
- `examples/dsl-calc-parser.ssc`
- `examples/dsl-json-parser.ssc`
- Update `docs/specs/dsl.md` status → landed
- Update `MILESTONES.md` — mark v1.20 landed
