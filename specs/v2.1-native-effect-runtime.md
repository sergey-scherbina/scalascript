# ScalaScript 2.1 native portable effect runtime

> 2026-07-14 multiplicity refinement: the raw `Pure | Op` protocol remains
> reusable, while typed `.ssc effect` now has the source-level one-shot contract
> specified in [`control-one-shot-guard.md`](control-one-shot-guard.md).

## Overview

The v2 kernel already contains a target-neutral algebraic-effect value contract:
`effect.perform` creates an `Op(label, argument, continuation)` and
`effect.handle` folds that computation through a one-argument partial-function
handler. The compatibility frontend uses it, but the self-hosted frontend does
not yet preserve the same source semantics:

- ordinary `effect E:` declarations are erased through a fragile layout skip;
- `handle(body) { case ... }` is parsed with the handler partial function and
  then deliberately lowered as `body` alone;
- the resulting unhandled `Op` is reported differently by VM and direct ASM.

ScalaScript 2.1 must lower explicit user effects and handlers entirely through
the existing portable CoreIR primitives, without Scalameta, v1 bridge state,
an exception-shaped public ABI, or backend-specific semantics. A JVM runtime may
use a private unwind carrier for a structured run failure, but generated `.ssc`
`try/catch` must rethrow it and the runner boundary must recover the typed failure.

## Interface

```scalascript
effect Console:
  def readLine(): String
  def writeLine(value: String): Unit

val result = handle(program()) {
  case Console.readLine(resume)       => resume("World")
  case Console.writeLine(msg, resume) => println(msg); resume(())
  case Return(value)                  => value
}
```

Each declared operation has the source-level behavior:

```text
effect E:       E.op(a1, ..., an) = effect.perform.oneshot("E", "op", a1, ..., an)
multi effect E: E.op(a1, ..., an) = effect.perform("E.op",         a1, ..., an)
```

Zero arguments use the portable unit payload, one argument remains one value,
and multiple arguments use the internal `__EffArgs__` envelope so the handler
event has fields `(a1, ..., an, resume)`. A genuine tuple argument remains a
single argument.

`handle(computation, handler)` lowers to:

```text
effect.handle(<computation>, <one-argument partial-function closure>)
```

The computation is allowed to evaluate to an `Op`; top-level unhandled-effect
reporting happens only after the enclosing handle has had the opportunity to
consume it. `resume(v)` invokes the captured continuation and then applies the
same deep handler to the next computation. A raw/`multi effect` resume closure is
reusable, so calling it more than once implements multi-shot branching. A plain
`effect` resume delegates through one shared atomic base gate; a second or
concurrent losing invocation is `AlreadyResumed` and never executes the suffix.

## Design

The frontend should retain an `effect_decl` node containing the effect name and
operation signatures, using the same explicit layout-body ownership as object
and given bodies. Lowering emits operation closures (or an equivalent explicit
effect receiver) whose semantic operation is `effect.perform.oneshot` or raw
reusable `effect.perform` according to the declaration bit.

The parser already converts `{ case ... }` into a one-argument lambda matching
the event ADT. The lowerer's current special case that detects
`handle(computation)(handler)` and returns only `computation` must instead emit
`effect.handle` with both resolved operands. `PortableEffects` owns argument
packing, deep resume, one shared one-shot gate, multi-shot reuse, and `Return`
fallback. The one additive generic primitive name does not change CoreIR's term
inventory or the `Op` value shape. Its semantic ABI id is
`effect.perform.oneshot@1`; the current in-process `Prim` encoding carries the
name unchanged, concrete runtime/AOT allowlists add it explicitly, and future
durable dependency manifests record the versioned id. Changing self-hosted
lowering requires the affected ssc1 fixed-point gate; it does not change the
CoreIR codec.

The older KV9 transform that recognizes a multi-effect call inside a function
and adds a private continuation parameter is incompatible with this contract:
ordinary callers cannot supply that hidden argument, and its list-only
`flatMap` semantics bypass declared operations. It must not participate once
an effect declaration emits the appropriate `effect.perform*` operation.

Standard runner semantics remain a separate sub-slice because they add named
standard behavior beyond the generic language construct. Likewise,
`Dataset.of` and the recovery parser's `Parser.regex` are unresolved selected
calls that happen to fall back to an `Op`; they are not evidence that
`effect.handle` itself should know Dataset or Parser.

Rejected alternatives:

- treating an operation as a host exception, which cannot support multi-shot
  resume or deterministic direct ASM;
- silently discarding the handler and accepting matching backend failures;
- special-casing Console/Choose/Fail, Dataset, or Parser in the core runtime;
- invoking the compatibility frontend when a handler is present.

## Baseline

Installed compiler-free main on 2026-07-11 produces:

- `effects.ssc`: VM/ASM both exit 1 with `unbound global: greet`, no stdout;
- `algebraic-effects.ssc`: VM exits on unhandled `Logger.log`; ASM prints six
  partial lines/Op values and exits on `arity: 2 expected, 1 given`;
- `dataset-stats.ssc`: VM exits on unhandled `Dataset.of`; ASM prints four raw
  Op lines then `match: no arm for Op/3`;
- `dsl-sql-recovery.ssc`: VM exits on unhandled `Parser.regex`; ASM exits on
  `match: no arm for Op/3`.

Explicit compatibility output is six lines for `effects.ssc` and eleven for
`algebraic-effects.ssc`. The latter runner behavior is TI-8.2d2w2. Dataset is
queued as a core-free standard provider and parser recovery as a self-hosted
companion/declaration ownership slice. Current release baseline before this
work is 194/194 front/check, 44/82 runtime, parity 31 identical / 35 both-fail /
129 skips with zero mismatch/one-sided rows, and 23 blocking taxonomy rows.
